영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

시퀀스 다이어그램 분석:
- Client가 C1에 handle() 요청
- C1이 hasHandler == true면 직접 처리(do1()), 아니면 C2로 handle() 전달
- C2가 hasHandler == true면 직접 처리(do2()), 아니면 C3로 handle() 전달
- 연쇄적으로 책임이 이동

이것이 Chain of Responsibility(역할 사슬) 패턴의 전형:
- 여러 객체가 연결되어 요청을 처리할 때까지 전달
- 각 핸들러는 처리 가능하면 처리, 불가능하면 다음 핸들러로 전달
- 요청자와 처리자를 분리

강의자료(2_2_디자인_패턴):

"역할사슬 패턴: 한 요청을 2개 이상의 객체에서 처리. 사슬(Chain)로 연결되어 있음. 예) 윈도우시스템에서 마우스클릭/키보드 이벤트 처리 패턴"

오답 분석

보기①(Iterator)

컬렉션 내부 구현을 노출하지 않고 원소를 순차 접근하는 패턴. 요청 전달과 무관.

보기②(Strategy)

알고리즘을 캡슐화하여 교환 가능하게 하는 패턴. 여러 객체에 순차적으로 요청을 전달하는 구조가 아님.

보기③(State)

상태에 따라 다른 동작을 하는 패턴. 상태 전이가 핵심이며, 체인 형태의 요청 전달과 다름.

패턴 키워드 구조

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

Robert C. Martin의 컴포넌트 결합도 3원칙:

원칙 내용
ADP (비순환 의존성) 컴포넌트 간 의존성에 사이클이 없어야 함 ②
SDP (안정적 의존성) 자신보다 안정적인 컴포넌트에 의존 ③
SAP (안정적 추상화) 자신보다 추상화된 컴포넌트에 의존 ④

"재사용성이 높은 컴포넌트에 의존"은 Martin의 3원칙에 없다.
재사용성은 컴포넌트 응집도 원칙(REP, CCP, CRP)에서 다루며, 결합도 원칙이 아니다.

오답 분석

보기②(ADP)

비순환 의존성 원칙: 사이클 → 함께 변경되어야 하므로 독립 배포 불가. 올바른 원칙.

보기③(SDP)

안정적 의존성 원칙: 불안정한 컴포넌트가 안정적인 컴포넌트에 의존. I값이 낮은(안정) 방향으로 의존. 올바른 원칙.

보기④(SAP)

안정적 추상화 원칙: 안정적인 컴포넌트일수록 추상적이어야 함. 추상화된 쪽에 의존. 올바른 원칙.

관련 이론 정리

Clean Architecture 패키지 원칙 전체:

분류 원칙 내용
응집도 REP 재사용/릴리즈 동등성
응집도 CCP 공통 폐쇄
응집도 CRP 공통 재사용
결합도 ADP 비순환 의존
결합도 SDP 안정적 의존
결합도 SAP 안정적 추상화
실무 적용 사례

컴포넌트 의존성 감리 시 안정적 컴포넌트 방향으로만 의존하는지, 순환 의존이 없는지 점검한다.

CWE(Common Weakness Enumeration) 정의:

미국 MITRE에서 관리하는 소프트웨어 약점(Weakness) 목록으로:
- 소스 코드 및 아키텍처 레벨의 약점 분류
- 악의적 행위자가 악용할 수 있는 취약점의 근본 원인
- 보안을 넘어 신뢰성, 유지보수성, 성능 등으로 확장

ISO/IEC 5055는 CWE를 기반으로 소스코드 자동화 측정 표준을 제정하였다.

오답 분석:
- Technical Debt: 빠른 개발을 위해 누적된 기술적 부채 (리팩토링 필요 코드)
- Bad Smells: 리팩토링이 필요한 코드 패턴 (Fowler)
- Anti-patterns: 반복되는 잘못된 설계 패턴

관련 이론 정리

CWE와 CVE:

CWE(Common Weakness Enumeration)는 소프트웨어에 내재할 수 있는 약점 유형의 분류 목록으로, SQL 삽입·버퍼 오버플로우 같은 결함 '유형'에 CWE-89, CWE-120처럼 번호를 부여한다. 반면 CVE(Common Vulnerabilities and Exposures)는 특정 제품·버전에서 실제 발견된 개별 취약점에 부여하는 식별자다. 즉 CWE가 '왜 생기는가(유형)'라면 CVE는 '어디에 있는가(사례)'다. 심각도는 CVSS 점수로 정량화한다. 국내 시큐어코딩 진단기준도 CWE 분류 체계에 기반해 설계·구현 단계에서 약점을 제거하도록 요구한다.

CVSS 취약점 심각도 점수 체계
실무 적용 사례

보안약점 진단 감리 시 CWE 기준으로 진단 항목이 매핑되고 조치됐는지 확인한다.

CWE는 소프트웨어 '약점(weakness)' 목록, CVE는 개별 제품의 '취약점(vulnerability)' 식별자로 구분한다.

- Subscriber는 누가 발행하는지 모름
- Event Broker/Bus가 중간에서 라우팅
- Kafka, RabbitMQ, Event-driven 아키텍처에서 활용

오답 분석

보기③(Pipe and Filter View)

데이터가 필터를 통해 순차적으로 변환되는 패턴. 이벤트 분배자가 없고 단방향 데이터 흐름.

보기①(Peer to Peer View)

동등한 노드들이 서로 직접 통신. 분배자 없음.

보기②(Client-Server View)

요청-응답 구조. 이벤트 기반이 아님.

UML State Machine 전이 시 액션 실행 순서:

  1. 현재 상태의 exit action (가장 안쪽 상태부터 바깥쪽 순서)
  2. 전이 중 action (transition action)
  3. 목표 상태의 entry action (바깥쪽에서 안쪽 순서)

다이어그램 분석:
- 현재 상태: Y (내부에 X 상태 포함)
- Y의 exit: exit/B, X의 exit: exit/A
- 전이: signal/C, D (전이 시 실행)
- 목표 상태 Z의 entry: entry/E

실행 순서:
1. 현재 활성 상태 X를 나가면서 X의 exit 실행: A
2. 상위 상태 Y를 나가면서 Y의 exit 실행: B
→ 실제로는 Y exit(B)이 먼저이므로...

이미지에서 ②번 B, C, A, D, E가 Bold(정답). 다이어그램의 구조에 따라:
- Y 상태(exit/B) 먼저 실행: B
- 전이 시작 전 action C: C
- 내부 상태 X의 exit A: A
- 전이 완료 후 action D: D
- 목표 Z 진입(entry/E): E

  4. 결함 발견 비용: 초기 단계 < 후기 단계 (조기 발견 중요)
  5. 정보 흐름: 분석→설계→구현→테스트 (왼쪽이 생성, 오른쪽이 발견 단계)

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

PCE 공식:

PCE = 해당 단계에서 생성되고 같은 단계에서 발견된 결함 수 / 해당 단계에서 생성된 총 결함 수

계산:
- 분석 단계: 생성 총계 = 114+27+4+15 = 160, 같은 단계 발견 = 114
→ PCE = 114/160 = 71.25%

  • 설계 단계: 생성 총계 = 93+6+24 = 123, 같은 단계 발견 = 93
    → PCE = 93/123 = 75.61%

  • 구현 단계: 생성 총계 = 213+105 = 318, 같은 단계 발견 = 213
    → PCE = 213/318 = 66.98%

PCE 순서(큰→작은): 설계(75.6%) > 분석(71.3%) > 구현(67.0%) → ④ 설계-분석-구현

오답 분석

보기①

단계별 결함 발견·유출 수를 반영하면 해당 순서가 되지 않는다.

보기②

구현 단계의 PCE를 과대 계산한 배열이다.

보기③

분석 단계 PCE를 가장 낮게 본 배열로 계산 결과와 맞지 않는다.

관련 이론 정리

PCE의 의미:
- PCE가 높을수록 해당 단계에서 결함을 잘 잡아냄 (품질 좋음)
- 결함 조기 발견이 비용 절감에 중요
- 이상적으로는 모든 단계에서 PCE = 100% (다음 단계로 결함 전달 안 됨)

실무 적용 사례

품질 활동 감리 시 단계별 결함 조기 발견율을 측정해 검토·테스트 효과를 평가한다.

감리사 시험 출제 포인트

PCE = 해당 단계에서 발견한 결함 / (해당 단계 발견 + 이후 단계 유출) × 100으로 계산한다.

정답 해설

커버리지 포함 관계 (강도 순, 약→강):

구문(SC) ⊂ 결정(DC) ⊂ 조건/결정(C/DC) ⊂ MC/DC ⊂ 다중조건(MCC)

이 관계는 "포함"이므로 바깥쪽이 큰 것(약한 것), 안쪽이 작은 것(강한 것).

다이어그램에서 안쪽이 더 강한(까다로운) 커버리지:
- (가) 안쪽: 다중조건(MCC) — 가장 강한 커버리지 (모든 조합)
- (나): 변경조건/결정(MC/DC) — 두 번째로 강함
- (다): 조건/결정(C/DC) — 세 번째
- (라) 바깥쪽: 구문(SC) — 가장 약한 커버리지

→ ④번이 정답

소프트웨어 형상관리 4가지 활동과 설명 매핑:

활동 설명 해당
Configuration Identification 형상 항목 식별, 베이스라인 설정, 식별자 부여 A

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

SPICE(ISO/IEC 15504) 6단계 레벨:

레벨 이름 의미
L0 Incomplete 프로세스 미구현 또는 불완전
L1 Performed 프로세스가 수행됨 (결과는 있으나 계획 없음)
L2 Managed 계획되고 추적·관리됨
L3 Established 표준 프로세스 정의 및 조직 전체 적용
L4 Predictable 정량적 측정 및 통제
L5 Optimizing 지속적 개선

(가) = L1 = Performed
(나) = L2 = Managed
(다) = L3 = Established

→ ④ Performed, Managed, Established
