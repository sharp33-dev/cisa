영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

적합성(conformance)은 데이터가 정해진 표준 코드 값이나 표준 도메인(허용 값 범위)에 부합하는 정도를 나타내는 데이터 품질 기준이다. 표준 코드/도메인을 위반하면 적합성 오류가 발생한다.

문제의 "표준 코드 값 또는 표준 도메인 값에 위배될 경우 발생"은 정확히 적합성 오류의 정의이므로 정답은 ③번이다. 충분성(데이터 존재·완전성), 사실성(실제와 일치), 필수성(필수 항목 존재)은 각각 다른 품질 차원이다.

오답 분석

보기①

충분성은 필요한 데이터가 충분히 존재하는지(완전성)에 대한 기준으로, 표준 코드·도메인 위반과는 다르다.

보기②

사실성은 데이터가 실제 사실과 일치하는지(정확성)의 기준이다.

보기③

정답. 표준 코드/도메인 값 준수 여부 = 적합성.

보기④

필수성은 반드시 있어야 할 필수 데이터의 존재 여부에 대한 기준이다.

관련 이론 정리

데이터 품질 기준(차원) 예시:

기준 의미
적합성 표준 코드·도메인 규칙 준수
정확성/사실성 실제 사실과 일치
완전성/충분성 필요한 데이터가 빠짐없이 존재
일관성 값 간 모순 없음
유일성 중복 없음
실무 적용 사례

데이터 품질 감리 시 코드 값·도메인 규칙(예: 성별 코드, 날짜 형식) 위반 여부를 프로파일링해 적합성 오류를 진단하고 정제 규칙을 권고한다.

감리사 시험 출제 포인트

품질 차원(적합성·정확성·완전성 등)의 정의를 구분한다. "표준 코드/도메인 위반 = 적합성"을 정확히 연결한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

블록에서 레코드가 실제로 들어가는 공간 = 블록 크기 − 헤더 = 4096 − 12 = 4084 바이트.
블록당 저장 가능 레코드 수 = ⌊4084 / 300⌋ = ⌊13.6⌋ = 13개.

  • 첫 번째 블록: 13개 저장 → 300×13 = 3900 바이트 사용, 잔여 = 4096 − 12 − 3900 = 184 바이트(①의 196은 오답).
  • 두 번째 블록: 남은 15 − 13 = 2개 저장 → 300×2 = 600 바이트 사용, 잔여 = 4096 − 12 − 600 = 3,484 바이트.

따라서 "두 번째 블록에 3,484 바이트가 남는다"는 ②번이 옳다.

오답 분석

보기①

첫 번째 블록 잔여는 4096 − 12 − 3900 = 184 바이트로, 196이 아니다.

보기②

정답. 두 번째 블록은 2개(600B) 저장, 잔여 4096 − 12 − 600 = 3,484 바이트.

보기③

보기④

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

슈퍼키는 튜플을 유일하게 식별하는 속성 집합으로, 반드시 어떤 후보키(Registration 또는 Vehicle_id)를 포함해야 한다.

  • ① (State, Year, Model): Vehicle_id도 없고, Registration(State+Number)을 완성하지도 못한다(Number 없음). 따라서 어떤 후보키도 포함하지 못하므로 슈퍼키가 아니다.
  • ② (Vehicle_id, Number): 후보키 Vehicle_id를 포함 → 슈퍼키.
  • ③ (Registration, State): 후보키 Registration을 포함 → 슈퍼키.
  • ④ (Registration, Vehicle_id, Year): 후보키 두 개를 포함 → 슈퍼키.

따라서 슈퍼키가 아닌 것은 ①번이다.

보기①

정답(슈퍼키 아님). 후보키(Registration/Vehicle_id)를 포함하지 않아 유일 식별이 보장되지 않는다.

보기②

보기③

보기④

정답 해설

각 연산 결과의 카디널리티 상한(최대 가능 크기)은 다음과 같다.

  • |R ∪ S| ≤ |R| + |S| = 7 + 6 = 13
  • |R ∩ S| ≤ min(|R|, |S|) = min(7, 6) = 6
  • |R − S| ≤ |R| = 7
  • |R × S| = |R| × |S| = 7 × 6 = 42

가장 작은 상한값은 교집합 |R ∩ S|의 6이다. 따라서 정답은 ②번이다. 교집합은 두 릴레이션 모두에 존재하는 튜플만 남으므로 크기가 작은 릴레이션(|S|=6)을 넘을 수 없다.

뷰는 기본 테이블의 특정 행/열로 정확히 역매핑될 수 있을 때만 갱신 가능하다. 다음의 경우 뷰 갱신이 제한된다.

- 나: DISTINCT·GROUP BY(집계) 뷰 → 원본 행과 1:1 대응이 깨짐 → 변경 불가.
- 다: 기본 테이블의 기본키가 뷰에 없으면 → 어느 행을 갱신할지 식별 불가 → 변경 불가.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

함수 종속의 추론 규칙(암스트롱 공리와 파생 규칙)을 검토한다.
- ① X ⊇ Y이면 X → Y : 반사(reflexivity) 규칙 → 옳음.
- ② X → YZ이면 X → Y, X → Z : 분해(decomposition) 규칙 → 옳음.
- ③ X → Y이면 XZ → YZ : 증가(augmentation) 규칙 → 옳음.
- ④ XZ → YZ이면 X → Y : 이는 증가 규칙의 역방향으로 일반적으로 성립하지 않는다. 예컨대 XZ가 함께 있어야 YZ를 결정하는데, Z를 제거한 X만으로 Y를 결정한다는 보장이 없다. 따라서 옳지 않다.

정답(옳지 않은 것)은 ④번이다.

오답 분석

보기①

보기②

분해 규칙: X → YZ면 X → Y, X → Z로 분해 가능. 성립.

보기③

증가 규칙: X → Y면 양변에 Z를 더해 XZ → YZ. 성립.

보기④

정답(옳지 않음). 증가 규칙의 역(XZ→YZ ⇒ X→Y)은 일반적으로 성립하지 않는다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

외래키는 자식(EMPLOYEE.Dno)이 부모(DEPARTMENT.Dnumber)를 참조한다. 참조 옵션은 부모(DEPARTMENT) 쪽의 변경이 발생할 때 자식(EMPLOYEE)이 어떻게 반응할지를 정의한다.
- ON DELETE SET DEFAULT: 부모(Dnumber) 삭제 시 자식(Dno)을 기본값으로 설정.
- ON UPDATE CASCADE: 부모(Dnumber) 갱신 시 자식(Dno)도 연쇄 갱신.

④는 "DEPARTMENT의 Dnumber 갱신 → EMPLOYEE의 Dno 연쇄 갱신"으로 ON UPDATE CASCADE와 정확히 일치하므로 옳다. 정답은 ④번이다.

①·②는 자식(EMPLOYEE) 변경이 부모(DEPARTMENT)에 영향을 준다고 방향을 반대로 서술했고, ③은 삭제 시 NULL이 아니라 SET DEFAULT(기본값)이므로 틀렸다.

보기①

방향 반대. 참조 옵션은 부모 변경 시 자식에 적용된다(자식 삭제가 부모에 영향 X).

보기②

방향 반대. 자식(EMPLOYEE) 갱신이 부모(DEPARTMENT)를 바꾸지 않는다.

보기③

부모 삭제 시 자식은 SET DEFAULT(기본값)이지 NULL이 아니다.

보기④

정답. 부모(Dnumber) 갱신 → 자식(Dno) 연쇄 갱신(ON UPDATE CASCADE).

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

R(A, B, C, D)에서 A→BCD, B→ACD이므로 후보키는 A와 B(각각 모든 속성 결정)이다. 그런데 D→C는 결정자 D가 후보키가 아니므로 BCNF(및 3NF)를 위반한다.

BCNF를 만족하도록 위반 종속 D→C를 분리한다.
- R2(D, C): D→C를 담는 릴레이션(D가 키).
- R1(A, B, D): 나머지. A·B가 후보키이고 D는 외래키(R2 참조).

이 분해는 무손실이며 두 릴레이션 모두 BCNF를 만족한다. 따라서 정답은 ①번 R1(A, B, D), R2(D, C)이다.

오답 분석

보기②

R2(C, D)는 D→C를 담지만 R1(A, B, C)에 C가 남아 D→C 위반이 분리되지 않거나 무손실이 깨질 수 있어 부적절하다.

보기③

R2(B, D) 구성이 D→C 위반을 해소하지 못하고 C의 배치가 잘못됐다.

보기④

R2(B, A) 등 구성이 D→C 위반 분리와 무손실 요건을 충족하지 못한다.

관련 이론 정리

BCNF 분해 절차:

단계 내용
1 후보키 식별(A, B)
2 BCNF 위반 FD 탐지(D→C, D 비후보키)
3 위반 FD를 별도 릴레이션으로(R2(D,C))
4 나머지에 결정자(D) 외래키로 유지(R1(A,B,D))
실무 적용 사례

정규화 감리 시 비후보키 결정자(D→C 같은)를 분리해 갱신 이상을 제거하고, 분해가 무손실·종속성 보존을 만족하는지 검증한다.

감리사 시험 출제 포인트

BCNF 위반 결정자(D)를 찾아 분리하고, 남은 릴레이션에 결정자를 외래키로 유지하는 분해를 선택한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

"red 보트와 green 보트를 모두 예약한 선원"은 두 조건을 동시에 만족(교집합)해야 한다. 하나의 예약 행(R, B)은 red이면서 동시에 green일 수 없으므로(②의 B.color='red' AND B.color='green'은 항상 거짓), Reserves와 Boats를 각각 두 벌(R1·B1, R2·B2)로 자기 조인하여 같은 선원(S.sid)이 red 보트(B1)와 green 보트(B2)를 각각 예약했음을 표현해야 한다.

③번은 정확히 이 구조로, 같은 sid가 red 보트와 green 보트를 각각 예약한 경우를 조인으로 찾는다. 따라서 정답은 ③번이다. ①(OR)·④(UNION)는 red 또는 green(합집합)이라 요구와 다르고, ②는 한 행에서 red=green을 요구해 항상 공집합이다.

오답 분석

보기①

OR는 red 또는 green을 예약한 선원(합집합)으로 "모두"가 아니다.

보기②

한 예약 행에서 color='red' AND color='green'은 동시에 성립 불가 → 항상 빈 결과.

보기③

정답. Reserves·Boats 두 벌로 자기 조인하여 같은 선원이 red·green을 각각 예약했음을 표현.

보기④

UNION은 red 예약 선원 ∪ green 예약 선원(합집합)으로, 교집합("모두")이 아니다.

관련 이론 정리

"모두 만족" 질의 구현:

방법 내용
자기 조인 동일 키를 여러 벌 조인(각 조건별)
INTERSECT 각 조건 결과의 교집합
나눗셈/이중 NOT EXISTS 전량 만족(for all)
OR/UNION 합집합("또는") → 부적절
실무 적용 사례

"A와 B를 모두 이용한 고객" 같은 교집합 조회 감리 시 자기 조인/INTERSECT로 구현했는지, OR/UNION으로 잘못 합집합 처리하지 않았는지 확인한다.

감리사 시험 출제 포인트

"모두(AND)"는 교집합이며, 한 행에서 red=green 불가라는 점을 이용한 자기 조인 필요성을 이해했는지 묻는다. OR/UNION 함정에 유의.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

표준 SQL의 조인 구문은 A JOIN B ON (조건) 또는 A NATURAL JOIN B 또는 A JOIN B USING(공통컬럼) 형태다. 이들을 혼용할 수 없다.

  • ④: STUDENT JOIN ENROLL ON (STUDENT.Sno = ENROLL.Sno) — 올바른 ON 조인. WHERE로 Cno='C413' 필터. 정답.
  • ①: JOIN ... WITH(Sno) — WITH는 조인 키워드가 아니다(잘못된 구문).
  • ②: NATURAL JOIN ... USING(Sno) — NATURAL JOIN과 USING을 함께 쓸 수 없다.
  • ③: NATURAL JOIN ... ON(...) — NATURAL JOIN과 ON을 함께 쓸 수 없다.

따라서 문법적으로 올바른 것은 ④번이다.

오답 분석

보기①

WITH(Sno)는 존재하지 않는 조인 구문이다.

보기②

NATURAL JOIN은 공통 컬럼으로 자동 조인하므로 USING을 추가로 쓸 수 없다(구문 충돌).

보기③

NATURAL JOIN에 ON 조건절을 함께 붙일 수 없다(구문 충돌).

보기④

정답. JOIN ... ON (조건) + WHERE 필터로 올바른 표준 구문.

관련 이론 정리

조인 구문 형태:

형태 예
ON A JOIN B ON A.k=B.k
USING A JOIN B USING(k)
NATURAL A NATURAL JOIN B
혼용 NATURAL+ON, NATURAL+USING 불가
실무 적용 사례

조인 쿼리 리뷰 감리 시 표준 조인 구문을 정확히 사용하고, NATURAL JOIN의 암묵적 공통 컬럼 조인으로 인한 예기치 않은 결과를 방지하는지 확인한다.

감리사 시험 출제 포인트

JOIN 구문의 올바른 형태와 혼용 불가(NATURAL+ON/USING)를 구분한다. WITH 같은 비존재 키워드를 배제한다.

정답 해설

T1이 집계 질의(범위 조회)를 수행하는 사이에 T2가 새로운 행을 INSERT하고 커밋하면, T1이 다시 범위 질의를 수행할 때 이전에 없던 "유령" 행이 나타난다. 이것이 유령 읽기(phantom read)이다.

  • 오손 읽기(dirty read): 커밋되지 않은 데이터를 읽음(여기서는 T2가 커밋함 → 해당 없음).

  • 유령 읽기(phantom read): 새 행 삽입/삭제로 결과 집합의 행 수가 달라짐 → 본 상황에 해당.

따라서 정답은 ②번이다. (③ cascading read, ④ serializable read는 표준 격리성 이상 현상 명칭이 아니다.)

정답. 범위 질의 사이 새 행 삽입 → 유령 읽기.

  • ② 기본 2PL: 로크를 거는 확장 단계와 푸는 축소 단계로 구성(한 번 풀면 다시 못 잠금).

정답 해설

HDFS는 마스터-슬레이브 구조로, NameNode가 파일→블록→위치의 메타데이터를 관리하고 어느 DataNode에 블록이 있는지 유지한다. DataNode는 실제 데이터 블록을 저장하고 클라이언트에 제공한다. Secondary NameNode는 NameNode의 메타데이터 체크포인트/백업 역할을 한다.

① 데이터의 실제 저장 위치를 알 필요 없이 데이터베이스의 논리적 이름만으로 데이터에 접근할 수 있는 위치 투명성(location transparency)을 보장해야 한다.
② 분산 데이터베이스와 관련된 트랜잭션들이 동시에 수행되더라도 항상 일관성을 유지하는 장애 투명성(failure transparency)을 보장해야 한다.
③ 동일한 데이터가 여러 지역에 중복해서 저장되는 경우 사용자가 중복을 인식하지 못하게 하는 중복 투명성(replication transparency)을 보장해야 한다.
④ 하나의 릴레이션을 더 작은 조각(단편)으로 나누고 각 조각을 별개의 릴레이션으로 처리할 때 완전성, 회복성, 분리성을 보장하는 단편화 투명성(fragmentation transparency)을 보장해야 한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

②는 "트랜잭션들이 동시에 수행되더라도 항상 일관성을 유지"하는 성질을 장애 투명성(failure transparency)이라고 서술했으나, 이 설명은 병행 투명성(concurrency transparency)에 해당한다. 장애 투명성은 시스템·구성요소 장애가 발생해도 트랜잭션이 정확히 완료(원자성 보장)되는 것을 사용자가 인식하지 못하게 하는 성질이다. 명칭과 정의가 어긋났으므로 ②가 적절하지 않다. 정답은 ②번이다.

①(위치 투명성), ③(중복 투명성), ④(단편화 투명성)는 각각 정의와 명칭이 정확히 일치한다.

보기①

옳음. 논리적 이름만으로 접근 = 위치 투명성.

보기②

정답(부적절). "동시 수행 시 일관성 유지"는 병행 투명성의 설명이며 장애 투명성이 아니다.

보기③

옳음. 중복 저장을 사용자가 인식하지 못하게 = 중복(복제) 투명성.

보기④

옳음. 릴레이션을 단편으로 나눠도 인식하지 못하게 = 단편화 투명성.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

CAP 정리의 핵심은 분산 시스템에서 일관성(C)·가용성(A)·분할 감내(P) 세 성질을 동시에 모두 만족할 수 없고, 최대 두 개만 선택할 수 있다는 것이다. 특히 네트워크 분할(P)은 분산 환경에서 불가피하므로 실제로는 C와 A 사이에서 절충한다.

③은 "세 가지를 모두 보장할 것을 요구한다"고 서술했으나, CAP 정리는 정반대로 세 성질을 동시에 만족할 수 없다고 말한다. 따라서 ③이 적절하지 않으며 정답은 ③번이다. ①(복제 분산 요구사항)·②(CAP 정의)·④(궁극적 일관성 수용)는 옳다.

옳음. NoSQL은 강한 일관성 대신 궁극적 일관성을 흔히 수용한다.

정답 해설

네 측면 모두 관계형 DB보다 NoSQL에 가까운 특성이다.
- (가) 스키마리스/유연한 스키마: NoSQL의 대표 특징(관계형은 고정 스키마).
- (나) 클러스터 확장성(수평 확장): NoSQL이 분산·클러스터에 강함.
- (다) 단순 검색 기능: NoSQL은 복잡한 조인·SQL 대신 단순 조회 중심.
- (라) 일관성 보장 어려움: NoSQL은 강한 일관성보다 궁극적 일관성(BASE)을 수용.

따라서 네 측면 모두 NoSQL에 가까우므로 정답은 ④번이다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설
  • (가): k-Means는 초기 중심점(centroid) 선택에 따라 지역 최적해로 수렴이 달라질 수 있다 → 옳음(그래서 k-means++ 같은 초기화 개선을 사용).
  • (나): k-Means는 구형(원형·볼록) 군집을 가정하므로, 군집 모양이 구형이 아닐 때는 오히려 잘 동작하지 않는다 → 틀림.
  • (다): 평균(중심)을 사용하므로 이상치에 민감하여 이상치가 있으면 잘 동작하지 않는다 → 옳음.

따라서 옳은 것은 가·다이며 정답은 ②번이다. (나)는 사실과 반대(비구형에 약함)이다.

오답 분석

보기①·④

'나'(비구형에 잘 동작)를 포함하고 있어 옳지 않다. k-Means는 구형 군집에 적합하다.

보기②

정답. 초기 중심점 민감(가)·이상치에 취약(다)이 옳다.

보기③

'나'를 포함해 옳지 않다.

관련 이론 정리

k-Means 특성:

항목 내용
초기값 민감 초기 중심에 따라 결과 변동(k-means++)
군집 형태 구형(볼록)에 적합, 비구형에 약함

실무 적용 사례

군집 분석 감리 시 데이터 분포(비구형)·이상치 존재에 따라 k-Means 대신 DBSCAN·GMM 등 대안이 적절한지, 초기화·k 선정이 타당한지 검토한다.

감리사 시험 출제 포인트

k-Means가 구형 군집에 적합(비구형에 약함)하고 이상치에 취약하다는 특성을 정확히 아는지 묻는다. "비구형에 잘 동작"이라는 반대 서술(나)이 함정이다.
