영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

LRU(Least Recently Used)는 "가장 오랫동안 사용되지 않은(가장 오래전에 참조된) 페이지"를 교체 대상으로 선택하는 페이지 교체 알고리즘이다. 문제의 서술과 정확히 일치하므로 정답은 ②번이다.

LRU는 시간 지역성(최근 쓰인 페이지가 곧 다시 쓰일 가능성이 높음)에 근거해 성능이 우수하지만, 참조 시각을 기록·관리해야 해 구현 부담이 있다. ①FIFO는 적재 순서가 가장 오래된 페이지를, ③Round-Robin과 ④SPN은 페이지 교체가 아니라 CPU 스케줄링 알고리즘이므로 이 문제와 층위가 다르다.

오답 분석

보기①

FIFO는 '먼저 적재된' 페이지를 교체하는 것으로, '가장 오래 사용되지 않은'과 기준이 다르다(벨레이디 이상 현상 발생 가능).

보기②

정답. LRU는 가장 오랫동안 참조되지 않은 페이지를 교체한다.

보기③

Round-Robin은 시간 할당량 기반 CPU 스케줄링 알고리즘으로 페이지 교체와 무관하다.

보기④

SPN(Shortest Process Next)은 CPU 스케줄링 알고리즘이다.

관련 이론 정리

가상 메모리에서 페이지 부재(page fault)가 발생하고 빈 프레임이 없으면 교체할 페이지를 골라야 한다. OPT(최적)는 앞으로 가장 오래 쓰이지 않을 페이지를 교체해 이론적 최소 부재를 내지만 미래를 알아야 해 구현 불가하며 성능 비교 기준으로만 쓴다. FIFO는 적재 순서대로 교체해 단순하지만 프레임을 늘려도 부재가 늘 수 있는 벨레이디의 이상 현상이 나타난다. LRU는 과거 참조 시각을 근거로 미래를 근사해 성능이 좋고 이상 현상이 없으나 오버헤드가 있어, 실무에서는 참조 비트를 이용한 클럭(2차 기회) 알고리즘으로 근사한다. LFU는 참조 횟수가 가장 적은 페이지를 교체한다. 감리에서는 메모리 부족·스래싱 징후와 워킹셋 관리, 페이지 부재율 지표를 점검한다.

페이지 교체 알고리즘:

알고리즘 교체 기준
OPT 미래에 가장 늦게 쓰일 페이지
FIFO 가장 먼저 적재된 페이지
LRU 가장 오래 참조되지 않은 페이지
LFU 참조 횟수가 가장 적은 페이지
실무 적용 사례

서버 성능 감리 시 페이지 부재율·스와핑 발생량을 점검해 메모리 증설이나 워킹셋 조정이 필요한지 판단한다.

감리사 시험 출제 포인트

페이지 교체(FIFO·LRU·LFU·OPT)와 CPU 스케줄링(RR·SPN·SJF)을 섞어 놓은 보기 구성에 유의한다. "가장 오랫동안 사용하지 않은"=LRU가 핵심 단서다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

UDP 헤더는 8바이트로 출발지 포트, 목적지 포트, 길이(Length), 체크섬(Checksum) 네 필드만 갖는다. TCP 헤더에도 체크섬 필드가 있으므로, 두 프로토콜에 공통으로 존재하는 필드는 체크섬이다. 따라서 정답은 ①번이다.

②순서 번호, ③긴급 포인터, ④ACK 번호는 신뢰성·순서보장·흐름제어를 위한 TCP 전용 필드로 UDP에는 존재하지 않는다(UDP는 비연결·비신뢰 전송이라 이런 기능이 없다).

오답 분석

보기①

정답. 체크섬은 TCP·UDP 모두에 존재하는 오류 검출 필드다.

보기②

순서 번호는 바이트 스트림 순서보장을 위한 TCP 전용 필드다.

보기③

긴급 포인터는 긴급 데이터 위치를 가리키는 TCP 전용 필드다.

보기④

ACK 번호는 수신 확인을 위한 TCP 전용 필드다.

관련 이론 정리

전송계층의 두 프로토콜은 제공 기능이 크게 다르다. UDP는 비연결형으로 헤더가 8바이트(출발지·목적지 포트, 길이, 체크섬)로 매우 단순해 오버헤드가 작고 빠르지만, 순서보장·재전송·흐름/혼잡 제어를 제공하지 않는다. TCP는 연결형으로 헤더가 최소 20바이트이며 순서번호·확인번호·윈도 크기·플래그(SYN/ACK/FIN 등)·긴급 포인터 등을 두어 신뢰성·순서보장·흐름제어·혼잡제어를 제공한다. 두 프로토콜의 공통 요소는 포트 번호(프로세스 식별)와 체크섬(오류 검출)이다. 감리·설계에서는 응용의 실시간성·신뢰성 요구에 맞는 전송 프로토콜 선택이 타당한지 점검한다.

TCP vs UDP 헤더 필드:

필드 TCP UDP
출발지/목적지 포트 O O
체크섬 O O
순서번호·ACK번호 O X
긴급 포인터·윈도 O X
실무 적용 사례

네트워크 설계 감리 시 실시간 스트리밍(UDP)과 신뢰성 요구 트랜잭션(TCP) 선택의 적정성, UDP 사용 시 응용 계층의 손실 보완 설계를 검토한다.

감리사 시험 출제 포인트

UDP 헤더 4필드(포트2·길이·체크섬)를 암기하면 공통 필드 문제를 즉시 해결할 수 있다. 순서번호·ACK는 TCP 전용이다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

NFT(Non-Fungible Token, 대체 불가 토큰)는 디지털 창작물에 고유 식별자를 부여해 생성 일시·창작자·소유권·거래 이력을 블록체인에 기록함으로써, 소유권이 이전돼도 원본성과 소유 관계를 투명하게 증명하는 기술이다. 문제의 "등기부등본처럼 원본 소유권을 보증"이라는 서술이 NFT의 핵심 기능과 일치하므로 정답은 ①번이다.

②블록체인은 NFT가 올라타는 기반 분산원장 기술이고, ③암호화폐는 서로 대체 가능(fungible)한 화폐성 토큰이며, ④메타버스는 가상세계 플랫폼 개념으로 소유권 증명 기술이 아니다.

오답 분석

보기①

정답. NFT는 디지털 자산의 고유성과 소유권·이력을 증명한다.

보기②

블록체인은 NFT를 구현하는 기반 기술(분산원장)로, 소유권 증명 '토큰' 자체는 아니다.

보기③

암호화폐는 상호 대체 가능한 토큰으로 개별 원본성을 증명하지 않는다.

보기④

메타버스는 가상공간 플랫폼 개념으로 본 설명과 무관하다.

관련 이론 정리

블록체인은 거래 기록을 블록에 담아 체인으로 연결하고 다수 노드가 분산 저장·합의하는 기술로, 위·변조가 어렵고 투명성이 높다. 그 위에서 대체 가능 토큰(FT)인 암호화폐와 대체 불가 토큰(NFT)이 발행된다. NFT는 각 토큰이 고유 ID와 메타데이터를 가져 서로 교환 불가하며, 디지털 아트·게임 아이템·티켓·증명서 등에 적용된다. 표준으로 ERC-721, ERC-1155 등이 있고, 실제 콘텐츠는 용량 문제로 IPFS 등 외부에 저장하고 링크만 온체인에 두는 경우가 많아 원본 보존·링크 유효성 이슈가 있다. 감리에서는 블록체인 도입 시 합의 알고리즘·성능·개인정보(파기 곤란) 이슈와 오프체인 저장의 무결성을 점검한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

RAID 5는 데이터를 블록 단위로 스트라이핑하고 패리티를 특정 디스크에 몰지 않고 모든 디스크에 분산 저장하는 방식으로, 정식 명칭이 "block-interleaved distributed parity"다. 패리티를 분산했기 때문에 RAID 4에서 발생하던 전용 패리티 디스크의 쓰기 병목이 사라진다. 문제 설명과 정확히 일치하므로 정답은 ④번이다.

①RAID 2는 비트 단위 + 해밍코드, ②RAID 3은 바이트 단위 + 전용 패리티, ③RAID 4는 블록 단위 + 전용 패리티 디스크로, 모두 "패리티 분산"이라는 핵심 조건을 만족하지 않는다.

오답 분석

보기①

보기②

RAID 3은 바이트 단위 스트라이핑에 전용 패리티 디스크를 사용한다.

보기③

RAID 4는 블록 단위지만 패리티를 전용 디스크 한 곳에 저장해 병목이 생긴다(분산 아님).

보기④

정답. RAID 5는 블록 단위 + 분산 패리티로 병목을 해소한다.

관련 이론 정리

RAID 5는 실무에서 가장 널리 쓰이는 레벨 중 하나로, 최소 3개 디스크가 필요하며 전체 용량 중 디스크 1개분이 패리티로 소모된다(N개 디스크 → 용량 N−1). 디스크 1개 장애까지 패리티 연산으로 복구할 수 있으나, 재구축(rebuild) 중에는 성능이 저하되고 추가 장애에 취약하다. 쓰기 시 기존 데이터·패리티를 읽어 새 패리티를 계산해 다시 쓰는 쓰기 패널티(write penalty)가 있다. 대용량 디스크에서는 재구축 시간이 길어 2차 장애 위험이 커지므로, 이중 패리티로 2개 장애를 견디는 RAID 6나 미러링+스트라이핑인 RAID 10을 선택하기도 한다. 감리에서는 데이터 중요도·성능 요구에 따른 레벨 선택과 핫스페어·백업 병행 여부를 점검한다.

RAID 4 vs 5 vs 6:

레벨 스트라이핑 패리티 허용 장애
RAID 4 블록 전용 디스크 1개
RAID 5 블록 분산 1개
RAID 6 블록 이중 분산 2개
실무 적용 사례

스토리지 구성 감리 시 RAID 5의 쓰기 패널티·재구축 위험을 평가하고, 대용량 환경에서 RAID 6·RAID 10 대안 검토와 백업 정책을 점검한다.

감리사 시험 출제 포인트

"블록 단위 + 분산 패리티"=RAID 5, "블록 + 전용 패리티"=RAID 4를 구분한다. 해밍코드는 RAID 2다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

POST는 요청 메시지 몸체(body)에 데이터를 담아 서버로 전송하며, 주로 신규 리소스 생성·등록에 사용된다. 문제의 서술과 정확히 일치하므로 정답은 ②번이다.

보기①

보기②

보기④

관련 이론 정리

HTTP 메소드는 리소스에 대한 행위를 나타낸다. GET(조회)·HEAD(헤더 조회)는 안전(safe)하고 멱등이며, PUT(전체 수정/생성)·DELETE(삭제)는 안전하지 않지만 멱등(여러 번 수행해도 결과 동일)하다. POST(생성·처리 요청)는 안전하지도 멱등하지도 않아 중복 요청 시 리소스가 여러 번 생성될 수 있어 중복 방지 설계가 필요하다. PATCH는 부분 수정이다. GET은 파라미터가 URL에 노출되고 길이 제한·캐싱 대상이 되는 반면, POST는 본문으로 보내 상대적으로 노출이 적으나 그 자체가 암호화는 아니므로 HTTPS가 필요하다. 감리에서는 REST API 설계가 메소드 의미(멱등성 포함)에 맞는지, 민감정보 전송이 GET으로 노출되지 않는지 점검한다.

엔디안(endianness)은 다바이트 데이터를 메모리에 배치하는 순서 규약이다. 빅 엔디안은 최상위 바이트를 낮은 주소에 두어 사람이 읽는 순서와 같고, 네트워크 전송 표준(네트워크 바이트 순서)으로 채택돼 있다. 리틀 엔디안은 최하위 바이트를 낮은 주소에 두며 x86 계열 CPU가 사용한다. ARM 등은 두 방식을 모두 지원하는 바이 엔디안이다. 서로 다른 엔디안 시스템 간에 이진 데이터를 주고받으면 값이 깨지므로, 네트워크 통신에서는 htonl/htons 같은 변환 함수로 바이트 순서를 맞춘다. 파일 포맷·프로토콜 명세에도 엔디안이 규정된다. 감리에서는 이기종 시스템 연계 시 엔디안·정렬(alignment)·자료형 크기 차이로 인한 데이터 변환 오류 대책을 점검한다.

정답 해설

ARP(Address Resolution Protocol)는 네트워크 계층 주소인 IP 주소를 링크(데이터링크) 계층 주소인 MAC 주소로 변환하는 프로토콜이다. 문제 서술과 정확히 일치하므로 정답은 ①번이다.

②RARP는 반대로 MAC 주소로 IP 주소를 알아내는(디스크리스 부팅 등) 프로토콜이고, ③ICMP는 오류 보고·진단(ping, traceroute)용, ④IGMP는 멀티캐스트 그룹 가입·관리용 프로토콜이다.

오답 분석

보기①

정답. ARP는 IP 주소를 MAC 주소로 변환한다.

보기②

RARP는 MAC → IP 변환으로 방향이 반대이며, 현재는 BOOTP/DHCP로 대체됐다.

보기③

ICMP는 오류·상태 메시지 전달용으로 주소 변환 기능이 없다.

보기④

IGMP는 멀티캐스트 그룹 관리용이다(2023년 90번 참조).

관련 이론 정리

같은 LAN에서 IP 패킷을 실제로 전달하려면 목적지의 MAC 주소가 필요하다. ARP는 목적지 IP를 담은 ARP Request를 브로드캐스트하고, 해당 IP를 가진 호스트가 자신의 MAC을 유니캐스트 ARP Reply로 응답한다. 결과는 ARP 캐시에 일정 시간 보관해 반복 질의를 줄인다. 다른 네트워크로 보낼 때는 목적지 호스트가 아니라 게이트웨이의 MAC을 ARP로 얻는다. ARP는 인증이 없어 위조 응답으로 트래픽을 가로채는 ARP 스푸핑에 취약하며, 정적 ARP·DAI(Dynamic ARP Inspection)로 방어한다. IPv6에서는 ARP 대신 ICMPv6 기반 NDP(Neighbor Discovery)를 사용한다. 감리에서는 ARP 스푸핑 방어와 게이트웨이 설정을 점검한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

생성다항식을 비트열로 바꿀 때는 각 차수의 계수를 비트로 표현한다. 최고차수가 x¹⁶이므로 비트열 길이는 17비트(x¹⁶ ~ x⁰)다. 항이 존재하는 차수만 1이 된다.

  • 존재하는 항: x¹⁶, x¹², x⁵, x⁰(=1)
  • 비트 위치(16→0): 16번째=1, 12번째=1, 5번째=1, 0번째=1, 나머지=0

이를 나열하면 1 0001 0000 0010 0001(17비트)이 되어 정답은 ③번이다. 이는 CRC-16-CCITT의 표준 생성다항식이다.

①·②는 16비트로 길이가 맞지 않고(최고차 항 x¹⁶ 누락), ④는 비트 패턴이 항의 위치와 일치하지 않는다.

오답 분석

보기①

16비트로 x¹⁶ 항이 빠져 있고 1의 위치도 12·5·0차와 맞지 않는다.

보기②

0과 1이 반전된 형태로 생성다항식 항과 일치하지 않는다.

보기③

정답. 17비트로 16·12·5·0차 위치만 1이다.

보기④

17비트이지만 1의 위치가 해당 차수와 어긋난다(반전 형태).

관련 이론 정리

CRC는 데이터를 계수가 0/1인 다항식으로 보고, 약속된 생성다항식 G(x)로 나눈 나머지를 검사값(FCS)으로 붙여 전송한다. 수신 측은 같은 방식으로 나누어 나머지가 0이면 오류 없음으로 판정한다. 생성다항식의 최고차수 n은 검사비트 수 n을 뜻하고, 비트열 표현은 n+1비트가 된다. 표준 다항식으로 CRC-16-CCITT(x¹⁶+x¹²+x⁵+1), CRC-32(이더넷) 등이 있다. CRC는 XOR 기반 시프트 레지스터로 하드웨어 구현이 쉽고, 연속된 오류(버스트 에러)를 강하게 검출해 이더넷·저장장치·통신에 널리 쓰인다. 다만 오류 검출만 하고 정정은 못 하므로 재전송(ARQ)과 함께 쓴다. 감리에서는 전송 계층별 오류 검출 방식과 재전송 정책의 적정성을 점검한다.

다항식 ↔ 비트열:

차수 16 12 5 0
계수 1 1 1 1
그 외 0 0 0 0
실무 적용 사례

통신·저장 시스템 감리 시 채택된 CRC 다항식(CRC-16/32)의 검출 능력이 요구 신뢰성에 부합하는지, 재전송 메커니즘이 갖춰졌는지 검토한다.

MAC 주소는 48비트로 앞 24비트가 제조사 식별자(OUI), 뒤 24비트가 장치 고유번호다. 첫 바이트의 최하위 비트인 I/G 비트가 0이면 개별(유니캐스트), 1이면 그룹(멀티캐스트)이며, 모든 비트가 1이면 브로드캐스트다. 한편 IP 주소 체계에서 IPv4는 유니캐스트·멀티캐스트·브로드캐스트를, IPv6는 유니캐스트·멀티캐스트·애니캐스트를 지원하고 브로드캐스트를 폐지했다. 애니캐스트는 같은 주소를 여러 서버에 부여해 라우팅상 가장 가까운 노드로 보내는 방식으로 DNS 루트 서버·CDN에 활용된다. 즉 애니캐스트는 3계층 개념임을 구분해야 한다. 감리에서는 주소 체계 설계와 멀티캐스트·브로드캐스트 트래픽 통제를 점검한다.

IPv6(3계층) 유니·멀티·애니캐스트(브로드캐스트 없음)
실무 적용 사례

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

recv()는 연결이 설정된 소켓(TCP 등 연결성 서비스)에서 상대가 보낸 데이터를 읽어오는 함수다. 문제의 "상대 프로세스가 전송한 데이터를 소켓을 통해 읽으며 연결성 서비스에 사용"이라는 서술과 정확히 일치하므로 정답은 ③번이다.

①connect()는 클라이언트가 서버에 연결을 요청하는 함수, ②listen()은 서버가 연결 요청을 대기(수신 대기열 설정)하는 함수, ④sendto()는 비연결형(UDP)에서 목적지 주소를 지정해 송신하는 함수다(수신은 recvfrom).

보기②

관련 이론 정리

소켓은 응용 프로그램이 전송계층을 이용하는 API다. TCP(연결형) 흐름은 서버가 socket→bind→listen→accept로 연결을 받고, 클라이언트는 socket→connect로 접속한 뒤, 양측이 send/recv(또는 write/read)로 데이터를 주고받고 close로 종료한다. UDP(비연결형)는 연결 설정 없이 socket→bind 후 sendto/recvfrom으로 매 전송마다 상대 주소를 지정한다. 따라서 함수 이름만으로 연결형/비연결형을 구분할 수 있다(send/recv=연결형, sendto/recvfrom=비연결형). accept()는 대기열의 연결 요청을 수락해 새 소켓을 반환한다. 감리에서는 소켓 통신의 예외·타임아웃 처리, 리소스(소켓) 누수, 보안(TLS 적용) 여부를 점검한다.

함수 역할 유형

listen/accept 연결 대기/수락(서버) 연결형
send/recv 송신/수신 연결형
sendto/recvfrom 송신/수신(주소 지정) 비연결형
실무 적용 사례

통신 모듈 감리 시 소켓 타임아웃·재연결 처리, 소켓 누수, 대용량 동시접속 시 자원 한계(파일 디스크립터) 관리를 점검한다.

감리사 시험 출제 포인트

정답 해설

BGP는 네 가지 메시지를 사용한다: Open(피어 세션 개설), Update(경로 정보 광고·철회), Keepalive(세션 유지), Notification(오류 발생 시 통지 후 연결 종료).

①은 "자신의 BGP 정보를 상대방에게 알려주기 위해 Notification 메시지를 사용한다"고 했으나, 경로 정보를 알리는 것은 Update 메시지이고 Notification은 오류 통지용이므로 틀린 설명이다. 따라서 가장 적절하지 않은 것은 ①번이다.

보기②

보기③

관련 이론 정리

BGP4는 인터넷의 사실상 표준 경로 벡터(path vector) 프로토콜로, 단순 홉 수가 아니라 AS 경로(AS_PATH) 등 다양한 속성(LOCAL_PREF, MED, NEXT_HOP, ORIGIN)을 기준으로 정책적 경로 선택을 한다. AS_PATH에 자신의 AS가 있으면 경로를 거부해 루프를 방지한다. 신뢰적 전달을 위해 TCP 179를 쓰고, 초기에는 전체 경로표를 교환한 뒤 이후에는 변경분만 Update로 증분 전달하며 Keepalive로 세션을 유지한다. 내부 라우팅용 IGP(OSPF·RIP)와 달리 BGP는 조직(AS) 간 정책 라우팅을 담당한다. 잘못된 경로 광고로 트래픽이 탈취되는 BGP 하이재킹 위험이 있어 RPKI·필터링으로 방어한다. 감리에서는 대외 회선 라우팅 정책·필터링·이중화를 점검한다.

감리사 시험 출제 포인트

DVMRP는 거리벡터 기반 멀티캐스트 라우팅 프로토콜이다.

관련 이론 정리

멀티캐스트는 하나의 송신자가 특정 그룹(클래스 D, 224.0.0.0~239.255.255.255)에 속한 다수 수신자에게 효율적으로 전달하는 방식이다. 구조는 두 층으로 나뉜다. 호스트-라우터 구간은 IGMP(IPv6는 MLD)가 담당해 그룹 가입/탈퇴를 관리하고, 라우터-라우터 구간은 PIM·DVMRP·MOSPF·CBT 같은 멀티캐스트 라우팅 프로토콜이 배포 트리(소스 기반 트리 또는 공유 트리)를 구성한다. IGMP는 v1(가입/질의), v2(탈퇴 메시지 추가), v3(소스 지정 SSM 지원)로 발전했다. L2 구간에서는 IGMP 스누핑으로 불필요한 플러딩을 줄인다. 감리에서는 IPTV·실시간 방송 서비스의 멀티캐스트 설계와 스누핑·대역 관리를 점검한다.

멀티캐스트 프로토콜 구분:

구간 프로토콜
호스트↔라우터 IGMP(IPv6: MLD)
라우터↔라우터 PIM, DVMRP, MOSPF, CBT
실무 적용 사례

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

CHAP은 비밀번호를 네트워크에 직접 보내지 않는 챌린지-응답(challenge-response) 방식 인증 프로토콜이다. 서버가 난수 챌린지를 보내면 클라이언트는 비밀번호와 챌린지를 해시한 응답값만 전송하고, 서버가 같은 계산으로 검증한다. 또한 연결 중 주기적으로 재인증해 보안성이 높다.

④는 "클라이언트가 인증요청 패킷에서 사용자 이름과 비밀번호를 보낸다"고 서술했는데, 이는 PAP(Password Authentication Protocol)의 방식(평문 ID/PW 전송)이므로 CHAP 설명으로는 틀리다. 따라서 정답은 ④번이다. ①②③은 모두 옳은 설명이다.

오답 분석

보기①

옳음. LCP는 PPP 링크의 설정·구성·품질 확인·해제를 담당한다.

보기②

옳음. NCP는 상위 네트워크 계층 프로토콜을 설정·구성한다.

보기③

옳음. IPCP는 IP 설정을 담당하는 NCP의 한 종류다.

보기④

정답(부적절). 사용자 이름·비밀번호를 그대로 보내는 것은 PAP다. CHAP은 챌린지-응답 해시를 사용한다.

관련 이론 정리

PPP는 점대점 링크에서 다양한 상위 프로토콜을 실어 나르는 데이터링크 프로토콜로, 세 구성요소를 갖는다: 프레임 캡슐화(HDLC 유사), LCP(링크 설정·유지·해제, 옵션 협상), NCP(상위 프로토콜별 설정, 예: IP용 IPCP). 연결 과정은 Dead→Establish(LCP)→Authenticate(PAP/CHAP)→Network(NCP)→Open→Terminate 순이다. 인증에서 PAP는 2-way로 ID/PW를 평문 전송해 취약하고, CHAP은 3-way 챌린지-응답과 주기적 재인증으로 재전송 공격에 강하다. 최근에는 EAP 기반 인증도 쓰인다. 감리에서는 원격 접속 링크의 인증 방식이 평문 전송이 아닌지, 암호화(터널링) 적용 여부를 점검한다.

PAP vs CHAP:

항목 PAP CHAP
방식 ID/PW 평문 전송 챌린지-응답 해시
핸드셰이크 2-way 3-way
재인증 없음 주기적
보안성 낮음 높음
실무 적용 사례

원격 접속·전용회선 감리 시 PAP 같은 평문 인증 사용 여부를 점검하고 CHAP/EAP·암호화 터널(VPN) 적용을 권고한다.

감리사 시험 출제 포인트

정답(부적절). kNN은 레이블을 사용하는 지도학습이다. 비지도학습은 k-means 같은 군집화다.

관련 이론 정리

kNN은 대표적 인스턴스 기반·게으른 학습 알고리즘으로, 별도의 학습(모델 파라미터 추정) 과정 없이 학습 데이터를 저장해 두었다가 예측 시 거리(유클리드·맨해튼 등)를 계산해 가장 가까운 k개의 이웃을 찾고 다수결(분류) 또는 평균(회귀)으로 결과를 낸다. k가 작으면 잡음에 민감(과적합), 크면 경계가 뭉개져 과소적합되므로 교차검증으로 정한다. 특성 스케일 차이가 거리 계산을 왜곡하므로 정규화가 필수이며, 차원이 높아지면 거리 개념이 무의미해지는 차원의 저주에 취약하다. 이름이 비슷한 k-means는 레이블 없이 군집을 찾는 비지도학습으로 반드시 구분해야 한다. 감리에서는 모델 선택·전처리(정규화)·평가 방법의 타당성을 점검한다.

kNN vs k-means:

레이블 필요 불필요
k의 의미 이웃 수 군집 수
실무 적용 사례

분류 모델 감리 시 kNN 적용의 전처리(정규화)·k 선정 근거, 대용량 데이터에서의 예측 지연과 인덱싱(KD-Tree 등) 대책을 검토한다.

감리사 시험 출제 포인트

kNN=지도학습, k-means=비지도학습을 확실히 구분한다. 이름 유사성을 이용한 함정이 자주 출제된다.

①은 틀림: 전가상화에서 게스트는 하드웨어에 직접 접근하지 못하고 하이퍼바이저가 명령을 가로채 에뮬레이션한다. ②도 틀림: 전가상화는 게스트 커널을 수정할 필요가 없다(수정이 필요한 쪽은 반가상화). ④도 틀림: 명령어 트랩·에뮬레이션 오버헤드 때문에 일반적으로 반가상화가 전가상화보다 성능이 우수하다(단, 하드웨어 지원 가상화 등장 이후 격차는 축소).

보기①

보기②

부적절. 전가상화는 게스트 커널 코드 변경이 필요 없다는 것이 핵심 장점이다.

보기④

부적절. 일반적으로 반가상화가 전가상화보다 성능 오버헤드가 작다.

관련 이론 정리

하이퍼바이저 기반 가상화는 구현 방식에 따라 나뉜다. 전가상화(Full virtualization)는 하드웨어를 완전히 에뮬레이션해 게스트 OS를 수정 없이 그대로 구동할 수 있으나, 특권 명령을 트랩·변환하는 오버헤드가 있다(예: 초기 VMware). 반가상화는 게스트 커널을 수정해 하이퍼콜로 하이퍼바이저와 협력하므로 오버헤드가 작아 성능이 좋지만, 커널 수정이 불가능한 OS(예: 폐쇄형 OS)에는 적용이 어렵다(예: Xen). 이후 Intel VT-x·AMD-V 같은 하드웨어 지원 가상화가 등장해 수정 없이도 높은 성능을 내면서 두 방식의 성능 차이가 줄었다. 하이퍼바이저는 하드웨어 위에 바로 올라가는 Type 1(베어메탈)과 호스트 OS 위의 Type 2로도 구분된다. 감리에서는 가상화 방식 선택, 자원 과할당, 가상화 계층 보안을 점검한다.

성능 상대적 낮음 상대적 높음
대표 예 VMware(초기) Xen
적용성 넓음 수정 가능한 OS
실무 적용 사례

가상화 인프라 감리 시 채택된 가상화 방식과 하드웨어 지원(VT-x/AMD-V) 활용, 자원 과할당(오버커밋)에 따른 성능 저하 위험을 점검한다.

정답 해설

ReLU(Rectified Linear Unit, f(x)=max(0,x))는 신경망 뉴런의 출력에 비선형성을 부여하는 활성함수다. 따라서 정답은 ②번이다.

①RMSProp, ③SGD, ④Adam은 모두 손실함수를 최소화하기 위해 가중치를 갱신하는 최적화(optimizer) 알고리즘으로 활성함수가 아니다. 활성함수와 옵티마이저는 역할이 전혀 다르므로 혼동하지 않아야 한다.

보기④

관련 이론 정리

신경망 학습에는 역할이 다른 두 부류의 기법이 관여한다. 활성함수는 각 뉴런의 출력에 비선형을 도입해 다층 신경망이 복잡한 함수를 표현하게 한다(없으면 층을 쌓아도 선형 변환에 불과). 대표적으로 시그모이드(0~1, 기울기 소실 문제), tanh(−1~1), ReLU(계산이 단순하고 기울기 소실을 완화해 은닉층 표준), Leaky ReLU, 출력층 다중분류용 Softmax가 있다. 최적화 기법(옵티마이저)은 손실의 기울기를 이용해 가중치를 갱신하는 방법으로, 기본 경사하강법에서 SGD(미니배치), 모멘텀, RMSProp(적응적 학습률), Adam(모멘텀+RMSProp)으로 발전했다. 감리에서는 모델 구조·하이퍼파라미터 선택 근거와 학습 수렴·과적합 관리(검증셋·조기종료)를 점검한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

전이 학습(Transfer Learning)은 대량 데이터로 미리 학습된(pre-trained) 모델의 지식을 가져와 새로운(주로 데이터가 적은) 과제의 모델 학습에 활용하는 기법이다. 문제의 "학습된 모델을 이용하여 새로운 모델을 학습시키는 것"이라는 서술과 정확히 일치하므로 정답은 ②번이다.

①사전 학습은 대규모 데이터로 기반 모델을 만드는 앞 단계 과정 자체를 가리키는 말로, "학습된 모델을 이용해 새 모델을 학습"하는 행위와는 구분된다. ③지도 학습과 ④강화 학습은 학습 데이터·보상 구조에 따른 학습 패러다임 분류로 층위가 다르다.

오답 분석

보기①

사전 학습은 기반 모델을 미리 학습시키는 단계로, 그 모델을 활용하는 과정(전이)과 구분된다.

보기②

정답. 전이 학습은 사전학습 모델의 지식을 새 과제로 옮겨 활용한다.

보기③

지도 학습은 레이블된 데이터로 학습하는 패러다임 분류다.

보기④

강화 학습은 보상을 최대화하도록 행동을 학습하는 패러다임이다.

관련 이론 정리

전이 학습은 데이터·연산 자원이 부족한 현실적 제약을 극복하는 표준 기법이다. 절차는 대개 ① 대규모 범용 데이터로 사전학습(pre-training)된 모델(ImageNet 학습 CNN, BERT·GPT 등)을 가져와, ② 새 과제 데이터로 파인튜닝(fine-tuning)하는 방식이다. 이때 하위 층은 일반적 특징(엣지·기본 문법)을 담고 있어 동결(freeze)하고 상위 층만 재학습하거나, 전체를 낮은 학습률로 재학습한다. 특징 추출기로만 쓰는 방식도 있다. 원 과제와 대상 과제의 도메인이 지나치게 다르면 성능이 떨어지는 부정적 전이(negative transfer)가 발생할 수 있다. 최근에는 소수 파라미터만 조정하는 LoRA 등 효율적 기법도 쓰인다. 감리에서는 사전학습 모델의 라이선스·편향·보안과 파인튜닝 데이터 적정성을 점검한다.

관련 개념 구분:

용어 의미

파인튜닝 전이 학습의 구체 방법(재학습)
실무 적용 사례

AI 도입 감리 시 공개 사전학습 모델 활용의 라이선스 준수, 편향·보안 검증, 파인튜닝 데이터의 품질·개인정보 처리 적정성을 점검한다.

감리사 시험 출제 포인트

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

포트 번호는 전송계층이 어느 응용 프로세스로 데이터를 전달할지 식별하는 값으로, IP가 호스트 간(host-to-host) 전달을 담당한다면 포트는 프로세스 간(process-to-process) 전달을 담당한다. ②가 이를 정확히 서술하므로 정답은 ②번이다.

①은 틀림: 포트 번호는 16비트(0~65535)다. ③도 틀림: TCP와 UDP는 서로 독립된 포트 공간을 가져 같은 번호(예: 53)를 각각 사용할 수 있다. ④도 틀림: Well-known 포트는 서버가, 동적(임시) 포트는 클라이언트가 주로 사용하므로 설명이 뒤바뀌었다.

오답 분석

보기①

부적절. 포트 번호는 32비트가 아니라 16비트다.

보기②

정답. 포트는 프로세스 간 전달을 위한 식별자다.

보기③

부적절. TCP·UDP는 포트 공간이 독립적이라 같은 번호를 동시에 쓸 수 있다.

보기④

부적절. Well-known(서버)과 동적 포트(클라이언트)의 사용 주체가 뒤바뀌었다.

관련 이론 정리

TCP/IP에서 통신 종단을 식별하는 요소는 계층별로 다르다. MAC 주소는 물리적 인터페이스, IP 주소는 호스트, 포트 번호는 호스트 내 응용 프로세스를 식별하며, 이 조합(프로토콜, 출발지 IP·포트, 목적지 IP·포트)을 소켓 5-튜플이라 한다. 포트는 16비트로 0~65535이며 IANA가 잘 알려진 포트(0~1023, 서버 표준 서비스), 등록 포트(1024~49151), 동적/사설 포트(49152~65535, 클라이언트 임시 포트)로 구분한다. 서버는 고정된 well-known 포트에서 대기하고 클라이언트는 접속 시마다 임시 포트를 할당받는다. TCP와 UDP는 별도의 포트 공간을 가지므로 DNS가 TCP 53·UDP 53을 모두 쓰는 것이 가능하다. 감리에서는 불필요 포트 개방 최소화와 방화벽 정책을 점검한다.

포트 번호 범위:

범위 구분 주 사용
0~1023 Well-known 서버 표준 서비스
1024~49151 등록 포트 특정 애플리케이션
49152~65535 동적/사설 클라이언트 임시 포트
실무 적용 사례

방화벽·서버 보안 감리 시 개방 포트 최소화, 서비스-포트 매핑 적정성, 임시 포트 범위 설정과 세션 고갈 위험을 점검한다.
