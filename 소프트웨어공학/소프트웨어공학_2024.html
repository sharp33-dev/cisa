정답 해설

inception → elaboration → construction → transition은 애자일의 Sprint 사이클이 아니라 RUP(Rational Unified Process)의 4단계 개발 단계이다.

강의자료(소프트웨어공학_Part_I)에 따르면:
- RUP 4단계: 도입(Inception) → 정련(Elaboration) → 구축(Construction) → 전환(Transition)
- Scrum Sprint: 스프린트 계획 → 스프린트 실행 → 스프린트 검토 → 스프린트 회고 (2~4주 반복)
- XP 반복: Story → Task → Test → Code → Integrate 주기 반복

나머지 보기 분석:
- ①: XP의 On-site Customer(고객이 팀원으로 참여) 개념 → 정확
- ②: 애자일의 점진적·반복적 release 특징 → 정확
- ③: Scrum의 표준 프로세스 흐름 → 정확

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

COCOMO II 재사용 모델의 AAF(Adaptation Adjustment Factor) 공식:

AAF = 0.4×DM + 0.3×CM + 0.3×IM

여기서 가중치:
- DM(설계 변경률): 0.4 → 가장 높은 가중치
- CM(코드 변경률): 0.3
- IM(통합 노력률): 0.3

③번 설명 — "코드 변경률(CM)에 대한 가중치(0.3)가 설계 변경률(DM)의 가중치(0.4)보다 높다" → 오류

실제로는 DM(0.4) > CM(0.3) = IM(0.3)이다. 설계 변경률이 코드 변경률보다 가중치가 더 높다.

나머지 보기:
- ①: COCOMO II는 규모의 경제 반영. SF(Scale Factor) 합이 5를 초과하면 규모의 비경제(비선형 증가) 발생 → "경제 효과를 가지지 못한다"는 경우도 있어 논란 여지가 있으나, CM(0.3)>DM(0.4)는 명백한 오류
- ②: FP에서 EI(외부입력)와 EQ(외부조회)는 동일한 가중치 구조를 가짐 → 정확
- ④: DM=0 또는 CM=0이면 기존 소프트웨어를 전혀 수정하지 않는 것이므로 SU=0 → 정확

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

문제가 요구하는 것: "브랜치 생성 + HEAD를 그 브랜치로 이동"

Git 브랜치 관련 명령어:

명령어 기능
git branch bugFix01 브랜치만 생성 (HEAD는 그대로)
git checkout bugFix01 브랜치로 전환 (HEAD 이동)
git checkout -b bugFix01 브랜치 생성 + 전환 동시에
git merge bugFix01 브랜치 병합
git add bugFix01 파일 스테이징 (브랜치와 무관)

"새로운 브랜치를 생성한 다음, HEAD가 이 브랜치를 가리키게"

→ 두 단계: git branch bugFix01 (생성) + git checkout bugFix01 (전환)

→ 또는 한 번에: git checkout -b bugFix01

문제에서 "생성한 다음"이라는 표현은 브랜치는 이미 생성되었거나 생성+전환을 함께 수행한다는 의미. HEAD를 새 브랜치로 이동시키는 핵심 명령어는 git checkout bugFix01.

현재 Git(2.23+)에서는 git switch bugFix01도 동일 기능이나, 보기에는 checkout이 주어졌다.

오답 분석

보기①(git add)

파일을 스테이징 영역에 추가하는 명령. 브랜치와 무관.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

단계별 포인터 연산 분석:

ary[2][4] = {{10,20,30,40}, {50,60,70,80}}
int (*ptr)[4] = ary;  // ptr은 4개 int를 가리키는 포인터 배열

a 계산: *(*(ptr + 1) + 3)
- ptr + 1: 두 번째 행(ary[1]) 가리킴 = {50,60,70,80}
- *(ptr + 1): ary[1]의 첫 번째 원소 주소 = &ary[1][0]
- *(ptr + 1) + 3: &ary[1][3] = 80의 주소
- *(*(ptr + 1) + 3) = 80

b 계산: **ptr++
- 후위증가: 먼저 현재 ptr 값으로 평가, 이후 ptr 증가
- **ptr = *ptr[0] = ary[0][0] = 10
- 이후 ptr++ → ptr이 ary[1]을 가리킴

c 계산: **ptr
- ptr은 이미 ary[1]을 가리킴 (b에서 증가됨)
- **ptr = ary[1][0] = 50

결과: a + b + c = 80 + 10 + 50 = 140

정답 해설

Cause-effect Graphing의 Cause 제약 조건 종류:

제약 기호 의미
Exclusive(E) E 두 Cause 중 최대 하나만 참 가능
Inclusive(I) I 최소 하나는 반드시 참
One and only one(O) O 정확히 하나만 참
Requires(R) R 하나가 참이면 다른 하나도 반드시 참

Mask(마스크)는 Cause 간 제약이 아니라 Effect(결과) 간의 제약 조건이다. Mask는 한 Effect가 참일 때 다른 Effect를 무효화(마스크)하는 제약으로, Cause 제약이 아닌 Effect 제약에 해당한다.

따라서 Cause간의 제약이 아닌 것 = Mask → ①번

Cause O 제약: 정확히 하나만 참. Cause 간 제약 맞음.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

Base Choice 방법 공식:

테스트케이스 수 = 1 + Σ(각 파라미터의 값 수 - 1)

계산:
- 글꼴 이름: 4개 → (4-1) = 3
- 글꼴 크기: 5개 → (5-1) = 4
- 스타일: 3개 → (3-1) = 2

총 = 1(기본 케이스) + 3 + 4 + 2 = 10개

Base Choice 방법 원리:
1. 각 파라미터에서 기본값(base value) 하나씩 선택
2. 기본값들로 구성된 기본 테스트케이스 1개 생성
3. 각 파라미터에 대해 기본값이 아닌 나머지 값들을 순차적으로 기본 케이스에 적용

예시: 기본 케이스 = (궁서, 10, 보통)
- 글꼴 이름 변형: (굴림,10,보통), (바탕,10,보통), (고딕,10,보통) → 3개
- 글꼴 크기 변형: (궁서,12,보통), (궁서,14,보통), (궁서,16,보통), (궁서,18,보통) → 4개
- 스타일 변형: (궁서,10,밑줄), (궁서,10,기울임) → 2개

합계: 1 + 3 + 4 + 2 = 10개

오답 분석

가장 큰 파라미터(크기 5개)만 계산한 오류.

전체 조합 수: 4×5×3=60. 이는 전체 조합 테스트(Exhaustive Test)로 Base Choice가 아님.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

Spring MVC 요청 처리 흐름:

Client
  → ① Request → (가) DispatcherServlet
  → ② (가) → (나) HandlerMapping (어떤 Controller?)
  → ③ HandlerMapping → (가) (Controller 정보 반환)
  → ④ (가) → Controller (실제 처리)
  → ⑤ Controller → (다) ModelAndView 반환
  → ⑥ (다) → (가) DispatcherServlet
  → ⑦ (가) → (라) ViewResolver (뷰 이름 → 실제 뷰)
  → ⑧ (라) → View
  → ⑨ View → Response → Client

이미지에서 (가)는 중앙 허브 역할 = DispatcherServlet (Front Controller)

따라서:
- (가) DispatcherServlet: 모든 요청의 중앙 진입점
- (나) HandlerMapping: 요청 URL에 맞는 Controller 찾기
- (다) ModelAndView: Controller의 처리 결과(데이터+뷰이름)
- (라) ViewResolver: 뷰 이름을 실제 View 파일로 변환

컴포넌트 역할

HandlerMapping URL → Controller 매핑
Controller 실제 비즈니스 로직 처리

ViewResolver View 이름 → 실제 View 파일

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

DIP(Dependency Inversion Principle) 의존성 역전 원칙:

"고수준 모듈은 저수준 모듈에 의존해서는 안 된다. 둘 다 추상화에 의존해야 한다."
"추상화는 세부사항에 의존해서는 안 된다. 세부사항이 추상화에 의존해야 한다."

문제 설명: "추상 클래스/인터페이스가 구체 클래스보다 덜 변하므로, 가능한 추상 클래스에 의존해야 한다" → DIP의 핵심 내용

오답 분석

보기①(ISP)

인터페이스 분리 원칙: 클라이언트가 사용하지 않는 메서드를 포함한 인터페이스를 강제받지 않아야 함.

보기③(OCP)

개방/폐쇄 원칙: 확장에는 열려 있고, 수정에는 닫혀 있어야 함.

보기④(LSP)

리스코프 치환 원칙: 자식 클래스는 부모 클래스를 완전히 대체할 수 있어야 함.

  2. ADD/CHGA/CFP는 이후 기준(VAFA), DEL은 이전 기준(VAFB)
  3. DEL은 빼지 않고 더함 (삭제 작업도 비용 발생)
  4. VAFA = 개선 후 조정인자, VAFB = 개선 전 조정인자
  5. FP 5가지 요소: EI(외부입력), EO(외부출력), EQ(외부조회), ILF(내부논리파일), EIF(외부인터페이스파일)

IFPUG EFP(개선 기능점수) 공식:

EFP = [(ADD + CHGA + CFP) × VAFA] + (DEL × VAFB)

각 변수 의미:
- ADD: 추가된 기능의 미조정 기능점수
- CHGA: 수정된 기능의 미조정 기능점수 (수정 후 값)
- CFP: 데이터 변환 기능점수
- DEL: 삭제된 기능의 미조정 기능점수
- VAFA: 개선 프로젝트 종료 후 값 조정인자
- VAFB: 개선 프로젝트 이전 값 조정인자

핵심 포인트: DEL(삭제 기능)은 개선 프로젝트 이전 상태의 VAFB를 적용하여 더한다(양수).
- 삭제된 기능도 해당 프로젝트에서 처리된 것이므로 비용 산정에 포함
- VAFB(이전 값 조정인자)를 적용하여 DEL × VAFB를 더함
