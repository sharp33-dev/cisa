영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

리버스 엔지니어링(역공학)은 완성된 장치·프로그램을 분석해 내부 구조와 동작 원리를 역으로 파악하는 과정이다. 보안 분야에서는 악성코드 분석, 취약점 연구, 상호운용성 확보, 소스가 없는 레거시 시스템의 유지보수에 활용된다. 문제 설명과 정확히 일치하므로 정답은 ④번이다.

①백도어는 정상 인증을 우회해 시스템에 접근할 수 있도록 몰래 심어 둔 통로, ②포맷 스트링은 사용자 입력이 형식 문자열로 사용될 때 발생하는 취약점, ③문맥교환은 CPU가 실행 중인 프로세스를 전환할 때 상태를 저장·복원하는 운영체제 개념으로 모두 다르다.

오답 분석

보기①

백도어는 인증을 우회하는 숨겨진 접근 경로로 분석 과정이 아니다.

보기②

포맷 스트링은 취약점 유형이며 구조 분석 기법이 아니다.

보기③

문맥교환은 OS의 프로세스 전환 메커니즘으로 보안 분석과 무관하다.

보기④

정답. 완성품을 분석해 원리를 도출하는 과정이 리버스 엔지니어링이다.

관련 이론 정리

역공학은 순공학(설계 → 구현)의 반대 방향으로 구현물 → 설계·원리를 도출한다. 소프트웨어 역공학의 절차는 실행 파일의 구조(PE/ELF 헤더)·문자열·API 호출을 살피는 정적 분석(IDA Pro, Ghidra)과, 실제 실행하며 흐름을 추적하는 동적 분석(OllyDBG, x64dbg, 샌드박스)으로 나뉜다. 악성코드는 이를 방해하기 위해 패킹·난독화·안티디버깅 기법을 쓴다. 역공학은 악성코드 대응·취약점 진단·상호운용성 확보 같은 정당한 목적에는 필요하지만, 타인의 소프트웨어를 무단 분석해 복제·우회하면 저작권법·라이선스 위반이 될 수 있어 허용 범위를 지켜야 한다. 반대로 자사 제품 보호를 위해서는 난독화·코드 서명·무결성 검증을 적용한다. 감리에서는 악성코드 분석 체계와 소스 보호 대책을 점검한다.

역공학 관련 개념:

동적 분석 실행하며 동작 추적(디버거)
방해 기법 패킹·난독화·안티디버깅

침해사고 대응 감리 시 악성코드 역공학 분석 역량·절차와, 자사 애플리케이션의 난독화·위변조 방지 적용 여부를 점검한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

도전-응답(Challenge-Response) 방식은 서버가 난수(challenge)를 생성해 클라이언트에 보내면, 클라이언트가 그 값과 공유 비밀키로 응답값(OTP)을 계산해 회신하는 방식이다. 문제 설명이 바로 이 흐름이므로 정답은 ③번이다.

②시간 동기화 방식(TOTP)은 서버·토큰이 현재 시각을 기준으로 각자 OTP를 생성하며 별도 전송이 없다. ④이벤트 동기화 방식(HOTP)은 카운터 값이 증가할 때마다 OTP를 생성한다. ①S/KEY는 초기 비밀값에 해시를 여러 번 적용한 해시 체인을 역순으로 사용하는 방식이다.

오답 분석

보기①

S/KEY는 해시 체인을 역순으로 사용하는 방식으로 서버가 난수를 보내는 구조가 아니다.

보기②

시간 동기화 방식은 현재 시각을 기준으로 양측이 독립 생성하며 난수 전송이 없다.

보기③

정답. 서버가 난수를 보내고 클라이언트가 이를 이용해 OTP를 만드는 방식이다.

보기④

이벤트 동기화 방식은 카운터 증가에 따라 OTP를 생성한다.

관련 이론 정리

OTP는 재사용 공격(replay attack)을 막기 위해 매번 다른 비밀번호를 쓰는 인증 방식으로, 지식 기반 인증의 약점을 보완하는 소유 기반 인증 수단이다. 생성 방식은 ① 시간 동기화(TOTP): 30초 등 일정 주기의 시각을 입력으로 사용, 서버·토큰의 시간 동기화가 필요하고 오차 보정(window)이 필요하다. ② 이벤트 동기화(HOTP): 버튼을 누를 때마다 카운터가 증가하며, 사용자가 여러 번 눌러 카운터가 어긋나면 재동기화가 필요하다. ③ 도전-응답: 서버 난수를 사용해 동기화 문제가 없으나 추가 통신 절차가 필요하다. 다중 인증(MFA)에서 OTP는 대표적인 두 번째 요소이며, 최근에는 SMS OTP의 탈취 위험(심 스와핑) 때문에 앱 기반 OTP나 FIDO 인증이 권장된다. 감리에서는 관리자·외부 접속에 MFA 적용 여부를 점검한다.

OTP 생성 방식 비교:

방식 입력값
시간 동기화(TOTP) 현재 시각
이벤트 동기화(HOTP) 카운터
도전-응답 서버가 보낸 난수
S/KEY 해시 체인
실무 적용 사례

인증 체계 감리 시 관리자 계정·VPN 접속에 OTP 등 다중 인증이 적용됐는지, SMS OTP 의존도와 대체 수단(앱·FIDO)을 점검한다.

감리사 시험 출제 포인트

"서버가 난수를 보낸다" = 도전-응답, "시각 기준" = TOTP, "카운터" = HOTP의 입력값 구분이 핵심이다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

전자서명은 문서의 해시값(다이제스트)에 개인키를 적용해 생성되므로, 문서가 다르면 해시가 달라 서명값도 완전히 달라진다. 따라서 한 문서의 서명을 복사해 다른 문서에 붙이면 검증 시 해시가 일치하지 않아 즉시 위조로 판별된다. 이것이 전자서명의 재사용 불가(재사용 방지) 성질이다.

④는 "복사하여 다른 문서의 서명으로 사용할 수 있다"고 했으므로 이 성질에 정면으로 어긋나며 정답은 ④번이다. ①(변경 불가), ②(개인키 소유자만 생성 → 서명자 인증), ③(부인 방지)은 모두 옳다.

오답 분석

보기①

옳음. 서명 후 원본이나 서명을 수정하면 검증에 실패한다(변경 불가).

보기②

옳음. 개인키 소유자만 생성할 수 있어 서명자를 인증할 수 있다.

보기③

옳음. 제3자가 공개키로 검증할 수 있어 부인 방지가 성립한다.

보기④

정답(부적절). 서명은 해당 문서의 해시에 종속되므로 다른 문서에 재사용할 수 없다.

관련 이론 정리

전자서명이 갖춰야 할 요건은 다섯 가지로 정리된다. 위조 불가(unforgeable): 개인키 없이는 만들 수 없다. 서명자 인증(authentic): 누가 서명했는지 확인 가능하다. 재사용 불가(not reusable): 한 문서의 서명을 다른 문서에 쓸 수 없다. 변경 불가(unalterable): 서명 후 문서를 바꾸면 검증에 실패한다. 부인 방지(non-repudiation): 서명자가 서명 사실을 부인할 수 없다. 이런 성질은 서명이 문서 해시 + 개인키의 함수이기 때문에 자연스럽게 성립한다. 다만 공개키가 실제 그 사람의 것임은 서명 자체로 보증되지 않으므로 CA가 발급한 인증서와 PKI가 필요하며, 서명 시점 증명을 위해 타임스탬프를 함께 쓴다. 감리에서는 전자서명 검증 로직과 인증서 유효성(폐기 목록·OCSP) 확인 절차를 점검한다.

정답 해설

ECC의 안전성은 소인수분해가 아니라 타원곡선 위에서의 이산대수 문제(ECDLP)에 기반한다. 즉 타원곡선 위의 점 P와 kP가 주어졌을 때 스칼라 k를 구하기 어렵다는 성질을 이용한다.

①은 "소인수분해 문제에 기반"이라고 했는데 이는 RSA의 근거이므로 틀렸다. 따라서 정답은 ①번이다.

②(ECDH로 키 교환, ECDSA로 전자서명 가능), ③(같은 안전성 대비 연산량·키 길이가 작음), ④(ECC 256비트 ≈ RSA 3072비트 수준의 안전성)는 모두 옳다.

보기②

옳음. ECDH(키 교환)와 ECDSA(전자서명)로 활용된다.

보기④

옳음. 짧은 키로 RSA와 대등한 보안 강도를 제공한다.

관련 이론 정리

공개키 알고리즘은 기반 난제로 계열이 갈린다. 소인수분해: RSA. 이산대수(유한체): Diffie-Hellman, ElGamal, DSA. 타원곡선 이산대수: ECDH, ECDSA, ECIES. ECC의 강점은 키 길이 대비 안전성으로, ECC 256비트가 RSA 3072비트에 대응해 연산·저장·전송 부담이 훨씬 적다. 이 때문에 모바일·IoT·스마트카드처럼 자원이 제한된 환경과 TLS의 기본 키 교환(ECDHE)에 널리 쓰인다. 다만 곡선 파라미터 선택이 안전성에 직결되어 검증된 표준 곡선(P-256, Curve25519 등)을 사용해야 하며, 서명 시 난수(k) 재사용 금지(재사용하면 개인키가 노출된다)가 중요하다. 양자컴퓨터의 쇼어 알고리즘은 RSA와 ECC를 모두 무력화할 수 있어 PQC 전환이 논의된다. 감리에서는 알고리즘·키 길이·곡선 표준 적용을 점검한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

WPA-EAP(WPA-Enterprise)는 802.1X와 RADIUS 인증 서버를 이용해 사용자별로 인증하고, 인증 성공 시 세션마다 서로 다른 암호화 키를 동적으로 생성·분배한다. 이것이 사전 공유 키를 모두가 함께 쓰는 WPA-PSK(Personal)와의 결정적 차이이며, 한 사용자의 키가 노출돼도 다른 세션에 영향을 주지 않는다.

④는 "세션마다 지정된 암호화 키를 재사용한다"고 서술했으므로 동적 키 생성이라는 핵심 특성과 어긋나 틀렸다. 따라서 정답은 ④번이다.

①(WPA2는 AES 기반 CCMP 사용), ②(WPA2-PSK 환경에서 단말이 공유 키 기반으로 접속), ③(EAP는 아이디·패스워드 등 사용자 자격증명 기반 인증)은 모두 옳은 설명으로 제시되었다.

오답 분석

보기①

옳음. WPA2는 AES 기반 CCMP로 암호화하며 WPA의 TKIP보다 안전하다.

보기②

옳음(문항 기준). WPA2-PSK는 사전 공유 키 기반으로 접속 단말에 키가 적용된다.

보기③

옳음. WPA-EAP는 사용자 자격증명 기반 인증 방식을 사용한다.

보기④

정답(부적절). WPA-EAP는 세션마다 키를 새로 생성하며 재사용하지 않는다.

관련 이론 정리

무선랜 보안은 WEP → WPA → WPA2 → WPA3로 발전했다. WEP은 RC4와 짧은 IV를 써서 쉽게 해독되어 폐기됐고, WPA는 TKIP로 임시 보완했으며, WPA2는 AES 기반 CCMP를 채택해 사실상 표준이 됐다. WPA3는 사전 공유 키 방식의 오프라인 사전공격을 막는 SAE(Dragonfly) 핸드셰이크와 개방형 네트워크의 개별 암호화(OWE)를 도입했다. 운용 모드는 가정·소규모의 Personal(PSK)과 기업용 Enterprise(EAP/802.1X)로 나뉘며, 후자는 RADIUS 서버로 사용자별 인증과 세션별 동적 키를 제공해 계정 회수·감사가 가능하다. EAP 방식으로 EAP-TLS(인증서), PEAP·EAP-TTLS(터널 내 ID/PW)가 있다. 감리에서는 업무망 무선의 Enterprise 인증 적용과 게스트망 분리를 점검한다.

무선 보안 방식:

방식 암호 키 관리
WEP RC4 정적(취약)
WPA TKIP 임시 보완
WPA2 CCMP(AES) PSK 또는 EAP
WPA-EAP — 세션별 동적 키
실무 적용 사례

무선 네트워크 감리 시 업무망에 WPA2/WPA3-Enterprise(802.1X)가 적용됐는지, PSK 공유로 인한 계정 추적 불가 문제가 없는지 점검한다.

관련 이론 정리

해시 함수는 구조에 따라 특성이 다르다. 머클-담고르 구조(MD5, SHA-1, SHA-2)는 메시지를 블록으로 나눠 압축 함수를 반복 적용하며, 마지막에 길이 정보를 덧붙이기 때문에 길이 확장 공격(length extension attack)에 취약하다(그래서 HMAC 같은 구성이 필요하다). 스펀지 구조(SHA-3/Keccak)는 내부 상태에 입력을 흡수한 뒤 출력을 짜내는 방식으로 길이 확장 공격에 강하고, 출력 길이를 자유롭게 조절할 수 있어(SHAKE) 난수 생성·스트림 암호로도 쓰인다. 중요한 점은 SHA-3가 SHA-2의 취약점 때문에 나온 것이 아니라, SHA-1·SHA-2가 같은 구조를 공유하는 위험에 대비한 구조적 다양성 확보 목적이라는 것이다. 따라서 SHA-2는 지금도 안전하며 병행 사용된다. MD5·SHA-1은 충돌이 발견되어 사용 금지다. 감리에서는 취약 해시 사용 여부를 점검한다.

관련 이론 정리

ICMP는 오류 보고·진단용 프로토콜이지만 여러 공격에 악용된다. Ping of Death는 65,535바이트를 초과하는 ICMP 패킷을 단편화해 보내 재조합 시 버퍼 오버플로를 유발했고(현재는 대부분 패치됨), Smurf는 출발지를 피해자로 위조한 에코 요청을 브로드캐스트해 증폭된 응답을 피해자에게 몰리게 하며, ICMP Flooding은 대량 에코 요청으로 대역·자원을 소모한다. ICMP Redirect는 경로를 조작한다. 이 때문에 경계에서 ICMP를 제한하는 것이 일반적이나, ICMP를 전면 차단하면 경로 MTU 탐색(PMTUD)이 깨져 통신 장애가 생길 수 있어 Fragmentation Needed(type 3 code 4) 등 필수 메시지는 허용해야 한다. 또 iptables는 3·4계층 필터이므로 ARP 같은 2계층 공격은 스위치 기능(DAI)으로 대응해야 한다. 감리에서는 ICMP 정책의 균형을 점검한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

Reflective(반사형) XSS는 악성 스크립트를 URL 파라미터에 실어 보내면 서버가 이를 응답 페이지에 그대로 반사(reflect)해 피해자 브라우저에서 실행되는 유형이다. 서버에 저장되지 않으므로 공격자가 피해자에게 링크를 전달해 클릭을 유도해야 하며, 성공 시 쿠키 탈취·피싱 사이트 이동이 일어난다. 문제 설명이 정확히 이 흐름이므로 정답은 ②번이다.

①Stored XSS는 게시글·댓글 등으로 서버에 저장되어 해당 페이지를 여는 모든 사용자에게 실행되므로 링크 클릭 유도가 필요 없다. ③SQL Injection은 DB 질의문 조작, ④Command Injection은 OS 명령 실행으로 스크립트 실행과 목적이 다르다.

관련 이론 정리

XSS는 저장 위치·실행 경로로 세 유형이 구분된다. Stored(저장형)는 서버 DB에 저장되어 지속적·광범위한 피해를 주고, Reflected(반사형)는 요청이 즉시 응답에 반사되어 피싱 메일·메신저로 링크를 유포하는 방식으로 공격되며, DOM 기반은 서버를 거치지 않고 클라이언트 스크립트가 URL 조각 등을 DOM에 삽입하며 발생한다. 근본 대응은 출력 시 컨텍스트에 맞는 인코딩(HTML·속성·JS·URL)이며, 입력 검증(화이트리스트), CSP(Content Security Policy)로 인라인 스크립트 실행 제한, 쿠키의 HttpOnly(스크립트의 쿠키 접근 차단)·Secure·SameSite 속성을 병행한다. XSS는 세션 탈취로 이어져 세션 하이재킹의 주요 경로가 되므로 인증 이후 중요 작업에 재인증을 요구하는 것도 완화책이다. 감리에서는 출력 인코딩과 CSP 적용을 점검한다.

실무 적용 사례

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

브라우저가 HTTP 프록시를 거쳐 HTTPS 사이트에 접속할 때의 표준 동작은 다음과 같다.

  • 가(옳음): 브라우저는 CONNECT 메서드로 프록시에 "목적지 호스트:443으로 터널을 열어 달라"고 요청한다.
  • 라(옳음): 터널이 열리면 브라우저와 목적지 웹서버가 직접 종단 간 SSL/TLS 핸드셰이크를 수행하고, 프록시는 암호화된 바이트를 그대로 TCP 수준에서 중계할 뿐 내용을 보지 못한다.

따라서 옳은 것은 가·라이고 정답은 ②번이다.

나는 틀림: GET/POST 등 일반 메서드는 평문 HTTP 프록시 요청에 쓰이며 HTTPS 터널링에는 CONNECT를 쓴다. 다도 틀림: 구간별로 TLS를 각각 맺는 것은 SSL 인터셉션(중간자형 검사) 구성일 때이며, 표준 CONNECT 터널링에서는 종단 간 단일 TLS 세션이 성립한다.

오답 분석

보기①

'나'(일반 메서드 사용)는 HTTPS 터널링 동작이 아니므로 부적절하다.

보기②

정답. 가(CONNECT 사용)·라(종단 간 TLS, 프록시는 TCP 중계)가 옳다.

보기③

'나'·'다' 모두 표준 HTTPS 프록시 동작과 다르다.

보기④

'다'(구간별 별도 TLS)는 SSL 인터셉션 구성에 해당하며 표준 동작이 아니다.

관련 이론 정리

HTTP CONNECT는 프록시에게 목적지와의 TCP 터널 수립을 요청하는 메서드로, 주로 HTTPS 트래픽을 프록시 경유로 전달할 때 쓰인다. 터널이 성립하면 프록시는 암호문을 그대로 릴레이하므로 종단 간 기밀성이 유지되고, 프록시는 접속 대상 호스트명·포트와 트래픽 양 정도만 알 수 있다. 반면 기업 환경에서 내부 사용자의 HTTPS 트래픽을 검사해야 하는 경우 SSL/TLS 인터셉션(SSL 가시성 장비)을 도입하는데, 이때는 프록시가 자체 CA 인증서로 위조 인증서를 발급해 브라우저↔프록시, 프록시↔서버의 두 개 TLS 세션을 만들고 중간에서 복호화·검사한다. 이 방식은 보안 검사에 유용하지만 사실상 합법적 중간자 공격이라 사생활·법적 이슈, 인증서 관리 부담, 인증서 고정(pinning) 앱과의 충돌이 발생한다. 감리에서는 인터셉션 도입 시 고지·동의와 예외 정책을 점검한다.

프록시 경유 HTTPS 동작:

구성 TLS 세션 프록시의 가시성
표준(CONNECT 터널) 종단 간 1개 호스트명·트래픽량만
SSL 인터셉션 구간별 2개 평문 내용까지
실무 적용 사례

보안 프록시 감리 시 SSL 인터셉션 도입 여부와 그에 따른 개인정보·사생활 고지, 금융·의료 사이트 예외 처리 정책을 점검한다.

감리사 시험 출제 포인트

HTTPS 프록시 = CONNECT + 종단 간 TLS가 표준이며, 구간별 TLS 2개는 인터셉션 구성이라는 구분이 핵심이다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

솔트(salt)는 패스워드에 덧붙이는 무작위 값으로, 같은 패스워드라도 매번 다른 키가 생성되게 하여 사전 공격(dictionary attack)과 미리 계산된 레인보우 테이블 공격을 무력화하는 것이 목적이다.

③은 솔트가 중간자 공격(MITM)을 방지한다고 서술했는데, MITM은 통신 경로에서 가로채는 공격이라 상호 인증·인증서 검증·채널 암호화(TLS)로 막아야 하며 솔트와는 무관하다. 따라서 가장 적절하지 않은 것은 ③번이다.

①(PBE는 Java 암호 패키지에 포함되고 PGP의 키 관리에도 사용), ②(KEK로 CEK를 보호하는 2단계 키 구조), ④(패스워드 + 난수 솔트를 입력으로 KEK 생성)는 모두 옳다.

관련 이론 정리

PBE는 사람이 기억하는 패스워드로부터 암호키를 만들어 데이터를 보호하는 방식이다. 패스워드는 엔트로피가 낮아 그대로 키로 쓰면 위험하므로, 키 유도 함수(KDF)로 안전한 키를 만든다. 핵심 요소는 두 가지다. 솔트는 각 사용자·항목마다 다른 무작위 값을 더해 동일 패스워드가 동일 키가 되지 않게 하여 사전 계산 공격을 막고, 반복 횟수(iteration)를 통한 키 스트레칭은 키 유도에 의도적으로 시간을 들여 무차별 대입 비용을 크게 높인다. 대표 KDF로 PBKDF2(반복), bcrypt, scrypt(메모리 사용량 증가로 GPU 공격 방어), Argon2(현재 권장)가 있다. 구조적으로는 패스워드로 유도한 KEK가 실제 데이터 암호키(CEK)를 감싸는 방식이라, 패스워드 변경 시 CEK를 재암호화만 하면 되어 효율적이다. 감리에서는 KDF 사용과 반복 횟수, 솔트 길이·유일성을 점검한다.

정답 해설

「개인정보의 안전성 확보조치 기준」은 정보 유형에 따라 암호화 방식을 달리 규정한다. 판단 기준은 나중에 원본을 복원(복호화)해야 하는가이다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

코드의 문제를 하나씩 보면 다음과 같다.

  • scanf("%s", _username): 길이 제한 없이 입력을 받아 malloc으로 할당한 32바이트 힙 버퍼를 넘길 수 있다 → 힙 오버플로우(②)
  • strcpy(username, _username): 길이 검사 없이 복사해 스택의 32바이트 배열을 넘길 수 있다 → 스택 오버플로우(①)
  • 위 두 오버플로우로 인접 메모리를 침범하면 메모리 접근 위반(③)이 발생할 수 있다.

반면 ④ 정수 오버플로우는 정수 연산 결과가 자료형의 표현 범위를 초과할 때 발생하는데, 이 코드에는 크기 계산이나 산술 연산 자체가 없으므로 해당하지 않는다. 따라서 가장 적절하지 않은 것은 ④번이다.

안전한 수정은 scanf("%31s", ...)처럼 길이를 제한하고 strncpy·snprintf 등 경계를 지정하는 함수를 사용하는 것이다.

정답(부적절). 코드에 정수 산술 연산이 없어 정수 오버플로우와 무관하다.

관련 이론 정리

메모리 관련 취약점은 발생 영역과 원인으로 구분된다. 스택 오버플로우는 지역 변수 배열을 넘겨 복귀 주소(RET)를 덮어써 임의 코드 실행으로 이어지고, 힙 오버플로우는 동적 할당 영역을 넘겨 힙 메타데이터나 함수 포인터를 조작한다. 정수 오버플로우는 크기 계산 시 값이 넘쳐 의도보다 작은 버퍼가 할당되고 이어서 버퍼 오버플로우로 연결되는 경우가 많다. 그 밖에 Use-After-Free(해제 후 사용), 이중 해제, 포맷 스트링이 있다. 대응은 코드 수준의 안전 함수 사용(strncpy·snprintf·fgets)과 입력 길이 검증, 컴파일러·OS의 스택 카나리·ASLR·DEP/NX·FORTIFY_SOURCE, 그리고 정적분석·퍼징이다. 근본적으로는 메모리 안전 언어(Rust 등) 채택도 논의된다. 감리에서는 위험 함수 사용 여부와 보호 옵션 활성화를 점검한다.

메모리 취약점 구분:

취약점 원인

힙 오버플로우 동적 할당 버퍼 초과

실무 적용 사례

C/C++ 코드 감리 시 gets·strcpy·sprintf·scanf("%s") 등 위험 함수 사용을 정적분석으로 식별하고 안전 함수 대체와 컴파일 보호 옵션 적용을 권고한다.

정답 해설

「개인정보 보호법」 제3조 제5항은 "개인정보처리자는 개인정보 처리방침 등 개인정보의 처리에 관한 사항을 공개하여야 하며, 열람청구권 등 정보주체의 권리를 보장하여야 한다"고 규정한다. 즉 처리방침은 반드시 공개해야 하는 투명성 원칙의 핵심이다.

②는 "비공개하여야 하며"라고 서술해 법 규정과 정반대이므로 정답은 ②번이다.

관련 이론 정리

개인정보 처리방침은 처리 목적·항목·보유기간, 제3자 제공·위탁 현황, 정보주체의 권리와 행사 방법, 파기 절차, 안전성 확보조치, 개인정보 보호책임자 연락처 등을 담아 홈페이지 등에 상시 공개해야 하는 문서다. 이는 정보주체가 자신의 정보가 어떻게 처리되는지 알고 권리를 행사할 수 있게 하는 투명성 원칙의 구현이며, 변경 시에는 변경 사항을 공지해야 한다. 함께 보장해야 할 정보주체의 권리는 열람, 정정·삭제, 처리정지, 동의 철회, 자동화 결정에 대한 거부·설명 요구 등이다. 또한 제3조 제7항은 익명처리 우선 원칙을 명시해, 목적 달성이 가능하면 익명으로, 익명으로 곤란하면 가명으로 처리하도록 단계적 우선순위를 정했다. 감리에서는 처리방침의 공개·현행화, 법정 기재사항 충족, 권리 행사 창구 운영을 점검한다.
