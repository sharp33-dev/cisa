영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

OCP(Open Closed Principle): "소프트웨어 엔티티는 확장에는 열려 있어야 하고, 수정에는 닫혀 있어야 한다."

문제 핵심: "테스트 대상 클래스를 변경하지 않고도" 환경을 변경

이것이 OCP의 본질:
- 클래스 자체 코드를 수정(닫힘)하지 않고
- 외부 환경(의존성)을 교체(확장)하여 동작 변경

단위 테스트에서 Mock/Stub 객체를 주입하여 대상 클래스를 수정 없이 테스트 환경을 통제하는 것이 OCP를 따르는 설계의 대표 예이다.

오답 분석

보기①(SRP)

단일 책임 원칙: 하나의 클래스는 하나의 책임만. 테스트 환경 교체와 직접적 관련 없음.

보기③(DIP)

의존성 역전 원칙: 추상화에 의존. OCP를 구현하는 수단이 DIP이지만, "클래스를 변경하지 않고 환경 변경"의 직접적 설명은 OCP.

보기④(LSP)

리스코프 치환 원칙: 자식이 부모를 대체 가능. 직접 관련 없음.

관련 이론 정리

SOLID 원칙:

원칙 핵심 문장
SRP 하나의 클래스 = 하나의 변경 이유
OCP 확장 O, 수정 X
LSP 자식 = 부모 대체 가능
ISP 작고 구체적인 인터페이스
DIP 추상화에 의존

정답 해설

커버리지 포함 관계 (약 → 강):

문장(SC) ⊂ 결정(DC) ⊂ 조건/결정(C/DC) ⊂ MC/DC ⊂ 다중조건(MCC)

단, 조건 커버리지(CC)는 결정 커버리지의 상위 집합이 아니다.

각 항목 검증:

  • 가. 결정(DC) → 문장(SC): DC가 모든 분기를 커버하면 SC도 자동 만족. ✓ 참
  • 나. 결정/조건(C/DC) → 결정(DC): C/DC는 DC의 상위 집합이므로 DC 자동 만족. ✓ 참
  • 다. 조건(CC) → 결정(DC): ✗ 거짓
  • 예: if (A || B) 에서 조건 커버리지는 A=T, B=F / A=F, B=T 두 케이스로 달성 가능
  • 하지만 전체 결정의 T/F 분기를 모두 커버하지 않을 수 있음
  • 조건 커버리지가 결정 커버리지를 보장하지 않음
  • 라. 결정/조건(C/DC) → 조건(CC): C/DC는 조건의 T/F도 포함하므로 CC 자동 만족. ✓ 참

올바른 것: 가, 나, 라 → ③번

관련 이론 정리

조건 커버리지가 결정 커버리지를 보장 못하는 예:

if (a || b) { ... }  // True 분기
else { ... }         // False 분기

// 조건 커버리지 달성:
// TC1: a=T, b=F → 전체 True
// TC2: a=F, b=T → 전체 True
// → 결정(False 분기)이 한 번도 실행 안 됨!
// 결정 커버리지 미달!

정답 해설

시퀀스 다이어그램 분석:
- A가 요청 → B(Broker)가 pack(직렬화) + forward
- C(Naming Service)에서 서버 위치 탐색

→ Broker 패턴: 클라이언트와 서비스를 중개하는 Broker가 서비스 위치를 관리

Broker 패턴 특징:
- modifiability: 서비스 위치/구현 변경 시 클라이언트 변경 불필요
- availability: Broker가 서버 다중화, 장애 시 다른 서버 연결 가능
- performance: 부하 분산, 요청 라우팅 최적화 가능
- 이기종 언어/플랫폼 서비스 연동 가능 (③이 틀린 이유)
- 분산 복잡도 감소 (브로커가 복잡도 숨김, ① 틀림)

정답 해설

변화 분석:

(가) 문제: UI → Car → Person 체인 접근
- UI가 car.getOwner().getName() 처럼 Person에 직접 접근
- UI가 Car의 내부 구조(Person 존재)를 알고 있음 → 디미터 법칙 위반

(나) 개선: Car에 getOwnerName() 메서드 추가
- UI가 car.getOwnerName() 직접 호출
- Car가 Person에 대한 접근을 숨겨줌(delegate hide)
- UI는 Person 존재를 모름 → 결합도 감소

이것이 Hide Delegate:

"클라이언트가 서버의 위임 객체에 직접 접근하는 경우, 서버에 위임 메서드를 추가하여 위임 관계를 숨긴다"

오답 분석

보기②(Remove Middle Man)

Hide Delegate의 반대: 단순 위임 메서드를 제거하고 클라이언트가 직접 접근하게 함. 이 코드와 반대 방향.

보기③(Rename Method)

메서드 이름만 변경.

보기④(Encapsulate Downcast)

형변환(downcast)을 메서드로 감추는 기법.

관련 이론 정리

Hide Delegate vs Remove Middle Man:

기법 방향 목적
Hide Delegate 위임 숨기기 결합도 감소, 디미터 법칙 준수
Remove Middle Man 중간자 제거 불필요한 위임 메서드 제거
