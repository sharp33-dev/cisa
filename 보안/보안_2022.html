영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

전자서명은 송신자의 개인키로 메시지 다이제스트를 서명하고 수신자가 송신자의 공개키로 검증하는 방식으로, ① 송신자 인증(개인키 소유자만 서명 가능), ② 무결성(내용이 바뀌면 검증 실패), ③ 부인방지(제3자도 검증 가능)를 제공한다.

그러나 서명은 원문을 감추지 않으므로 기밀성은 제공하지 않는다. 기밀성이 필요하면 별도로 암호화를 병행해야 한다. 따라서 전자서명의 역할로 가장 적절하지 않은 것은 ③번이다.

오답 분석

보기①

전자서명은 개인키 소유자만 생성할 수 있어 송신자 인증을 제공한다.

보기②

서명 검증 과정에서 메시지 변조 여부를 확인할 수 있어 무결성을 제공한다.

보기③

정답(부적절). 기밀성은 암호화의 역할이며 전자서명은 제공하지 않는다.

보기④

누구나 공개키로 검증할 수 있어 송신 사실을 부인할 수 없다(부인방지).

관련 이론 정리

전자서명은 공개키 암호의 키 사용 방향을 반대로 적용한 것이다. 기밀성 제공 시에는 수신자의 공개키로 암호화하고 수신자가 개인키로 복호화하는 반면, 전자서명은 송신자의 개인키로 서명하고 누구나 송신자의 공개키로 검증한다. 이 방향 구분이 시험의 최다 함정이다. 실제 구현에서는 긴 원문 전체를 서명하면 느리므로 해시로 다이제스트를 만든 뒤 그 값을 서명한다. 서명 알고리즘으로 RSA-PSS, DSA, ECDSA가 쓰이며, 공개키가 진짜 그 사람의 것임을 보증하기 위해 인증기관(CA)이 발급한 인증서와 PKI 체계가 필요하다. 기밀성과 부인방지를 함께 원하면 서명 후 암호화(sign-then-encrypt) 순서로 결합한다. 감리에서는 전자거래의 부인방지 요건 충족 여부와 인증서 관리를 점검한다.

인증 MAC·전자서명
부인방지 전자서명만
실무 적용 사례

전자계약·전자결재 감리 시 부인방지가 요구되는 거래에 전자서명이 적용됐는지, 서명값·인증서·타임스탬프가 함께 보관되는지 점검한다.

감리사 시험 출제 포인트

전자서명 = 인증·무결성·부인방지(기밀성 ✕)가 핵심이다. MAC은 부인방지까지는 못 한다는 점과 세트로 암기한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

AES의 각 라운드는 네 연산으로 구성되며 처리 단위가 다르다.

  • SubBytes: 1바이트씩 S-box를 참조해 다른 1바이트로 치환(비선형성 제공)
  • ShiftRows: 상태 행렬의 각 행을 좌측으로 회전
  • MixColumns: 한 열(4바이트)을 유한체 연산으로 새로운 4바이트로 변환(확산 제공)
  • AddRoundKey: 라운드 키와 XOR

③은 SubBytes가 "4 바이트의 값을 새로운 4 바이트 값으로 변환"한다고 서술했는데, 이는 MixColumns의 설명이므로 틀렸다. 따라서 정답은 ③번이다. ①(블록 128비트 고정, 키 3종), ②(SPN 구조), ④(4개 서브 함수)는 모두 옳다.

오답 분석

보기①

옳음. AES의 블록 크기는 128비트 고정이고 키는 128·192·256비트를 선택한다.

보기②

옳음. DES는 Feistel, AES는 SPN 구조를 사용한다.

보기③

정답(부적절). 4바이트 단위 변환은 MixColumns이며 SubBytes는 1바이트 치환이다.

보기④

옳음. 각 라운드는 SubBytes·ShiftRows·MixColumns·AddRoundKey로 구성된다.

관련 이론 정리

AES(Rijndael)는 DES를 대체한 대칭키 블록암호 표준으로, 키 길이에 따라 라운드 수가 10(128비트)·12(192비트)·14(256비트)로 달라진다. 구조는 SPN(치환-순열 네트워크)으로, 치환(S-box)이 혼돈(confusion)을, 순열·행렬 변환(ShiftRows·MixColumns)이 확산(diffusion)을 담당한다. 마지막 라운드에서는 MixColumns를 생략한다. Feistel 구조(DES·SEED)는 블록을 반으로 나눠 한쪽만 처리하고 교환하는 방식이라 암·복호화 구조가 동일한 장점이 있는 반면, SPN은 전체 블록을 한 번에 처리해 병렬성이 좋다. 국산 알고리즘으로는 SEED(Feistel), ARIA(SPN, 경량), LEA(고속 경량)가 있다. AES는 현재까지 실용적 공격이 없어 안전하며, 운용 모드(GCM 등)와 결합해 사용한다. 감리에서는 알고리즘·키 길이·운용 모드의 적정성을 점검한다.

AES 라운드 연산:

연산 처리 단위 역할
SubBytes 1바이트 S-box 치환 혼돈
ShiftRows 행 단위 회전 확산
MixColumns 열(4바이트) 변환 확산
AddRoundKey 라운드 키 XOR 키 결합
실무 적용 사례

암호 적용 감리 시 DES·3DES 등 취약 알고리즘 잔존 여부를 확인하고 AES-128 이상 또는 국산 검증 알고리즘(ARIA·SEED·LEA)으로 전환을 권고한다.

감리사 시험 출제 포인트

SubBytes(1바이트 치환)와 MixColumns(4바이트 열 변환)의 처리 단위를 바꿔 놓는 것이 전형적 함정이다. AES=SPN, DES=Feistel도 필수 암기다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

②의 RBAC(역할 기반 접근통제) 설명이 정확하다. RBAC는 사용자 개개인이 아니라 조직의 역할(직무)에 권한을 부여하고 사용자를 역할에 배정해, 인사이동 시 역할만 바꾸면 되어 대규모 조직에서 관리가 효율적이다. 따라서 정답은 ②번이다.

①과 ④는 설명이 서로 뒤바뀌었다: 보안등급(레이블)으로 결정하는 것은 MAC(강제적 접근통제)이고, 사용자의 신원과 소유자 재량에 근거하는 것은 DAC(임의적 접근통제)다. ③도 틀림: ACL은 객체마다 접근 가능한 주체 목록을 저장하므로, 주체 수가 많아지면 목록이 길어져 관리가 비효율적이다.

보기①

보기②

정답. RBAC는 역할과 객체의 관계로 권한을 관리한다.

보기③

부적절. ACL은 주체 수가 많아지면 목록 관리가 비효율적이다.

보기④

관련 이론 정리

접근통제 정책은 세 가지가 기본이다. DAC(임의적)는 객체 소유자가 재량으로 권한을 부여하며 유연하지만 트로이 목마에 의한 권한 유출에 취약하다(대부분의 상용 OS 파일 권한). MAC(강제적)는 시스템이 주체의 인가 등급과 객체의 보안 레이블을 비교해 강제 적용하며 군·정부의 기밀 시스템에 쓰인다(BLP 모델 기반). RBAC(역할 기반)는 역할 단위로 권한을 묶어 최소권한·직무분리 적용이 쉬워 기업 환경 표준이 됐다. 구현 자료구조로는 객체 기준의 ACL(객체마다 주체 목록)과 주체 기준의 자격목록(Capability List)(주체마다 접근 가능 객체 목록)이 있으며, 둘을 행렬로 표현한 것이 접근통제 행렬이다. 최근에는 속성 기반의 ABAC로 확장된다. 감리에서는 권한 부여의 최소성과 주기적 검토를 점검한다.

MAC은 구현 방식에 따라 두 계열로 나뉜다. 해시 기반(HMAC)은 해시 함수에 키를 두 번 결합(ipad·opad)해 길이 확장 공격을 막으며 HMAC-SHA256 등으로 쓰인다. 블록암호 기반(CBC-MAC·CMAC)은 블록암호를 이용해 태그를 만들며, 원조 CBC-MAC은 가변 길이 메시지에서 취약점이 있어 이를 보완한 CMAC이 표준화됐다. MAC은 무결성 + 송수신자 간 인증을 제공하지만 키 공유 구조상 부인방지는 불가하며, 기밀성이 필요하면 암호화를 병행해야 한다. 이때 결합 순서에 따른 취약점을 피하기 위해 현대 시스템은 AEAD 모드(GCM·CCM)로 암호화와 인증을 한 번에 처리한다. 감리에서는 무결성 검증 적용 여부와 알고리즘 강도를 점검한다.

정답 해설

TCB(Transmission Control Block) 큐에서 연결 정보가 제거(Dequeue)되는 상황은 ① 3-way handshake가 완료되어 연결이 확립될 때(백로그 큐에서 accept 큐로 이동), ② 대기 시간이 타임아웃을 초과할 때, ④ 해당 연결에 대해 RST 패킷이 도착해 연결이 리셋될 때다.

반면 ③처럼 공격자가 랜덤하게 생성한(존재하지 않는) 출발지 IP로 SYN을 보내면, 서버가 SYN+ACK를 보내도 응답할 호스트가 없어 ACK가 오지 않는다. 그 결과 연결 정보는 half-open 상태로 큐에 계속 남아 타임아웃까지 자리를 차지하며, 이것이 바로 SYN Flooding 공격이 성립하는 원리다. 즉 ③은 제거가 아니라 잔류시키는 상황이므로 정답은 ③번이다.

보기④

관련 이론 정리

TCP 연결은 SYN → SYN+ACK → ACK의 3-way handshake로 성립하며, 서버는 SYN을 받은 시점에 연결 정보를 백로그(TCB) 큐에 저장하고 ACK를 기다리는 half-open 상태로 둔다. SYN Flooding은 위조된 출발지 IP로 SYN만 대량 전송해 이 큐를 가득 채워, 정상 사용자의 연결 요청이 거부되도록 만드는 자원 소모형 DoS다. 대응책으로는 ① SYN 쿠키(큐에 저장하지 않고 시퀀스 번호에 정보를 인코딩해 ACK가 와야 자원 할당), ② 백로그 큐 크기 증가, ③ 타임아웃 단축, ④ 방화벽·Anti-DDoS 장비의 SYN 프록시, ⑤ 출발지 위조 차단을 위한 Ingress 필터링이 있다. 유사 공격으로 연결을 오래 유지시키는 Slowloris가 있다. 감리에서는 DDoS 대응 설정과 임계치 모니터링을 점검한다.

정답 해설

④의 Switch Jamming(MAC 플러딩) 설명이 정확하다. 공격자가 위조된 MAC 주소를 대량으로 흘려보내 스위치의 MAC 주소 테이블(CAM 테이블)을 가득 채우면, 스위치는 목적지를 학습할 수 없어 허브처럼 모든 포트로 프레임을 플러딩하게 되고 이를 이용해 스니핑이 가능해진다. 따라서 정답은 ④번이다.

①②는 설명이 서로 뒤바뀌었다: 출발지 IP를 위조해 연속 전송하는 것은 SYN Flooding 계열이고, 대상 IP로 위장해 브로드캐스트로 전송하는 증폭 공격은 Smurf다. ③도 틀림: 신뢰관계 호스트를 마비시키고 IP를 속이는 것은 IP 스푸핑(세션 하이재킹) 설명이며, ARP Redirect는 위조 ARP로 자신을 라우터로 위장하는 공격이다.

보기②

부적절. 대상 IP로 위장해 브로드캐스트로 보내는 것은 Smurf의 설명이다.

부적절. 신뢰관계를 악용한 IP 속임은 IP 스푸핑이며, ARP Redirect는 라우터 위장이다.

정답. 위조 MAC으로 스위치 테이블을 채워 스니핑하는 것이 Switch Jamming이다.

관련 이론 정리

네트워크 공격은 목적에 따라 도청(스니핑), 위장(스푸핑), 서비스 방해(DoS)로 나뉜다. 스위치 환경에서 스니핑을 하려면 트래픽을 자신에게 끌어와야 하므로 Switch Jamming(MAC 플러딩)으로 스위치를 허브처럼 만들거나, ARP 스푸핑/Redirect로 자신을 게이트웨이로 위장한다. DoS 계열에서는 Smurf(출발지를 피해자로 위조한 ICMP를 브로드캐스트해 증폭된 응답이 피해자에게 쏟아지게 함)와 SYN Flooding(half-open 연결로 백로그 고갈)이 대표적이다. 대응은 스위치의 포트 보안(Port Security, MAC 수 제한), DAI, 브로드캐스트 지시 패킷 차단(no ip directed-broadcast), SYN 쿠키다. 감리에서는 스위치 보안 설정과 내부망 스니핑 대응을 점검한다.

SYN Flooding half-open 연결로 백로그 고갈
ARP Redirect 위조 ARP로 라우터 위장
실무 적용 사례

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

요구사항은 "특정 IP로부터 들어오는 패킷을 차단"하는 것이다.

  • 체인: 시스템으로 들어오는 패킷은 INPUT 체인에서 처리(모든 보기 동일)
  • 타깃(-j): 차단은 DROP(응답 없이 폐기)이며, REJECT는 거부 응답을 보내고 차단한다.

따라서 -j DROP을 사용한 ①이 정답이다.

②는 LOG로 기록만 하고 차단하지 않으며(LOG는 non-terminating 타깃), ③은 ACCEPT로 오히려 허용하고, ④는 RETURN으로 호출한 체인으로 복귀시킬 뿐 차단하지 않는다.

참고로 엄밀한 문법은 출발지 지정 옵션 -s를 붙여 iptables -A INPUT -s 192.168.20.22 -j DROP으로 작성해야 하나, 보기 중 요구를 충족하는 것은 ①뿐이다.

오답 분석

보기①

정답. INPUT 체인에 DROP 규칙을 추가해 해당 IP의 인입 패킷을 폐기한다.

보기②

LOG는 로그만 남기고 패킷을 통과시키므로 차단 효과가 없다.

보기③

ACCEPT는 허용 타깃이므로 요구와 정반대다.

보기④

RETURN은 이전 체인으로 복귀시키는 타깃으로 차단이 아니다.

관련 이론 정리

iptables는 리눅스 넷필터 기반 방화벽으로 테이블 → 체인 → 규칙 구조를 갖는다. 기본 테이블은 filter(허용·차단), nat(주소 변환), mangle(패킷 변조)이며, filter 테이블의 체인은 INPUT(로컬 수신), OUTPUT(로컬 송신), FORWARD(경유 전달)다. 주요 옵션은 -A(체인 끝에 추가), -I(맨 앞에 삽입), -D(삭제), -L(목록), -P(기본 정책), -s/-d(출발지/목적지), -p(프로토콜), --dport(목적지 포트)다. 타깃은 ACCEPT·DROP·REJECT·LOG·RETURN이 있으며 DROP은 조용히 폐기, REJECT는 ICMP 거부 응답 후 폐기로 스캔 대응 시 선택이 달라진다. 규칙은 위에서부터 순차 적용되므로 순서가 중요하며, 기본 정책을 DROP으로 두고 필요한 것만 허용하는 화이트리스트 구성이 권장된다. 감리에서는 방화벽 규칙의 최소성과 기본 정책을 점검한다.

iptables 주요 타깃:

ACCEPT 허용
DROP 응답 없이 폐기(차단)
REJECT 거부 응답 후 차단
LOG 기록만(통과)
실무 적용 사례

서버 방화벽 감리 시 기본 정책(-P)이 DROP인지, 규칙 순서가 의도대로인지, 재부팅 시 규칙이 유지되도록 저장·적용되는지 점검한다.

감리사 시험 출제 포인트

차단 = DROP(또는 REJECT), LOG는 차단이 아님이 핵심이다. INPUT/OUTPUT/FORWARD 체인 구분도 함께 출제된다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

ARP Poisoning은 위조된 ARP 응답으로 피해자의 ARP 캐시를 오염시키는 공격이므로, 방어의 핵심은 캐시가 임의로 갱신되지 않게 하는 것이다. ①(ARP 패킷으로 MAC 매핑이 동적으로 갱신되지 않도록 통제), ③(정적 ARP 테이블 설정), ②(비인가 외부 단말의 로컬 네트워크 접근 차단으로 공격자 진입 자체를 방지)는 모두 유효한 대책이다.

④의 자기 컴퓨터 MAC 주소를 주기적으로 변경하는 것은 공격자가 위조 ARP를 보내는 행위 자체를 막지 못하며, 오히려 정상 통신과 자산 식별에 혼란을 준다. 따라서 방지 방법으로 가장 적절하지 않은 것은 ④번이다.

오답 분석

보기①

옳음. ARP 패킷에 의한 동적 갱신을 제한하면 캐시 오염을 막을 수 있다.

보기②

옳음. 비인가 단말의 내부망 접근을 차단하면 공격자가 위조 ARP를 보낼 수 없다.

보기③

옳음. 정적 ARP 테이블은 위조 응답을 무시하게 하는 대표적 대책이다.

보기④

정답(부적절). MAC 주소 변경은 ARP 위조를 막지 못하며 방어 효과가 없다.

관련 이론 정리

ARP는 인증 절차가 없어 요청하지 않은 응답(gratuitous ARP)도 그대로 캐시에 반영하는 구조적 취약점을 갖는다. 이를 악용한 ARP Poisoning은 같은 브로드캐스트 도메인 내에서 중간자 공격(MITM)을 성립시켜 도청·변조·세션 하이재킹으로 이어진다. 대응은 계층적으로 이뤄진다. 단말 수준에서는 게이트웨이 등 중요 대상에 정적 ARP 등록, 네트워크 장비 수준에서는 DHCP 스누핑으로 신뢰할 수 있는 IP-MAC 바인딩 테이블을 만들고 이를 근거로 위조 ARP를 차단하는 DAI(Dynamic ARP Inspection), 포트 보안으로 MAC 수를 제한한다. 접근 통제 수준에서는 NAC로 비인가 단말을 차단한다. 근본적으로는 종단 간 암호화(HTTPS·SSH·VPN)로 도청되더라도 내용을 보호한다. 감리에서는 이들 설정의 적용 여부를 점검한다.

정답 해설

드로퍼(Dropper)는 자신의 내부에 압축·암호화된 형태로 악성코드를 품고 있다가 실행 시 이를 풀어 시스템에 생성(투하)하는 유형이다. 즉 설치·유포 도구이지 스스로 복제·전파하는 것이 아니다.

③은 "네트워크를 통해 자기 복제를 하며 전파된다"고 서술했는데, 이는 웜(Worm)의 정의이므로 틀렸다. 따라서 정답은 ③번이다.

①(트로이 목마: 정상 프로그램으로 위장해 정보 유출·원격 제어), ②(스파이웨어: 사용자 정보 수집·전송), ④(PUP: 동의를 얻어 설치되나 성능 저하·결제 유도 등 원치 않는 동작)는 모두 옳다.

보기②

관련 이론 정리

악성코드는 전파 방식으로 구분하면 명확하다. 바이러스는 숙주 파일에 기생해 그 파일이 실행될 때 감염을 확산하고, 웜은 숙주 없이 스스로 복제해 네트워크로 전파하며(취약점·메일·공유 폴더 이용) 대역폭을 소모한다. 트로이 목마는 복제하지 않고 유용한 프로그램으로 위장해 사용자가 직접 설치하게 만든다. 유포 단계 도구로는 내부에 페이로드를 품은 드로퍼와 외부에서 내려받는 다운로더가 있다. 목적별로는 랜섬웨어(파일 암호화 후 금전 요구), 스파이웨어·키로거(정보 탈취), 애드웨어·PUP(광고·결제 유도), 루트킷(은폐), 봇넷(원격 조종)이 있다. 대응은 백신·EDR, 패치 관리, 실행 파일 통제, 백업(랜섬웨어 대비)이다. 감리에서는 악성코드 대응 체계와 백업 복구 시험을 점검한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

리눅스에서 root의 직접 로그인 가능 터미널은 /etc/securetty에 정의되며, 이 파일에서 원격 터미널(pts/) 항목을 제거하면 root의 원격 로그인이 차단된다. 그리고 이 파일을 실제로 참조하도록 하는 것이 /etc/pam.d/login의 pam_securetty.so* 모듈 설정이다. 즉 두 파일이 함께 동작해 root 원격 접속을 통제하므로 정답은 ④번이다.

①의 /etc/crontab은 예약 작업 설정, ②의 /etc/shadow는 암호화된 비밀번호 저장, ③의 /etc/passwd는 계정 기본 정보 파일로 원격 접속 허용 여부를 통제하는 파일이 아니다(다만 셸을 /sbin/nologin으로 바꿔 로그인을 막는 우회적 방법은 있다).

참고로 SSH를 통한 접속은 별도로 /etc/ssh/sshd_config의 PermitRootLogin no 설정으로 차단해야 한다.

오답 분석

보기①

/etc/crontab은 예약 작업 파일로 원격 로그인 통제와 무관하다.

보기②

/etc/shadow는 비밀번호 해시 저장 파일이며 접속 터미널을 제한하지 않는다.

보기③

/etc/passwd는 계정 정보 파일로 securetty와 짝을 이루는 설정 파일이 아니다.

보기④

정답. /etc/securetty(허용 터미널 목록)와 /etc/pam.d/login(PAM 정책)의 조합이 맞다.

관련 이론 정리

리눅스 계정 보안의 핵심 파일은 역할이 뚜렷하다. /etc/passwd는 계정명·UID·GID·홈·셸을 담고(모두 읽기 가능), /etc/shadow는 암호 해시와 만료 정책을 담아 root만 읽을 수 있게 분리되어 있다. /etc/securetty는 root가 직접 로그인할 수 있는 터미널을 나열하고, /etc/pam.d/*는 서비스별 PAM(Pluggable Authentication Module) 인증 정책(계정·인증·세션·비밀번호 스택)을 정의한다. root 접근 통제의 실무 원칙은 root 직접 로그인 금지 + 일반 계정 로그인 후 su/sudo로 권한 상승이며, sudo 사용 이력이 남아 책임 추적성(accountability)이 확보된다. SSH에서는 PermitRootLogin no, 키 기반 인증, 포트 변경, AllowUsers 제한을 함께 적용한다. 감리에서는 root 원격 접속 차단과 sudo 정책·로그를 점검한다.

주요 계정 관련 파일:

파일 역할
/etc/passwd 계정 기본 정보
/etc/shadow 암호 해시·만료 정책
/etc/securetty root 로그인 허용 터미널
/etc/pam.d/login PAM 인증 정책
/etc/ssh/sshd_config SSH 접속 정책(PermitRootLogin)
실무 적용 사례

서버 보안 감리 시 root 원격 접속 차단(securetty·sshd_config), sudo 기반 권한 상승 정책, 관리자 접속 로그 보관 여부를 점검한다.

관련 이론 정리

위험관리는 위험을 완전히 없애는 것이 아니라 조직이 감내할 수 있는 수준(DoA, Degree of Assurance)으로 낮추는 활동이다. 위험은 통상 위험 = 자산 × 위협 × 취약점으로 표현되며, 분석 방법은 정성적(델파이·시나리오·순위결정), 정량적(연간예상손실 ALE = SLE × ARO), 또는 혼합 방식을 쓴다. 대응 전략은 위험 감소(통제 도입), 위험 회피(활동 중단), 위험 전가(보험·아웃소싱), 위험 수용(감내)의 네 가지이며, 대응 후 남는 것이 잔여 위험이다. 이 절차는 일회성이 아니라 환경 변화에 맞춰 주기적으로 반복(PDCA)되어야 하며, PMBOK의 프로젝트 위험관리(식별-정성분석-정량분석-대응계획-감시통제)와도 구조가 같다. 감리에서는 위험 평가 근거와 대응 계획의 이행 여부를 점검한다.

3 위험 대응 계획(감소·회피·전가·수용)
4 모니터링 및 통제(잔여·신규 위험 추적)

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

EBP(Extended Base Pointer)는 현재 함수의 스택 프레임 기준 주소를 담는 레지스터로, 함수의 인자(argument)와 지역변수는 EBP를 기준으로 오프셋을 더하거나 빼서 참조된다(예: [EBP+8]은 첫 인자, [EBP-4]는 지역변수). 문제 설명과 정확히 일치하므로 정답은 ④번이다.

②ESP(Stack Pointer)는 스택의 최상단(top)을 가리키며 push/pop에 따라 계속 변한다. ①EAX는 산술 연산과 함수 반환값을 담는 누산기, ③EDX는 데이터 레지스터로 곱셈·나눗셈 등에 보조적으로 쓰인다.

오답 분석

보기①

EAX는 누산기 레지스터로 연산 결과·함수 반환값을 담는다.

보기②

ESP는 스택 최상단을 가리키는 포인터로 기준 주소 역할이 아니다.

보기③

EDX는 데이터 레지스터로 스택 프레임 기준과 무관하다.

보기④

정답. EBP는 스택 프레임의 기준 주소로 인자·지역변수 참조에 사용된다.

관련 이론 정리

함수가 호출되면 스택에 스택 프레임이 만들어지며 구조는 위에서부터 함수 인자 → 복귀 주소(RET) → 이전 EBP(SFP) → 지역변수 순이다. 버퍼 오버플로우 공격은 지역변수 버퍼에 경계 검사 없이 긴 데이터를 써서 SFP와 복귀 주소(RET)를 덮어쓰고, RET를 공격자의 셸코드 주소로 바꿔 임의 코드 실행을 노린다. SFP 1바이트만 덮는 변형이 프레임 포인터 오버라이트 공격이다. 대응책은 코드 수준의 안전한 함수 사용(strncpy·snprintf)과 경계 검사, 컴파일러의 스택 카나리(Stack Guard), OS의 ASLR(주소 무작위화)·DEP/NX(데이터 영역 실행 방지), 그리고 정적분석 도구 적용이다. 감리에서는 C/C++ 개발 시 안전 함수 사용과 컴파일 보호 옵션 적용을 점검한다.

주요 레지스터:

레지스터 역할

ESP 스택 최상단 포인터
EIP 다음 실행 명령 주소
EAX 누산기·반환값
실무 적용 사례

C/C++ 시스템 감리 시 스택 카나리·ASLR·DEP 등 컴파일·OS 보호 기능이 활성화됐는지, 위험 함수(gets·strcpy) 사용 여부를 정적분석으로 점검한다.

정답 해설

블록체인의 블록은 헤더(header)와 바디(body)로 나뉜다. 헤더에는 이전 블록 해시, 머클루트(트랜잭션들의 해시 트리 최상단 값), 타임스탬프, 난이도, 논스 등 메타데이터와 해시값이 들어가고, 실제 거래(트랜잭션) 원본 데이터는 바디에 저장된다.

①은 헤더에 "거래별 트랜잭션의 원본 값"이 저장된다고 서술했으나, 헤더에는 트랜잭션을 요약한 머클루트 해시만 들어가므로 틀렸다. 따라서 정답은 ①번이다.

②(거래기록 공개로 투명성 확보), ③(구성원 상호 검증으로 조작 곤란), ④(퍼블릭은 개방적이나 합의 참여자가 많아 확장성·속도가 떨어짐)는 모두 옳다.

관련 이론 정리

블록 헤더의 머클루트는 블록에 담긴 모든 트랜잭션을 두 개씩 해시해 올라가는 머클트리(해시 트리)의 최상단 값이다. 이 구조 덕분에 트랜잭션 하나만 바뀌어도 머클루트가 달라져 위·변조가 즉시 탐지되고, 전체 블록을 받지 않고도 특정 거래의 포함 여부를 검증하는 SPV(간이 검증)가 가능하다. 또 헤더에 이전 블록의 해시가 포함되어 블록들이 사슬로 엮이므로, 과거 블록을 수정하려면 이후 모든 블록의 작업을 다시 해야 해 사실상 불가능하다(불변성). 참여 범위에 따라 퍼블릭(누구나 참여, 익명성·탈중앙성 높으나 느림), 프라이빗(단일 조직, 빠르고 통제 가능), 컨소시엄(복수 기관 공동 운영)으로 나뉘며, 기업 환경에서는 성능·규제 때문에 프라이빗·컨소시엄이 선호된다. 감리에서는 블록체인 유형 선택의 타당성과 성능·개인정보 설계를 점검한다.

정답 해설

DLP(Data Leakage/Loss Prevention)는 중요 정보가 외부로 나가는 것을 탐지·차단하는 솔루션으로, 메일·웹·메신저·USB·출력물은 물론 블루투스·테더링 같은 통신 채널까지 통제한다. 또한 정규식·키워드·지문(fingerprint) 기반의 내용 기반(콘텐츠) 분석으로 주민등록번호·카드번호·설계도면 등 중요 정보를 식별한다. 문제 설명과 정확히 일치하므로 정답은 ②번이다.

①DRM은 문서 자체를 암호화해 사용 권한(열람·편집·인쇄)을 통제하는 기술로, 유출 경로 차단이 주목적이 아니다. ③디지털 워터마킹은 콘텐츠에 식별 정보를 삽입해 출처 추적·저작권 증명에 쓰이고, ④전자서명은 인증·무결성·부인방지를 제공하는 암호 기술이다.

관련 이론 정리

DLP는 데이터의 상태별로 보호 지점이 나뉜다. 전송 중 데이터(Data in Motion)는 네트워크 DLP가 메일·웹·FTP 트래픽을 검사하고, 저장 데이터(Data at Rest)는 서버·PC를 스캔해 중요 정보 보관 현황을 파악하며, 사용 중 데이터(Data in Use)는 엔드포인트 DLP가 USB 복사·출력·화면 캡처·클립보드·블루투스·테더링을 통제한다. 탐지 방식은 키워드·정규식·사전, 문서 지문(fingerprint), 기계학습 분류가 쓰인다. 운영 시에는 탐지(모니터링) → 경고 → 차단의 단계적 적용으로 업무 영향을 관리하며, 오탐이 많으면 사용자 우회가 늘어 실효성이 떨어진다. DRM과 상호 보완적으로, DLP는 나가는 경로를, DRM은 문서 자체를 지킨다. 감리에서는 DLP 정책의 실효성과 예외 처리 이력, 로그 점검 체계를 확인한다.

데이터 상태별 DLP:

상태 통제 지점
Data in Motion 네트워크(메일·웹·FTP)
Data at Rest 서버·PC 저장소 스캔
Data in Use 엔드포인트(USB·출력·블루투스·테더링)
실무 적용 사례

정보유출 방지 감리 시 DLP 정책이 탐지에 그치지 않고 차단으로 이어지는지, 예외 승인 절차와 오탐 처리, 로그 점검 주기를 확인한다.

감리사 시험 출제 포인트

DLP(유출 경로·채널 통제)와 DRM(문서 자체 권한)의 구분이 핵심이다. "블루투스·테더링 차단"은 DLP의 단서다.

정답 해설

PPTP(Point-to-Point Tunneling Protocol)는 이름 그대로 PPP(2계층 프로토콜)를 확장해 터널링 기능을 더한 것으로, 데이터링크 계층(2계층)에서 동작한다.

④는 PPP 기반이라는 앞부분은 맞지만 "네트워크 계층에 적용되는 프로토콜"이라고 서술해 계층이 틀렸다. 따라서 가장 적절하지 않은 것은 ④번이다.

①(L2F는 Cisco가 개발한 2계층 터널링), ②(IPSec은 3계층 터널링 표준), ③(L2TP는 L2F와 PPTP의 장점을 통합)은 모두 옳다. 즉 보기 중 3계층은 IPSec뿐이고 PPTP·L2F·L2TP는 모두 2계층이라는 점이 핵심이다.

오답 분석

보기①

옳음. L2F는 Cisco가 개발한 데이터링크 계층 터널링 프로토콜이다.

보기②

옳음. IPSec은 네트워크(3) 계층에서 동작하는 표준 터널링·보안 프로토콜이다.

보기③

옳음. L2TP는 L2F와 PPTP의 장점을 결합해 만들어졌다.

보기④

정답(부적절). PPTP는 PPP 기반의 2계층 프로토콜이며 네트워크 계층이 아니다.

관련 이론 정리

VPN 터널링 프로토콜은 동작 계층으로 정리하면 명확하다. 2계층에는 PPTP(마이크로소프트 주도, MPPE 암호화, 현재는 취약해 권장하지 않음), L2F(Cisco), L2TP(둘의 통합, 자체 암호화가 없어 IPSec과 결합해 사용)가 있다. 3계층에는 IPSec이 있으며 AH(무결성·인증)와 ESP(기밀성 포함)를 제공하고 전송·터널 모드로 동작한다. 응용/세션 계층에는 SSL/TLS VPN(SSTP·OpenVPN)이 있어 브라우저 기반 접속과 방화벽 통과에 유리하다. 계층이 낮을수록 상위 응용에 투명하게 보호를 제공하고, 높을수록 세밀한 응용 단위 통제가 가능하다. 보안 강도 면에서 PPTP는 폐기 대상이며 IPSec 또는 TLS 기반이 권장된다. 감리에서는 VPN 프로토콜 선택과 암호 알고리즘 강도를 점검한다.

VPN 프로토콜 계층:

계층 프로토콜
2(데이터링크) PPTP, L2F, L2TP
3(네트워크) IPSec
상위(SSL/TLS) SSTP, OpenVPN, SSL VPN
실무 적용 사례

VPN 감리 시 PPTP 등 취약 프로토콜 사용 여부를 확인하고 IPSec·TLS 기반으로 전환, 다중 인증 적용을 권고한다.

감리사 시험 출제 포인트

PPTP·L2F·L2TP = 2계층, IPSec = 3계층의 구분이 핵심이다. 계층을 바꿔 서술한 보기가 전형적 오답이다.
