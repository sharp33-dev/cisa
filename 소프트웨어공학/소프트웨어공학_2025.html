정답 해설

Clean Architecture의 Instability(불안정성) 공식:

I = Fan-out / (Fan-in + Fan-out)

여기서:
- Fan-out = 해당 패키지가 의존하는(depend on) 다른 패키지의 수 (출력 의존성)
- Fan-in = 해당 패키지에 의존하는(depend upon) 다른 패키지의 수 (입력 의존성)

P3 패키지 분석:
- 다이어그램에서 P3는 P1(C11), P4(C41, C42) 방향으로 화살표가 있는 구조
- P3에 의존하는 패키지(Fan-in): C31이 P3에 들어오는 방향 → Fan-in = 2 (C31, C11 등)
- P3가 의존하는 패키지(Fan-out): C32, C41, C42 방향 → Fan-out = 2

계산: I = 2 / (2 + 2) = 0.5

Instability 값의 의미:
- I = 0: 완전히 안정적(Stable) - 다른 패키지들이 의존하지만 자신은 어디에도 의존하지 않음
- I = 1: 완전히 불안정적(Instable) - 자신은 다른 패키지에 의존하지만 아무도 의존하지 않음
- 중간값: 부분적 안정성을 나타냄

Clean Architecture의 안정된 의존성 원칙(SDP: Stable Dependencies Principle)에 따르면, 패키지는 자신보다 더 안정적인 패키지에만 의존해야 한다. 즉, I값이 높은(불안정한) 패키지에서 낮은(안정적인) 패키지 방향으로 의존성이 흘러야 한다.

오답 분석

보기①(0.3)

Fan-in = 7, Fan-out = 3으로 잘못 계산한 경우 나올 수 있는 값. 다이어그램 화살표 방향을 잘못 읽었을 때 발생하는 오류이다.

보기②(0.4)

Fan-in = 3, Fan-out = 2로 계산한 경우. 다이어그램의 의존성 방향을 일부 잘못 읽은 경우이다.

보기④(0.6)

Fan-in = 2, Fan-out = 3으로 계산한 경우. 일부 의존성을 중복 계산하거나 방향을 혼동한 경우이다.

정답 해설

ISO/IEC 25010:2011의 품질 특성 구조:

ISO/IEC 25010은 ISO/IEC 9126을 개선하여 8가지 품질 특성을 정의한다. 이 중 유지보수성(Maintainability)의 부특성은 다음과 같다:

부특성 설명
모듈성(Modularity) 컴포넌트 변경이 다른 컴포넌트에 미치는 영향 최소화
재사용성(Reusability) 다른 시스템에서 재사용 가능한 정도
분석성(Analysability) 결함의 영향 진단 용이성
변경성(Modifiability) 품질을 저하시키지 않고 효과적 변경 가능
시험성(Testability) 테스트 기준 설정 및 검증 용이성

보기 ①은 "다른 하드웨어, 운영환경 등에 효과적으로 적응"하는 능력을 설명하고 있는데, 이는 이식성(Portability)의 부특성인 적응성(Adaptability)에 해당한다.

반면:
- 보기②는 시험성(Testability) → 유지보수성의 부특성 ✓
- 보기③은 변경성(Modifiability) → 유지보수성의 부특성 ✓
- 보기④는 모듈성(Modularity) → 유지보수성의 부특성 ✓

강의자료(3_2_소프트웨어공학_Part_III_품질관리)에 따르면 ISO 9126의 이식성 부특성으로 환경적응성이 있으며, ISO 25010에서도 이식성 아래 Adaptability가 정의되어 있다. 다른 운영환경 적응은 이식성의 영역이다.

오답 분석

보기②

시험성(Testability) = 시스템에 대한 테스트 기준을 설정하고 충족 여부를 효과적으로 테스트할 수 있는 능력 → 유지보수성의 올바른 부특성이므로 오답이 아님.

보기③

변경성(Modifiability) = 새로운 결함을 유발하지 않고 기존 품질을 악화시키지 않으면서 효과적으로 변경 가능한 정도 → 유지보수성의 올바른 부특성이므로 오답이 아님.

보기④

모듈성(Modularity) = 특정 컴포넌트에 가해진 변경이 다른 컴포넌트에 미치는 영향 최소화 → 유지보수성의 올바른 부특성이므로 오답이 아님.

관련 이론 정리

ISO/IEC 25010 주요 품질 특성:

특성 부특성
기능성 기능 완전성, 기능 정확성, 기능 적합성
성능 효율성 시간 효율, 자원 효율, 용량
호환성 공존성, 상호운용성

9126 vs 25010 변경사항:
- 9126의 6개 특성 → 25010은 8개 특성
- 호환성(Compatibility)과 보안성(Security) 추가
- 이식성 유지하면서 적응성을 이식성 부특성으로 명확화

실무 적용 사례

감리 시 SW 품질 검토에서 ISO 25010 기준을 적용할 때:
- 유지보수성 평가: 코드 변경 시 영향도 분석, 단위 테스트 커버리지 측정
- 이식성 평가: 다른 OS/환경에서 동작 여부, 컨테이너화 수준 확인

감리사 시험 출제 포인트

정답 해설

시퀀스 다이어그램의 핵심 요소를 분석하면:

  1. UI가 Main에 service 요청
  2. Main이 데이터를 split(분할)
  3. par 블록: Main이 Controller1에 do(data1), Controller2에 do(data2)를 동시에(병렬로) 요청
  4. Main이 결과를 merge(병합)하여 result 반환

이는 전형적인 Master-Slave 패턴의 특징이다:
- Master(주인): Main 객체 — 작업을 분할하고, 여러 Slave에 위임하며, 결과를 통합
- Slave(하인): Controller1, Controller2 — Master의 지시에 따라 독립적으로 처리 수행
- 병렬 처리: par 블록으로 동시 실행
- 분할-처리-병합(Split-Execute-Merge) 구조

Master-Slave 패턴의 주요 활용:
- 병렬 계산이 필요한 복잡한 처리
- 오류 허용(Fault Tolerance): 여러 Slave 중 하나가 실패해도 계속 동작
- 정밀도 향상: 여러 결과를 비교하여 최적값 선택

오답 분석

보기①(Pipe and Filter)

파이프 필터는 데이터가 순차적으로 여러 처리 단계(필터)를 거치는 패턴. 병렬처리가 아닌 순차적 변환이 특징. "데이터 처리가 일련의 흐름을 가지고 있고, 흐름 상에서 중간에 처리"되는 구조이다. 이 문제의 병렬 분기와는 다르다.

보기③(Front Controller)

프론트 컨트롤러는 모든 요청을 하나의 진입점(Front Controller)에서 처리하는 패턴. 주로 웹 MVC에서 사용되며, 분할-병합 구조가 없다.

보기④(MVC)

MVC는 Model-View-Controller 패턴으로 UI(View), 비즈니스 로직(Model), 제어(Controller)를 분리하는 패턴. 병렬 작업 분배가 핵심이 아니다.

관련 이론 정리

주요 아키텍처 패턴 비교:

패턴 특징 사용 목적

- 관계: extends (상속, 빈 삼각형)

- 다. 장애로부터 복구될 때까지의 평균 시간(MTTR) → 회복성(Recoverability) 척도 → 신뢰성 ✓
- 라. 비인가된 접근에 의한 데이터 변조 방지 비율 → 보안성(Security)의 무결성(Integrity) 척도 → 신뢰성과 무관

라. "인위적 노이즈로 잘못된 작업 유도"

다음 의무사항을 가지는 오픈소스 라이선스로 가장 적절한 것은?

  • 가. 각 복제본에 적절한 저작권 안내와 보증책임이 없음을 명시할 것
  • 나. 본 라이선스를 언급하는 안내 사항과 보증책임 관련 고지사항을 원본 그대로 유지할 것

  • 라. 라이브러리 형태로의 수정을 허용하며, 만약 수정한 경우 수정사실과 날짜를 파일에 명기할 것

  • 사. 사용자가 라이브러리를 수정해도 응용 프로그램을 사용할 수 있도록 허용할 것

정답 해설

각 라이선스 핵심 특징:

라이선스 카피레프트 특징
GPL 강한 카피레프트 파생 저작물 전체 소스 공개 의무
LGPL 약한 카피레프트 라이브러리 사용만으로 소스 공개 면제
MPL 약한 카피레프트 파일 단위 카피레프트
BSD 허용적(Permissive) 소스 공개 의무 거의 없음
MIT 허용적 가장 자유로운 라이선스

문제의 의무사항 분석:
- 라. 라이브러리 형태로의 수정 허용, 수정사실 명기 → LGPL 특유의 라이브러리 관련 조항
- 바. 응용 프로그램에서 라이브러리 사용 명시 → LGPL의 고지 의무
- 사. 라이브러리 수정 후에도 응용 프로그램 사용 허용 → LGPL의 핵심: 응용 프로그램에는 카피레프트 전파 없음

LGPL의 핵심 차별점: GPL 라이브러리를 사용하는 응용 프로그램은 GPL로 배포해야 하지만, LGPL 라이브러리를 사용하는 응용 프로그램은 소스 공개 의무가 없다. 단, 라이브러리 자체 수정본은 공개해야 한다.

오답 분석

보기①(MPL)

MPL은 파일 단위 카피레프트로, 수정한 파일만 MPL로 공개하면 된다. 라이브러리 관련 특별 조항(바, 사)이 없다.

보기②(GPL)

GPL은 강한 카피레프트로 라이브러리를 포함하는 응용 프로그램 전체를 GPL로 배포해야 한다. "사용자가 라이브러리를 수정해도 응용 프로그램 사용 허용(사)"이라는 약화된 조항이 없다.

보기④(BSD)

BSD는 허용적 라이선스로 소스 공개 의무가 거의 없다. 마(소스코드 제공 또는 약정서)나 다(라이선스 사본 제공) 등의 강제 조항이 없다.

정답 해설

컨테이너의 핵심 특징:

컨테이너는 VM(가상머신)과 달리 호스트 OS의 커널을 공유하면서 격리된 환경을 제공하는 경량 가상화 기술이다.

각 설명의 적절성:

  • 가. "독립된 환경에서 동작하는 가상화의 일종" → 부분적으로 맞으나, 정확하지 않음. 컨테이너는 OS 수준의 가상화이며, VM처럼 완전한 독립 환경이 아닌 OS 커널을 공유하는 프로세스 격리. → 부적절 (포함)

  • 나. "컨테이너 간 운영체제 공유" → 정확: 컨테이너는 호스트 OS 커널을 공유함 → 적절 (미포함)

  • 다. "쿠버네티스 = 컨테이너 오케스트레이션 도구" → 정확 → 적절 (미포함)

  • 라. "컨테이너는 서로 다른 운영체제 사용 가능" → 부적절: 컨테이너는 호스트 OS 커널을 공유하므로, 다른 OS를 사용할 수 없다 (Linux 컨테이너는 Linux 호스트 커널 필요). VM은 다른 OS 가능. → 포함

  • 마. "컨테이너에서는 하이퍼바이저가 필요" → 부적절: 하이퍼바이저는 VM에서 필요한 기술. 컨테이너는 Docker 엔진 등 컨테이너 런타임을 사용. → 포함

  • 바. "모놀리식 아키텍처인 경우 컨테이너 기반 개발이 유리" → 부적절: 컨테이너는 마이크로서비스 아키텍처(MSA)에 유리함. 모놀리식 아키텍처는 단일 배포 단위이므로 컨테이너의 장점(독립 배포, 스케일 아웃)을 살리기 어려움. → 포함

적절하지 않은 것: 가, 라, 마, 바 → ④번

MSA 설계 절차의 논리적 순서:

  1. 다. 목표 식별 (신뢰성, 보안성 등 시스템이 달성해야 하는 작은 목표 식별)

  4. 라. 영향 분석 (현재 상태 진단, 변경 시 영향 분석)
    → 설계 후 현재 시스템에서 마이그레이션 시 영향 파악

정답 해설

강의자료(1_2_소프트웨어공학_Part_I)에 따르면:

"데이터 흐름 아키텍처 스타일(Data Flow Architecture Style): 소프트웨어 시스템을 연속적으로 연결된 입력데이터에 대한 일련의 변형 동작들로서 구성. 연속적인 컴포넌트들에 의한 데이터의 점진적 변형 — 일괄 순차형, 파이프 필터형 아키텍처 스타일"

Pipe and Filter 패턴 특징:
- 파이프(Pipe): 필터 간 데이터 전달 통로
- 필터(Filter): 데이터를 받아 변환하여 다음 단계로 전달하는 처리 단위
- 데이터는 일련의 필터를 거치며 점진적으로 변환
- 각 필터는 독립적으로 교체 가능 (재사용성)
- 대표 사례: Unix 파이프(|), ETL 프로세스, 컴파일러 단계 처리

문제 설명: "일련의 흐름 → 중간 처리 → 변환 완료 → 최종 목적지" = 전형적인 Pipe and Filter

오답 분석

보기①(Peer to Peer View)

P2P는 노드들이 동등한 역할로 서로 통신하는 구조. 중앙 서버 없이 각 노드가 클라이언트이자 서버. 데이터 흐름 변환과 무관.

보기②(Client-Server View)

클라이언트가 서버에 요청하고 서버가 응답하는 구조. 데이터 변환 파이프라인과 다름.

보기④(Publish-Subscribe View)

발행자가 이벤트를 발행하면 구독자가 수신하는 이벤트 기반 비동기 패턴. 데이터 변환 흐름과 다름.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

McCabe의 순환 복잡도 계산 방법:

방법 1 (결정 노드 수 이용):

V(G) = 결정 노드(분기점)의 수 + 1

코드에서 결정 노드(분기 발생 지점):
1. if x > 0: → 결정 노드 1개
2. if y > 0: → 결정 노드 1개
3. for i from 1 to 5: → 결정 노드 1개

결정 노드 수 = 3개
V(G) = 3 + 1 = 4

방법 2 (CFG 이용):

V(G) = E(엣지 수) - N(노드 수) + 2

순환 복잡도의 의미:
- 4 = 이 함수를 완전히 테스트하기 위한 최소 독립 경로 수
- 즉, 최소 4개의 테스트 케이스가 필요

오답 분석

보기①(3)

결정 노드만 세고 +1을 누락한 경우.

보기③(5)

else 분기를 별도 결정으로 잘못 카운트한 경우. if-else는 하나의 결정 노드.

보기④(6)

모든 분기 경로를 개별로 계산하는 오류.

관련 이론 정리

McCabe 순환 복잡도 기준:

V(G) 값 복잡도 수준 의미
1~10 낮음 단순한 코드
11~20 보통 적당한 복잡도
21~50 높음 재구조화 권고
50 이상 매우 높음 즉시 재작성 필요

시험 포인트: V(G) = 결정점 수 + 1

- Staff --(점선화살표)--> Task: 의존 (지역변수로 사용) ✓

- "보통의 경우는 Normal 시나리오 사용" → Normal = 기본(Base) 유스케이스
- "특정 조건 하에서 Conditional 시나리오 사용" → Conditional = 확장(Extension) 유스케이스, 조건부 실행

따라서:
- «extend» 관계 사용 (조건부 = 선택적)
- 화살표 방향: Conditional(확장) → Normal(기본)

강의자료(2_1, p.21)에서도 확인:

"«extend»는 바깥에서 안으로 들어온다" = 확장 UC → 기본 UC 방향

→ Normal ←(«extend»)— Conditional = ④번

오답 분석

보기①(Normal →«include»→ Conditional)

include 관계는 "반드시" 포함이므로 조건부 실행과 맞지 않음. 또한 화살표 방향도 "포함하는 UC → 포함되는 UC" 방향.

보기②(Normal ←«include»— Conditional)

include 스테레오타입 사용은 오류. "특정 조건 시에만" = extend 사용해야 함.

보기③(Normal →«extend»→ Conditional)

«extend» 스테레오타입은 맞으나 화살표 방향이 반대. extend는 확장 UC → 기본 UC 방향이어야 함.

관련 이론 정리

유스케이스 관계 정리:

«include» 예시:
  상품주문 ──«include»──> 로그인
  (주문 시 반드시 로그인 필요)
  화살표: 기본UC → 포함UC

«extend» 예시:
  도서대출 <──«extend»── 연체료부과
  (대출 시 연체 조건이면 연체료 부과)
  화살표: 확장UC → 기본UC

암기법:
- include = 강제 포함 → "포함하는 쪽"에서 "포함되는 쪽"으로 화살표
- extend = 조건부 확장 → "확장하는 쪽"(조건적 행동)에서 "기본 쪽"으로 화살표

실무 적용 사례

전자정부 시스템 요구사항 분석 시 유스케이스 다이어그램을 작성할 때:
- «include»: 공통 기능(인증, 로깅 등)을 여러 UC에서 반드시 사용
- «extend»: 예외 처리, 옵션 기능(특정 사용자 그룹에만 해당) 등 조건부 기능

감리 시 유스케이스 다이어그램에서 include/extend 화살표 방향 오류를 검토한다.

정답 해설

강의자료(2_2_디자인_패턴)에 따르면:

State Pattern: "상태가 변경될 때 마다 별도의 행위를 지정해야 되는 객체를 정의 및 조직화하는 방법을 제공하는 행위패턴. Client 상태에서 발생 가능한 상태(온, 오프 등)마다 수행되는 클래스"

문제 분석:
- (가) 리팩토링 전: 하나의 클래스에 if(주간)/else if(야간) 분기로 모든 상태별 동작 처리
- (나) 리팩토링 후: 주간 클래스, 야간 클래스로 분리 → 각 상태를 별도 클래스로 캡슐화

이것이 전형적인 State Pattern:
- Context(경비시스템): 현재 상태 객체를 유지
- State 인터페이스: 공통 메서드 정의
- ConcreteState(주간 클래스, 야간 클래스): 상태별 동작 구현

State Pattern은 전략 패턴(Strategy)과 유사하지만, 상태 전이에 초점을 맞추는 것이 차이점이다.

보기④(Template Method Pattern)

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

강의자료(3_1_소프트웨어공학_Part_III_테스트_유지관리)의 리팩토링 기법 분류:

문제 패턴 분석:
- 리팩토링 전: 하나의 Shape 클래스에 switch(_typecode) 문으로 LINE, RECTANGLE, OVAL 타입별 다른 동작 처리
- 리팩토링 후: Shape를 추상 클래스로, ShapeLine, ShapeRectangle, ShapeOval로 상속 구현

이 패턴이 "Replace Type Code with Subclasses":
- 분류 코드(type code)에 따라 동작이 달라지는 경우
- 각 분류 코드를 별도 서브클래스로 추출
- 다형성(Polymorphism)을 이용하여 switch/if 분기 제거

"Replace Type Code with Class"는 서브클래스 대신 단순히 클래스로 대체하는 경우 (switch가 아닌 단순 타입 코드를 클래스 타입으로).

오답 분석

보기①(Pull Up Method)

서브클래스의 동일한 메서드를 상위 클래스로 올리는 리팩토링. 반대 방향.

보기②(Replace Type Code with Class)

타입 코드를 클래스로 단순 대체. 서브클래스 계층 구조 생성이 아님.

보기③(Replace Parameter with Method)

메서드 파라미터 대신 해당 값을 계산하는 메서드를 호출하는 리팩토링.

관련 이론 정리

주요 리팩토링 기법:

기법 적용 상황
Extract Method 긴 메서드에서 코드 추출
Pull Up Method 서브클래스 중복 메서드 상위로
Replace Type Code with Subclasses switch 타입 분기 → 서브클래스
Replace Type Code with Class 단순 타입 코드 → 클래스
Move Method 메서드를 적절한 클래스로 이동
Inline Method 단순한 위임 메서드 제거

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

임계 경로(Critical Path) 정의:
- Slack Time(여유 시간)이 0인 작업들로 구성된 경로
- 이 경로의 작업이 지연되면 프로젝트 전체 일정 지연
- 짙은색 박스(slack time) 없이 빽빽하게 채워진 작업들

바 차트에서 slack time 유무 분석:
- A의 T4, T8, T11, T12 중 일부는 slack time 존재 → 비임계
- B의 T1, T3, T9 → slack time 없음 → 임계 경로 ✓
- C의 T2, T6, T10 중 일부는 slack time 존재 → 비임계
- D, E의 T7, T5는 slack time 존재 → 비임계

따라서 임계 경로 상 작업만 담당하는 인력 = B (T1, T3, T9 모두 임계)

관련 이론 정리

CPM(Critical Path Method) 핵심:

LFT 가장 늦은 완료 시간
Slack LST - EST = LFT - EFT
