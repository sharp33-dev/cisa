영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

네트워크 장비는 동작하는 OSI 계층에 따라 구분된다.

  • 가(옳음): 라우터는 IP 주소를 보고 경로를 결정하므로 네트워크 계층(3계층) 장비다.
  • 나(옳음): 브리지(및 L2 스위치)는 MAC 주소로 프레임을 전달하므로 데이터링크 계층(2계층) 장비다.
  • 다(옳음): 허브(리피터)는 전기 신호를 증폭·재전송할 뿐 주소를 해석하지 않으므로 물리 계층(1계층) 장비다.

관련 이론 정리

OSI 7계층은 물리-데이터링크-네트워크-전송-세션-표현-응용으로 구성되며, 각 계층에서 동작하는 대표 장비가 있다. 1계층은 허브·리피터로 신호만 다루어 모든 포트가 하나의 충돌·브로드캐스트 도메인을 공유한다. 2계층은 브리지·L2 스위치로 MAC 주소 테이블을 학습해 충돌 도메인을 포트별로 분할한다. 3계층은 라우터·L3 스위치로 IP 기반 라우팅을 하며 브로드캐스트 도메인까지 분할한다. 4계층 이상에서는 L4/L7 스위치·게이트웨이가 포트·응용 정보를 보고 부하분산이나 프로토콜 변환을 수행한다. 감리에서는 네트워크 구성도상 장비 배치가 세그먼트 설계·보안 경계와 일치하는지 점검한다.

계층별 장비:

계층 장비 식별 기준

2(데이터링크) 브리지·L2 스위치 MAC
3(네트워크) 라우터·L3 스위치 IP
4~7 L4/L7 스위치·게이트웨이 포트·응용
실무 적용 사례

네트워크 구성 감리 시 계층별 장비 배치, 브로드캐스트 도메인 분할(라우터·VLAN), 보안 경계(방화벽 위치)가 설계 의도에 맞는지 검토한다.

감리사 시험 출제 포인트

라우터(3)·브리지(2)·허브(1)의 계층 매핑은 기본 중의 기본이다. 게이트웨이(상위 계층), L4 스위치(전송) 등도 함께 정리한다.

관련 이론 정리

네트워크 토폴로지는 노드 연결 형태로, 각기 비용·신뢰성·확장성이 다르다. 성형(star)은 중앙 장치에 모든 노드를 연결해 장애 격리·관리가 쉽고 확장이 간단하지만 중앙 장치가 단일 장애점(SPOF)이고 케이블 비용이 크다(현대 이더넷 LAN의 표준 형태). 버스(bus)는 하나의 공유 회선에 노드를 붙여 비용이 낮지만 회선 장애 시 전체가 마비되고 충돌이 발생한다. 링(ring)은 순환 연결로 토큰 방식 제어가 가능하나 한 노드 장애가 전체에 영향을 줄 수 있어 이중 링으로 보완한다. 메시(mesh)는 노드 간 직접 연결로 신뢰성이 최고지만 링크 수가 n(n−1)/2로 급증해 비용이 크다. 감리에서는 가용성 요구에 맞는 토폴로지와 중앙 장치 이중화를 점검한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

RARP(Reverse ARP)는 자신의 링크 계층 주소(MAC)를 알고 있으나 IP 주소를 모르는 호스트가, MAC 주소를 이용해 네트워크 계층 주소(IP)를 알아내는 프로토콜이다. 문제의 "링크 계층 주소를 네트워크 계층 주소로 변환"과 정확히 일치하므로 정답은 ②번이다.

①ARP는 정반대로 IP→MAC 변환이고, ③ICMP는 오류·진단 메시지, ④NAT는 사설↔공인 IP 변환으로 계층 간 주소 변환이 아니다. 2023년 84번(ARP)과 방향이 반대인 문항으로, 두 해에 걸쳐 짝으로 출제된 셈이다.

오답 분석

보기①

ARP는 IP 주소를 MAC 주소로 변환하는 것으로 방향이 반대다.

보기②

정답. RARP는 MAC 주소로 IP 주소를 알아낸다.

보기③

ICMP는 오류 보고·진단(ping 등)용으로 주소 변환 기능이 없다.

보기④

NAT는 사설 IP와 공인 IP 간 변환으로, 계층 간(MAC↔IP) 변환이 아니다.

관련 이론 정리

RARP는 디스크가 없는 워크스테이션(diskless)처럼 부팅 시 자신의 IP를 모르는 장비가 서버로부터 IP를 받기 위해 고안됐다. 동작은 MAC을 담은 요청을 브로드캐스트하면 RARP 서버가 매핑 테이블을 조회해 IP를 응답하는 방식이다. 그러나 RARP는 동일 링크 내에서만 동작하고 IP 외 부가 정보(게이트웨이·DNS)를 줄 수 없어, 이후 BOOTP를 거쳐 DHCP로 대체되었다. DHCP는 IP·서브넷마스크·게이트웨이·DNS를 임대(lease) 방식으로 자동 할당하며, 릴레이 에이전트를 통해 다른 서브넷의 서버와도 통신한다. 감리에서는 IP 자동 할당 체계(DHCP)의 이중화·주소 풀 관리·보안(스푸핑 방지)을 점검한다.

주소 변환 프로토콜:

프로토콜 변환 방향

RARP MAC → IP
NAT 사설 IP ↔ 공인 IP
ICMP (변환 아님) 오류·진단
실무 적용 사례

네트워크 감리 시 IP 자동 할당이 DHCP로 적절히 운영되는지, 주소 풀 고갈·중복 할당·DHCP 스푸핑 대책(스누핑)이 있는지 점검한다.

CPU 스케줄링은 실행 중인 프로세스의 CPU를 강제로 회수할 수 있는지에 따라 나뉜다. 비선점(non-preemptive)은 프로세스가 스스로 종료하거나 I/O로 블록될 때까지 CPU를 유지하며, FCFS·SJF(비선점)·우선순위(비선점)가 해당한다. 문맥 교환이 적어 오버헤드가 낮지만 짧은 작업이 긴 작업 뒤에서 오래 기다리는 호위 효과(convoy effect)와 응답성 저하가 생긴다. 선점(preemptive)은 시간 할당량 만료나 더 높은 우선순위 프로세스 도착 시 CPU를 회수하며, 라운드로빈·SRT·선점형 우선순위·다단계 피드백 큐가 해당한다. 응답성과 공정성이 좋아 대화형·실시간 시스템에 적합하지만 문맥 교환 오버헤드가 크고 공유 자원 동기화가 필요하다. 감리에서는 워크로드 특성에 맞는 스케줄링 정책과 응답시간 목표 달성을 점검한다.

시스템 성능 감리 시 대화형·실시간 업무에 선점형 스케줄링과 적정 타임 퀀텀이 적용됐는지, 문맥 교환 과다로 오버헤드가 크지 않은지 점검한다.

정답 해설

라우팅 알고리즘과 방식의 대응은 다음과 같다.

  • 거리 벡터(Distance Vector) 라우팅 → Bellman-Ford 알고리즘 (예: RIP)
  • 링크 상태(Link State) 라우팅 → Dijkstra 최단경로 알고리즘 (예: OSPF)

④는 "Bellman-Ford를 사용하는 링크 상태 라우팅과 Dijkstra를 사용하는 거리 벡터 라우팅"이라고 서술해 두 알고리즘과 방식의 대응이 서로 뒤바뀌어 있으므로 틀린 설명이다. 따라서 가장 적절하지 않은 것은 ④번이다.

①(정적·동적 구성), ②(유니캐스트·멀티캐스트 라우팅 프로토콜 존재), ③(IGP·EGP 구분)은 모두 옳다.

보기①

옳음. 라우팅 테이블은 관리자가 수동 설정(정적)하거나 프로토콜로 자동 구성(동적)할 수 있다.

보기②

옳음. 유니캐스트용(RIP·OSPF·BGP)과 멀티캐스트용(PIM·DVMRP·MOSPF)이 각각 존재한다.

보기③

옳음. AS 내부는 IGP(RIP·OSPF), AS 간은 EGP(BGP)로 구분된다.

보기④

정답(부적절). Bellman-Ford=거리벡터, Dijkstra=링크상태로, 서술이 뒤바뀌었다.

관련 이론 정리

거리 벡터는 각 라우터가 이웃에게 자신의 전체 경로표(목적지-거리)를 주기적으로 알리고, 받은 정보로 최단 거리를 갱신하는 Bellman-Ford 기반 방식이다. 구현이 단순하나 수렴이 느리고 카운트 투 인피니티(라우팅 루프) 문제가 있어 스플릿 호라이즌·홀드다운으로 완화한다. 링크 상태는 각 라우터가 자신의 인접 링크 상태(LSA)를 전체에 플러딩해 모든 라우터가 동일한 토폴로지 지도를 갖고, 각자 Dijkstra 알고리즘으로 최단 경로 트리를 계산하는 방식이다. 수렴이 빠르고 루프에 강해 대규모 망에 적합하지만 계산·메모리 부담이 크다. AS 간 라우팅인 BGP는 정책 기반 경로 벡터 방식이다. 감리에서는 망 규모·수렴 요구에 맞는 프로토콜 선택과 인증·필터링을 점검한다.

링크 상태 Dijkstra OSPF, IS-IS
경로 벡터 정책 기반 BGP
실무 적용 사례

라우팅 설계 감리 시 망 규모에 따른 프로토콜 선택(소규모 RIP, 대규모 OSPF), 수렴 시간·루프 방지 설정의 적정성을 검토한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

TCP/UDP 체크섬 계산에 사용되는 가상 헤더(pseudoheader)는 IP 헤더에서 가져온 송신자 IP 주소, 수신자 IP 주소, 프로토콜 번호(TCP=6/UDP=17), 세그먼트 길이로 구성된다. 이는 세그먼트가 올바른 종단(IP)과 올바른 프로토콜로 전달됐는지까지 검증하기 위함이다.

④의 TTL(생존시간)은 라우터를 지날 때마다 1씩 감소하며 값이 변하는 필드다. 값이 홉마다 달라지면 송신 측과 수신 측의 체크섬 계산 결과가 일치할 수 없으므로 가상 헤더에 포함될 수 없다. 따라서 가장 적절하지 않은 것은 ④번이다.

오답 분석

보기①

프로토콜 번호는 가상 헤더에 포함되어 상위 프로토콜 오배달을 검출한다.

보기②

수신자 IP 주소는 가상 헤더 구성 요소다.

보기③

송신자 IP 주소도 가상 헤더 구성 요소다.

보기④

정답(부적절). TTL은 경로상에서 값이 변하므로 가상 헤더에 포함되지 않는다.

관련 이론 정리

TCP·UDP 체크섬은 세그먼트 헤더와 데이터뿐 아니라 가상 헤더를 함께 포함해 계산한다. 가상 헤더는 실제로 전송되지는 않고 계산에만 쓰이는 개념적 구조로, IP 주소와 프로토콜을 포함시켜 엉뚱한 호스트나 다른 프로토콜로 잘못 전달된 세그먼트를 검출한다. 계산은 16비트 단위 1의 보수 합의 1의 보수 방식으로 수행된다. UDP에서는 체크섬이 선택 사항(IPv4)이지만 IPv6에서는 필수다. 한편 IP 헤더 자체의 체크섬은 헤더만 대상으로 하며, TTL이 홉마다 바뀌므로 라우터가 매 홉 재계산한다(IPv6는 IP 헤더 체크섬을 아예 없앴다). 감리에서는 프로토콜 스택 구현·NAT 환경에서 체크섬 재계산 처리가 올바른지 점검한다.

수신자 IP 주소 O
프로토콜 번호 O
세그먼트 길이 O

실무 적용 사례

NAT·로드밸런서 도입 감리 시 IP 주소 변경에 따른 TCP/UDP 체크섬 재계산이 정상 처리되어 패킷 폐기가 발생하지 않는지 점검한다.

②의 Register Hazard는 표준 분류에 존재하지 않는 명칭이다(레지스터 관련 충돌은 데이터 해저드에 포함된다). 따라서 가장 적절하지 않은 것은 ②번이다.

관련 이론 정리

파이프라인은 명령어 실행을 단계(IF·ID·EX·MEM·WB)로 나눠 중첩 수행함으로써 처리량을 높이지만, 해저드가 발생하면 스톨(stall)로 성능이 떨어진다. 구조적 해저드는 자원 중복(명령/데이터 캐시 분리)으로, 데이터 해저드는 포워딩(bypassing)·스톨·컴파일러의 명령 재배치로, 제어 해저드는 분기 예측(branch prediction)·지연 분기·투기적 실행으로 완화한다. 데이터 해저드는 의존 유형에 따라 RAW(진정 의존, 가장 흔함), WAR(반의존), WAW(출력 의존)로 나뉘며, 레지스터 재명명(register renaming)으로 WAR·WAW를 해소한다. 감리·성능 분석에서는 CPI 증가 요인으로 해저드·캐시 미스를 함께 검토한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

Erasure Coding(EC)은 하둡 3.0에서 새로 도입된 저장 효율화 기술이다. 기존 HDFS는 데이터를 3벌 복제해 저장 오버헤드가 200%(원본 대비 3배)였는데, EC는 데이터를 조각내고 패리티 조각을 생성해 약 50% 수준의 오버헤드로 동일한 내결함성을 제공한다(RAID의 패리티 개념을 분산 저장에 적용). 따라서 정답은 ①번이다.

②Resource Manager와 ④YARN은 하둡 2.0에서 도입된 자원 관리 구성요소이고, ③Secondary NameNode는 하둡 1.0부터 존재하던 메타데이터 체크포인트 노드다.

보기②

관련 이론 정리

하둡은 버전마다 핵심 변화가 있다. 1.x는 HDFS + MapReduce(JobTracker·TaskTracker) 구조로, NameNode가 단일 장애점이었다. 2.x는 YARN을 도입해 자원 관리(ResourceManager·NodeManager)와 처리 엔진을 분리함으로써 MapReduce 외 Spark·Tez 등 다양한 엔진을 수용했고, NameNode HA(고가용성)와 페더레이션을 지원했다. 3.x는 Erasure Coding으로 저장 효율을 크게 높이고, NameNode 다중 스탠바이, YARN 타임라인 서비스 v2, GPU·FPGA 등 자원 유형 지원을 추가했다. EC는 저장 공간을 절약하지만 복구 시 재구성 연산·네트워크 비용이 커서 접근 빈도가 낮은 콜드 데이터에 주로 적용한다. 감리에서는 데이터 특성에 맞는 복제/EC 정책과 용량·성능 균형을 점검한다.

정답 해설

GAN(Generative Adversarial Network)은 가짜 데이터를 만드는 생성자(Generator)와 진짜/가짜를 판별하는 판별자(Discriminator) 두 신경망이 서로 경쟁(적대적)하며 함께 성능을 향상시키는 딥러닝 구조다. 문제의 "두 네트워크 간 상호 경쟁을 통한 성능 향상"이 GAN의 정의와 정확히 일치하므로 정답은 ②번이다.

①CNN은 합성곱으로 이미지 특징을 추출하는 구조, ③RNN은 순환 구조로 시퀀스를 처리하는 구조이며, ④SVM은 신경망이 아닌 마진 기반 분류기다.

오답 분석

보기①

보기②

보기③

관련 이론 정리

GAN은 생성자가 잠재 벡터로부터 데이터를 만들고, 판별자가 이를 실제 데이터와 구별하도록 학습하는 미니맥스 게임이다. 생성자는 판별자를 속이도록, 판별자는 속지 않도록 번갈아 학습하며 이론적으로 생성 분포가 실제 분포에 수렴한다. 이미지 생성·초해상도·스타일 변환·데이터 증강에 널리 쓰이지만, 학습이 불안정하고 생성자가 소수 패턴만 만들어내는 모드 붕괴(mode collapse)가 발생할 수 있어 WGAN·조건부 GAN 등 변형이 제안됐다. 같은 생성모델인 VAE는 확률적 잠재변수와 재구성 손실을 쓰는 접근으로 학습이 안정적이나 생성물이 흐릿한 경향이 있다. 최근에는 확산모델(diffusion)이 이미지 생성의 주류가 됐다. 감리에서는 생성형 AI의 데이터 저작권·딥페이크 오용·품질 평가를 점검한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

RAID 5는 데이터를 블록 단위로 스트라이핑하면서 패리티를 모든 디스크에 분산 저장한다. 패리티가 의미를 가지려면 최소 2개의 데이터 조각 + 1개의 패리티 조각이 필요하므로, 구축에 요구되는 최소 디스크 수는 3개다. 따라서 정답은 ②번이다.

디스크가 2개면 패리티가 데이터 복사본과 다를 바 없어(사실상 미러링) RAID 5의 의미가 없다. 참고로 이중 패리티를 쓰는 RAID 6는 최소 4개(데이터 2 + 패리티 2), RAID 0·RAID 1은 최소 2개다.

보기③

5개는 최소 요건보다 많다.

관련 이론 정리

RAID 레벨별 최소 디스크 수와 용량 효율은 설계 시 핵심 고려사항이다. RAID 0(스트라이핑)은 최소 2개, 가용 용량 100%지만 내결함성이 없다. RAID 1(미러링)은 최소 2개, 용량 효율 50%, 1개 장애 허용. RAID 5는 최소 3개, 가용 용량 (N−1)/N, 1개 장애 허용. RAID 6는 최소 4개, 가용 용량 (N−2)/N, 2개 장애 허용. RAID 10(1+0)은 최소 4개, 용량 효율 50%로 성능과 안정성을 함께 확보한다. 대용량 디스크 환경에서는 RAID 5 재구축 시간이 길어 2차 장애 위험이 커지므로 RAID 6·RAID 10을 권고하는 추세다. 감리에서는 업무 중요도·성능 요구에 따른 레벨 선택과 핫스페어·백업 병행을 점검한다.

정답 해설

세 방식은 프레임을 언제, 얼마나 읽고 전송을 시작하는지, 즉 전송 로직(스위칭 방식)의 차이로 구분된다.

  • Cut-Through: 목적지 MAC 주소(앞 6바이트)만 읽고 즉시 전송 → 지연 최소, 오류 프레임도 전달
  • Interim(Fragment-Free) Cut-Through: 앞 64바이트를 읽어 충돌 조각을 걸러낸 뒤 전송 → 절충
  • Store-and-Forward: 프레임 전체를 수신해 FCS(CRC) 검사 후 전송 → 지연 크지만 오류 차단

따라서 구분 기준은 ④ 전송 로직이다. 포트 인터페이스·학습 로직·입출력 포트는 이 분류와 무관하다.

관련 이론 정리

L2 스위치는 MAC 주소 테이블을 학습해 프레임을 목적지 포트로 전달하는데, 전송 시점에 따라 성능·신뢰성이 달라진다. Store-and-Forward는 전체 프레임을 버퍼링해 CRC를 검증하므로 오류·런트(짧은) 프레임을 걸러내지만 프레임 길이에 비례한 지연이 생기고, 속도가 다른 포트 간 전송(속도 변환)에 필수적이다. Cut-Through는 지연이 매우 작아 저지연이 중요한 환경(금융 트레이딩·HPC)에 유리하나 손상 프레임도 그대로 전달한다. Fragment-Free는 충돌이 대부분 앞 64바이트에서 발생한다는 점을 이용한 절충안이다. 현대 스위치는 오류율이 높아지면 자동으로 Store-and-Forward로 전환하는 적응형 방식을 쓰기도 한다. 감리에서는 지연 요구와 오류 차단 필요성에 맞는 스위칭 방식을 검토한다.

Store-and-Forward 전체 프레임 큼 있음(CRC)

정답 해설

IPv4 패킷이 단편화되면 각 조각은 별도의 IP 헤더를 갖는데, 필드별로 변경 여부가 다르다.

  • ① 전체 길이(Total Length): 조각마다 크기가 다르므로 변경된다.
  • ② 체크섬: 헤더 내용이 바뀌었으므로 재계산(변경)된다.
  • ③ 식별자(Identification): 같은 원본 패킷에서 나온 조각임을 나타내기 위해 모든 조각이 동일한 값을 유지한다 → 변경되지 않음.
  • ④ 단편화 오프셋: 조각의 위치를 나타내므로 조각마다 다르게 설정(변경)된다.

수신 측은 식별자가 같은 조각들을 모아 오프셋 순서대로 재조립하므로, 식별자는 반드시 불변이어야 한다. 따라서 정답은 ③번이다.

관련 이론 정리

멀티캐스트 라우팅은 수신자 분포에 따라 접근이 다르다. 밀집 모드(dense mode)는 수신자가 많다고 가정해 먼저 플러딩한 뒤 불필요한 가지를 잘라내는(prune) 방식으로 DVMRP·PIM-DM이 해당하며 소스별 트리를 만들어 최적 경로를 제공하지만 라우터의 상태 정보(소스×그룹)가 많아진다. 희소 모드(sparse mode)는 수신자가 드물다고 보고 명시적 가입(join)이 있을 때만 트리에 합류시키며, CBT·PIM-SM이 해당한다. 공유 트리는 라우터 상태가 그룹당 하나로 줄어 확장성이 좋지만, 코어/RP를 경유해 경로가 최적이 아닐 수 있고 코어가 병목·단일 장애점이 될 수 있다. 영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

SCTP의 연결(association) 설정은 4단계 핸드셰이크로 이뤄진다.

  1. 클라이언트 → 서버: INIT
  2. 서버 → 클라이언트: INIT ACK (상태를 저장하지 않고 쿠키를 담아 응답)
  3. 클라이언트 → 서버: COOKIE ECHO (받은 쿠키를 그대로 돌려보냄)
  4. 서버 → 클라이언트: COOKIE ACK (쿠키 검증 후 연결 성립)

따라서 괄호에 들어갈 메시지는 COOKIE ECHO이고 정답은 ①번이다. ②③④는 SCTP 표준에 없는 명칭이다.

오답 분석

보기①

정답. INIT ACK로 받은 쿠키를 되돌려 보내는 메시지가 COOKIE ECHO다.

보기②

COOKIE INIT은 SCTP 표준 메시지가 아니다.

보기③

COOKIE REQ도 존재하지 않는 명칭이다.

보기④

COOKIE SYN은 TCP의 SYN과 혼동한 잘못된 명칭이다.

관련 이론 정리

SCTP는 TCP·UDP의 장점을 결합한 전송 프로토콜로, 신뢰성 있는 전달을 제공하면서 메시지 지향이며 여러 특징을 갖는다. 멀티스트리밍은 하나의 연결 안에 여러 스트림을 두어 한 스트림의 손실이 다른 스트림을 막지 않게 해(HOL 블로킹 완화) TCP의 단점을 보완한다. 멀티호밍은 하나의 연결에 여러 IP 주소를 연결해 경로 장애 시 자동 전환하는 이중화를 제공한다. 연결 설정의 4-way handshake와 쿠키 메커니즘은 서버가 3단계까지 상태를 저장하지 않게 해 SYN 플러딩(DoS) 공격에 강한 구조를 만든다(TCP의 SYN 쿠키와 유사한 개념을 프로토콜에 내재화). 주로 통신망 시그널링(SIGTRAN), WebRTC 데이터 채널에 쓰인다. 감리에서는 통신 프로토콜 선택의 타당성과 이중화 설계를 점검한다.

SCTP 연결 설정:

순서 메시지 방향
1 INIT 클라이언트 → 서버
2 INIT ACK(쿠키) 서버 → 클라이언트
3 COOKIE ECHO 클라이언트 → 서버
4 COOKIE ACK 서버 → 클라이언트
실무 적용 사례

통신·시그널링 시스템 감리 시 SCTP 멀티호밍 구성으로 회선 이중화가 실제 동작하는지, DoS 방어 설정이 적정한지 점검한다.

감리사 시험 출제 포인트

SCTP는 4-way이며 순서(INIT→INIT ACK→COOKIE ECHO→COOKIE ACK)를 암기한다. TCP의 3-way와 구분한다.

관련 이론 정리

NAT는 IPv4 주소 부족 완화와 내부망 은닉을 위해 사설-공인 주소를 변환한다. Static NAT는 1:1 고정 매핑으로 외부에서 접근해야 하는 서버에 쓰이고, Dynamic NAT는 공인 IP 풀에서 필요할 때 동적으로 할당하되 동시 사용 수가 풀 크기로 제한된다. PAT/NAPT는 포트 번호를 조합해 하나의 공인 IP로 수많은 내부 호스트를 다중화하며 가장 널리 쓰인다. NAT는 내부 구조를 숨겨 보안에 일부 기여하지만 방화벽을 대체하지는 않으며, 종단 간 연결성을 깨뜨려 P2P·FTP·SIP 등에서 ALG나 홀펀칭·STUN이 필요하다. 또 로그에서 내부 호스트 추적이 어려워질 수 있어 매핑 로그 보존이 중요하다. 감리에서는 NAT 정책·로깅·포트포워딩 보안을 점검한다.

정답 해설

시그모이드 함수(σ(x)=1/(1+e⁻ˣ))는 전 구간에서 미분 가능해 오류 역전파에 사용할 수 있지만, 도함수의 최댓값이 0.25에 불과하고 입력이 크거나 작으면 도함수가 0에 가까워진다. 층을 깊게 쌓으면 이 작은 값들이 연쇄적으로 곱해져 앞쪽 층의 기울기가 소멸하는 기울기 소실(vanishing gradient) 문제가 발생한다. 문제의 두 조건(미분 가능 + 기울기 소실)을 모두 만족하므로 정답은 ③번이다.

①계단 함수는 불연속이라 미분 불가(역전파 사용 불가), ②ReLU는 양수 구간 미분값이 1로 기울기 소실을 완화하는 함수, ④선형 함수는 층을 쌓아도 선형 변환에 그쳐 비선형 표현이 불가능하다.

관련 이론 정리

역전파는 출력층 오차를 입력층 방향으로 전파하며 각 층의 기울기를 곱해 가중치를 갱신한다. 이때 활성함수의 도함수가 1보다 훨씬 작으면 곱이 지수적으로 작아져 기울기 소실이, 1보다 크면 기울기 폭발이 발생한다. 시그모이드(최대 0.25)와 tanh(최대 1이지만 포화 구간 존재)는 소실에 취약해 깊은 망의 학습을 어렵게 했고, 이를 해결한 ReLU(양수 구간 도함수 1)의 등장이 딥러닝 부흥의 계기가 됐다. ReLU도 음수 입력에서 뉴런이 죽는 문제(dying ReLU)가 있어 Leaky ReLU·ELU·GELU 등이 제안됐다. 그 밖에 배치 정규화·잔차 연결(ResNet)·적절한 가중치 초기화(He·Xavier)가 기울기 문제를 완화한다. 시그모이드는 지금도 이진 분류 출력층에 쓰인다. 감리에서는 모델 구조·학습 안정성 검증을 점검한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

DHCP는 DORA 4단계로 동작한다.

  1. Discover: 클라이언트가 DHCP 서버를 찾기 위해 브로드캐스트로 보내는 메시지
  2. Offer: 서버가 할당 가능한 IP 주소를 제안
  3. Request: 클라이언트가 특정 제안을 수락해 해당 IP를 요청
  4. ACK: 서버가 최종 확정·승인

③은 "DHCP Discover는 DHCP 서버가 IP 주소 할당 서버임을 알린다"고 서술했으나, Discover는 클라이언트가 서버를 찾기 위해 보내는 메시지이므로 주체가 잘못됐다. 따라서 가장 적절하지 않은 것은 ③번이다. ①②④는 모두 옳다.

오답 분석

보기①

옳음. ACK는 Request에 대한 서버의 최종 승인 응답이다.

보기②

옳음. Request는 클라이언트가 제안받은 IP를 요청·수락하는 메시지다.

보기③

정답(부적절). Discover는 서버가 아니라 클라이언트가 보내는 탐색 메시지다.

보기④

옳음. Offer는 서버가 할당 가능한 IP를 클라이언트에 제안하는 메시지다.

관련 이론 정리

DHCP는 IP 주소·서브넷 마스크·게이트웨이·DNS 등 네트워크 설정을 자동 할당해 관리 부담을 줄인다. 클라이언트는 부팅 시 IP가 없으므로 Discover를 브로드캐스트(출발지 0.0.0.0, 목적지 255.255.255.255)하고, 서버는 Offer로 응답한다. 여러 서버가 응답할 수 있어 클라이언트는 하나를 선택해 Request를 보내고 ACK로 확정된다. 주소는 임대(lease) 기간 동안 유효하며 만료 전 갱신(renew)한다. 브로드캐스트는 라우터를 넘지 못하므로 다른 서브넷의 서버를 쓰려면 DHCP 릴레이 에이전트가 필요하다. 보안상 위조 서버가 잘못된 게이트웨이를 배포하는 DHCP 스푸핑이 가능해 DHCP 스누핑으로 방어한다. 감리에서는 DHCP 이중화·주소 풀 관리·스누핑 설정을 점검한다.

DHCP DORA 절차:

순서 메시지 주체
1 Discover 클라이언트(브로드캐스트)
2 Offer 서버
3 Request 클라이언트
4 ACK 서버
실무 적용 사례

네트워크 운영 감리 시 DHCP 서버 이중화, 주소 풀 고갈 모니터링, DHCP 스누핑·릴레이 설정의 적정성을 점검한다.

감리사 시험 출제 포인트

DORA 각 메시지의 주체(클라이언트/서버)를 정확히 구분한다. Discover·Request는 클라이언트, Offer·ACK는 서버다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

RAID 10은 두 개씩 미러(RAID 1) 쌍을 만들고 그 쌍들을 스트라이핑(RAID 0)한 구조다. 디스크 8개면 미러 쌍 4개가 된다.

  • 읽기: 미러본도 읽기에 활용할 수 있으므로 8개 디스크 모두 동시에 읽기를 처리한다.
    → 8 × 100,000 = 800,000 IOPS
  • 쓰기: 하나의 데이터를 미러 쌍의 두 디스크에 모두 기록해야 하므로 유효 처리량은 절반이 된다(쓰기 패널티 2).
    → (8 × 100,000) ÷ 2 = 400,000 IOPS

따라서 읽기 800,000 / 쓰기 400,000 IOPS이고 정답은 ③번이다.

오답 분석

보기①

읽기도 400,000으로 계산했는데, 읽기는 미러본까지 활용해 800,000이다.

보기②

읽기·쓰기 값이 서로 뒤바뀐 형태로, 쓰기가 읽기보다 클 수 없다.

보기③

정답. 읽기 800,000(8×100k), 쓰기 400,000(8×100k÷2).

보기④

쓰기에 미러 중복 기록(패널티 2)을 반영하지 않은 값이다.

관련 이론 정리

RAID 레벨별 쓰기 패널티(하나의 논리적 쓰기에 필요한 물리 I/O 수)는 성능 산정의 핵심이다. RAID 0은 패널티 1로 읽기·쓰기 모두 N배. RAID 1/10은 미러 기록 때문에 쓰기 패널티 2(읽기는 N배, 쓰기는 N/2배). RAID 5는 데이터·패리티를 각각 읽고 쓰므로 패널티 4, RAID 6은 이중 패리티로 패널티 6이다. 따라서 쓰기가 많은 OLTP·DB 로그 볼륨에는 RAID 10이, 읽기 위주·용량 효율이 중요한 아카이브에는 RAID 5/6이 적합하다. 유효 IOPS = (디스크 수 × 디스크 IOPS) ÷ 쓰기 패널티(쓰기 비율 반영)로 산정하며, 실제로는 캐시·컨트롤러 성능도 영향을 준다. 감리에서는 워크로드의 읽기/쓰기 비율에 맞는 RAID 레벨 선정과 IOPS 산정 근거를 점검한다.

RAID 쓰기 패널티:

레벨 읽기 쓰기 패널티
RAID 0 N배 1
RAID 1/10 N배 2
RAID 5 N배 4
RAID 6 N배 6
실무 적용 사례

DB 스토리지 설계 감리 시 쓰기 비중이 높은 트랜잭션 시스템에 RAID 5 대신 RAID 10을 적용했는지, IOPS 산정에 쓰기 패널티를 반영했는지 검토한다.

감리사 시험 출제 포인트

RAID 10은 읽기 N배, 쓰기 N/2배를 공식처럼 기억한다. 쓰기 패널티(1/2/4/6)를 레벨별로 정리하면 계산형 문항에 대응된다.
