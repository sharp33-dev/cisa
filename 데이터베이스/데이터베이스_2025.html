영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

후보키는 릴레이션의 모든 속성을 함수적으로 결정하면서(유일성) 더 이상 줄일 수 없는(최소성) 속성 집합이다. 후보키를 찾는 핵심 원리는 "어떤 함수적 종속성의 오른쪽(결정 대상)에도 나타나지 않는 속성은 반드시 모든 후보키에 포함되어야 한다"는 것이다.

주어진 FD 집합 A → B, C → A, C → D, (A, F) → E 에서 각 속성이 오른쪽(RHS)에 등장하는지 보면, B·A·D·E는 RHS에 나타나지만 C와 F는 어떤 FD의 오른쪽에도 나타나지 않는다. 따라서 C와 F는 반드시 후보키의 구성요소가 되어야 한다.

이제 {C, F}의 폐포(closure) {C, F}+ 를 계산한다. C → A 이므로 A 추가, C → D 이므로 D 추가, A → B 이므로 B 추가, (A, F) → E 이므로(A와 F를 모두 가지고 있음) E 추가. 결과적으로 {C, F}+ = {A, B, C, D, E, F} 로 릴레이션 R의 전체 속성을 결정한다. 또한 C 단독, F 단독으로는 전체를 결정할 수 없으므로 최소성도 만족한다. 그러므로 후보키는 CF이며 정답은 ②번이다.

오답 분석

보기①

AF의 폐포 {A, F}+를 계산하면 A → B로 B, (A, F) → E로 E를 얻어 {A, B, E, F}에 그친다. C와 D를 결정하지 못하므로 후보키가 될 수 없다.

보기②

정답. {C, F}+가 전체 속성을 결정하고 최소성을 만족한다.

보기③

AC의 폐포 {A, C}+는 C → A, A → B, C → D로 {A, B, C, D}까지만 도달한다. E와 F를 결정하지 못한다(특히 F는 어떤 FD로도 유도 불가). 후보키가 아니다.

보기④

AFC는 C와 F를 포함해 전체를 결정하지만, 이미 CF만으로 전체가 결정되므로 A는 불필요(잉여)하다. 최소성(minimality)에 위배되어 후보키가 아니라 상위 개념인 슈퍼키(super key)에 해당한다.

관련 이론 정리

키는 릴레이션의 튜플을 식별하는 속성(집합)으로, 포함 관계에 따라 여러 종류로 나뉜다. 슈퍼키는 튜플을 유일하게 식별하는 모든 속성 집합(유일성만 요구), 후보키는 그중 더 줄일 수 없는 최소 슈퍼키(유일성+최소성), 기본키는 후보키 중 대표로 선정한 것, 대체키는 선정되지 않은 나머지 후보키다. 후보키를 찾는 핵심 도구가 폐포(closure) 계산인데, 어떤 함수 종속의 오른쪽(결정 대상)에도 나타나지 않는 속성은 다른 무엇으로도 유도될 수 없으므로 반드시 모든 후보키에 포함되어야 한다. 이 필수 속성들의 폐포를 계산해 전체 속성을 결정하면 후보키이고, 부족하면 결정자에 속성을 더해 나간다. 마지막으로 진부분집합이 전체를 결정하지 않는지 확인해 최소성을 검증한다. 데이터 모델링 감리에서는 이렇게 폐포 기반으로 식별자(기본키)의 유일성과 최소성을 점검한다.

후보키 결정 절차(폐포 이용):

단계 내용
1단계 어떤 FD의 RHS에도 없는 속성 식별 → 필수 포함 속성
2단계 해당 속성 집합의 폐포 계산
3단계 전체 속성 결정 시 후보키, 부족 시 다른 속성을 추가
4단계 최소성 검증(부분집합이 전체를 결정하면 후보키 아님)
  • 슈퍼키: 유일성만 만족(최소성 불필요)
  • 후보키: 유일성 + 최소성
  • 기본키: 후보키 중 대표로 선정한 키
실무 적용 사례

데이터 모델링 감리 시 논리 ERD의 각 엔터티에 대해 식별자(기본키)가 함수적 종속성 관점에서 실제로 모든 비식별 속성을 유일하게 결정하는지, 불필요한 속성이 키에 포함되어 최소성을 해치지 않는지를 폐포 기반으로 점검한다.

감리사 시험 출제 포인트

후보키 계산은 "RHS에 없는 속성은 반드시 키에 포함"이라는 규칙과 폐포 계산을 결합하면 빠르게 해결된다. 슈퍼키(④ AFC)를 후보키로 오인하도록 유도하는 최소성 함정이 단골로 출제된다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

분해가 원래 릴레이션을 복구하려면 두 가지가 모두 필요하다. 첫째 속성 보존(R1 ∪ R2 = R, 즉 원래의 모든 속성이 분해 결과에 남아 있어야 함), 둘째 무손실조인 조건(공통 속성 R1 ∩ R2가 R1 또는 R2 전체를 함수적으로 결정해야 함)이다. 하나라도 위배되면 조인 시 원본에 없던 스퓨리어스 투플이 생겨 복구가 불가능하다.

③번: R(A, B, C, D)를 R1(A, C), R2(A, B)로 분해하면 R1 ∪ R2 = {A, B, C} 로 속성 D가 완전히 사라진다. 속성 보존이 깨졌으므로 어떤 방법으로도 D를 복구할 수 없다. 따라서 복구 불가.

④번: R1(A, B), R2(A, C, D)는 속성은 모두 보존({A,B,C,D})되지만, 공통 속성이 {A}이다. A → R1(A, B)이려면 A → B가 필요한데 없고, A → R2(A, C, D)이려면 A → C가 필요한데 주어진 FD (A,B)→C, C→D, C→A로는 A 단독이 C를 결정하지 못한다(C→A는 역방향). 무손실조인 조건 미충족으로 복구 불가.

두 보기 모두 원본 복구가 불가능하므로 정답은 ③, ④ 복수정답이다.

오답 분석

보기①

R1(A, B, C, D)가 이미 원래 릴레이션 전체이므로 R2(C, D)와 조인해도 원본이 그대로 복구된다. 무손실 분해이다.

보기②

R1(A, B, C), R2(C, D)는 속성이 모두 보존되고 공통 속성 {C}가 C → D로 R2(C, D)를 결정한다. 무손실조인 조건을 만족하여 복구 가능하다.

보기③

정답. 속성 D가 분해 결과에서 누락되어 속성 보존 위배 → 복구 불가.

보기④

정답. 속성은 보존되나 공통 속성 {A}가 R1·R2 어느 쪽도 결정하지 못해 무손실조인 조건 위배 → 복구 불가.

관련 이론 정리

정규화 과정에서 하나의 릴레이션 R을 둘 이상으로 분해할 때는 무손실 조인(lossless-join)이 보장돼야 원래 데이터를 정확히 복원할 수 있다. 무손실이 성립하려면 두 조건이 필요하다. 첫째 속성 보존으로, 분해된 릴레이션들의 속성 합집합이 원래 R의 모든 속성을 포함해야 한다(하나라도 빠지면 그 속성은 영영 복원 불가). 둘째 무손실 조인 조건으로, 두 릴레이션의 공통 속성(교집합)이 둘 중 적어도 한쪽의 슈퍼키가 되어야 한다(즉 공통속성 → R1 또는 공통속성 → R2). 이 조건이 깨지면 다시 조인했을 때 원본에 없던 가짜 튜플, 곧 스퓨리어스 튜플(spurious tuple)이 생겨 정보가 왜곡된다. 함께 고려하는 성질로 원래의 함수 종속을 분해 후에도 유지하는 종속성 보존이 있으나, 무손실과 종속성 보존을 동시에 항상 만족시키지 못할 수도 있다. 감리에서는 테이블 분할 설계에서 조인 키가 한쪽 테이블의 키가 되는지 확인한다.

무손실(lossless-join) 분해 판정:

조건 내용
속성 보존 R1 ∪ R2 = R (모든 속성 유지)
무손실조인 (R1 ∩ R2) → R1 또는 (R1 ∩ R2) → R2 중 하나 성립
위배 결과 스퓨리어스(가짜) 투플 발생 → 정보 손실
  • 두 릴레이션 분해에서는 공통 속성이 최소 한쪽의 슈퍼키가 되면 무손실이다.
실무 적용 사례

정규화 감리 시 테이블 분할 설계가 무손실 분해인지 반드시 확인한다. 공통 컬럼(조인 키)이 어느 한 테이블의 후보키가 되는지 검토하고, 아니면 조인 시 데이터가 부풀려지는(중복·가짜 행) 위험을 지적한다.

정답 해설

완전 외부 조인은 공통 속성(B, C)으로 두 릴레이션을 조인하되, 매칭되는 투플은 결합하고, 매칭되지 않는 좌측·우측 투플은 상대편 속성을 NULL('–')로 채워 결과에 모두 포함시킨다. 여기서 핵심은 "한 번이라도 매칭에 성공한 투플은 NULL 보충 형태로는 나타나지 않는다"는 점이다.

S의 투플 (b1, c2, d2)는 R의 (a2, b1, c2), (a3, b1, c2)와 (B, C)=(b1, c2)로 매칭에 성공한다. 따라서 (a2, b1, c2, d2), (a3, b1, c2, d2)는 생성되지만, 이미 매칭되었으므로 좌측이 NULL인 (–, b1, c2, d2)는 생성되지 않는다. 보기 ②가 바로 이 형태이므로, 완전 외부 조인 결과로 나올 수 없는 투플은 ②번이다.

관련 이론 정리

조인은 공통 속성(조인 조건)을 기준으로 두 릴레이션을 결합하는데, 매칭되지 않는 튜플의 처리 방식에 따라 내부 조인과 외부 조인으로 나뉜다. 내부 조인(inner join)은 양쪽에서 매칭되는 튜플만 결합해 미매칭 튜플은 결과에서 사라진다. 외부 조인(outer join)은 미매칭 튜플을 보존하되 상대편 속성을 NULL로 채운다. 어느 쪽을 보존하느냐에 따라 좌측을 모두 보존하는 좌측 외부 조인, 우측을 보존하는 우측 외부 조인, 양쪽을 모두 보존하는 완전 외부 조인으로 구분된다. 핵심 원리는 "한 번이라도 매칭에 성공한 튜플은 NULL 보충 형태로 별도로 다시 나타나지 않는다"는 점이다. 즉 이미 실제 값과 결합된 튜플에 대해 좌측 또는 우측이 NULL인 가짜 조합은 생성되지 않는다. 데이터 대사·연계 감리에서는 미매칭(누락/과잉) 데이터를 보존해야 할 때 외부 조인을, 정합 데이터만 필요할 때 내부 조인을 쓰는지 확인한다.

외부 조인(Outer Join) 종류:

종류 결과
Left Outer 좌측 전체 + 매칭된 우측(미매칭 우측속성 NULL)
Right Outer 우측 전체 + 매칭된 좌측(미매칭 좌측속성 NULL)
Full Outer 좌·우측 전체(양쪽 미매칭 투플 모두 NULL 보충 포함)
  • 매칭된 투플은 실제 값으로 결합되며 NULL 보충본이 별도로 생기지 않는다.
실무 적용 사례

두 시스템 데이터 정합성 감리 시 완전 외부 조인으로 어느 한쪽에만 존재하는 데이터(누락/과잉)를 식별한다. NULL로 채워진 행이 곧 불일치 후보이며, 매칭된 행은 정상 연계 데이터로 판정한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

1단계 - 후보키(기본키) 결정: 어떤 FD의 오른쪽에도 나타나지 않는 속성은 EmpID와 ProjID이므로 후보키는 이 둘을 반드시 포함한다. {EmpID, ProjID}+를 계산하면 EmpID → EmpName(EmpName), (EmpID, ProjID) → DeptID(DeptID), DeptID → DeptName(DeptName)로 전체 속성 {EmpID, DeptID, ProjID, EmpName, DeptName}에 도달한다. 따라서 기본키 = (EmpID, ProjID), 기본키 구성 속성(prime attribute)은 EmpID, ProjID이다.

2단계 - 2NF 위반(부분 함수 종속) 탐지: 2NF는 "모든 비주요 속성이 기본키에 완전 함수 종속"일 것을 요구한다. 즉 기본키의 일부만으로 결정되는 부분 종속이 있으면 위반이다.
- EmpID → EmpName : 기본키의 일부인 EmpID만으로 EmpName이 결정됨 → 부분 함수 종속(2NF 위반)
- (EmpID, ProjID) → DeptID : 기본키 전체가 필요 → 완전 종속(정상)
- DeptID → DeptName : DeptID는 비주요 속성 → 이는 이행 종속(3NF 대상)이며 2NF에서는 다루지 않음

3단계 - 분해: 부분 종속 EmpID → EmpName을 별도 릴레이션으로 분리한다.
- R1(EmpID, EmpName)
- R2(EmpID, ProjID, DeptID, DeptName)

R1은 EmpID가 키, R2는 (EmpID, ProjID)가 키로 모두 부분 종속이 없어 2NF를 만족한다. (R2 내 DeptID → DeptName의 이행 종속은 3NF 단계에서 추가 분해 대상이나, 문제는 2NF까지만 요구.) 따라서 생성되는 릴레이션은 총 2개로 정답은 ②번이다.

관련 이론 정리

정규화는 함수 종속을 근거로 릴레이션을 단계적으로 분해해 삽입·삭제·갱신 이상(anomaly)을 제거하는 과정이며, 각 정규형은 특정 유형의 종속을 없앤다. 1NF는 모든 속성이 원자값이어야 하고, 2NF는 1NF를 만족하면서 기본키의 일부만으로 결정되는 부분 함수 종속을 제거한다(따라서 부분 종속은 복합키가 있을 때만 생기고, 단일 속성 키면 자동으로 2NF). 3NF는 비주요 속성이 기본키가 아닌 다른 속성을 거쳐 결정되는 이행 함수 종속을 제거하고, BCNF는 모든 함수 종속의 결정자가 후보키가 되도록 한다. 여기서 중요한 것은 문제에서 요구하는 정규형 단계까지만 분해해야 한다는 점으로, 2NF까지 요구했는데 이행 종속(3NF 대상)까지 미리 분해하면 과분해가 된다. 감리에서는 복합 식별자 엔터티의 부분 종속과 비식별 속성의 이행 종속을 근거로 정규화 수준의 적정성을 판단한다.

정규형 단계별 제거 대상:

정규형 조건 / 제거 대상
1NF 모든 속성이 원자값(atomic)
2NF 1NF + 부분 함수 종속 제거(비주요 속성이 기본키에 완전 종속)
3NF 2NF + 이행 함수 종속 제거
BCNF 3NF + 모든 결정자가 후보키
  • 부분 종속은 복합키(composite key)가 있을 때만 발생한다. 단일 속성 키면 자동으로 2NF.
  • 본 문항의 DeptID → DeptName은 이행 종속으로 3NF 대상 → 2NF 계산에서 제외해야 함정을 피한다.
실무 적용 사례

데이터 모델링 감리에서 복합 식별자를 가진 엔터티는 부분 종속 여부를 필수 점검한다. 예컨대 (사원, 프로젝트) 복합키 테이블에 사원명·부서명이 섞여 있으면 갱신 이상(anomaly)이 발생하므로, 사원 마스터를 분리하도록 권고한다.

감리사 시험 출제 포인트

"2NF까지"인지 "3NF까지"인지에 따라 릴레이션 개수가 달라진다(2 vs 3). 이행 종속(DeptID → DeptName)을 2NF 단계에서 미리 분해해 3개로 오답하도록 유도하는 것이 핵심 함정이다. 요구된 정규형 단계를 정확히 읽어야 한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

1:1 관계를 관계 모델로 사상하는 정상적인 방법은 다음 세 가지다. (1) 한쪽 엔터티에 상대 엔터티의 기본키를 외래키로 포함(①, ②), (2) 두 엔터티를 하나의 릴레이션으로 병합(④). 이 세 가지는 모두 두 엔터티 사이의 1:1 관계를 조인 또는 동일 행으로 표현할 수 있다.

그러나 ③번은 E1(PK1, A1, K1)과 E2(PK2, A2, K2)를 서로를 참조하는 외래키 없이 완전히 독립된 두 릴레이션으로 만든다. 이 경우 어떤 E1 인스턴스가 어떤 E2 인스턴스와 1:1로 연결되는지 표현할 방법이 사라져 관계 R 자체가 소실된다. 관계를 복원하려면 조인할 공통 키가 있어야 하는데 그것이 없다. 따라서 ER 스키마를 관계 모델로 사상하는 방법으로 가장 적절하지 않은 것은 ③번이다.

오답 분석

보기①

E2에 E1의 기본키를 외래키 FK1으로 넣어 1:1 관계를 표현한다. 표준적인 사상 방법으로 적절하다.

보기②

E1에 E2의 기본키를 외래키 FK2로 넣는 방식이다. 1:1에서는 어느 쪽에 외래키를 두어도 되므로 적절하다.

보기③

정답(부적절). 두 릴레이션에 상호 참조 외래키가 없어 1:1 관계 R을 표현·복원할 수 없다.

보기④

1:1 관계, 특히 양쪽 참여가 필수(전체 참여)인 경우 두 엔터티를 한 릴레이션으로 병합하는 것은 널(NULL)을 줄이는 효과적인 사상이다. 적절하다.

관련 이론 정리

ER 다이어그램을 관계 모델(테이블)로 변환할 때는 관계의 카디널리티(1:1, 1:N, N:M)에 따라 외래키 배치나 별도 릴레이션 생성 방식이 달라진다. 1:1 관계는 한쪽 엔터티에 상대의 기본키를 외래키로 넣거나, 참여가 양쪽 다 필수면 두 엔터티를 하나로 병합할 수 있다. 1:N 관계는 반드시 N측 엔터티에 1측의 기본키를 외래키로 둔다(1측에 N개를 담을 수 없기 때문). N:M 관계는 어느 쪽에도 외래키를 둘 수 없으므로 양측 기본키를 조합한 별도의 교차(관계) 릴레이션을 신설하고, 관계 자체의 속성도 여기에 둔다. 공통 원칙은 어떤 사상 방식이든 두 엔터티를 이어주는 외래키(또는 병합)가 반드시 존재해야 관계 정보가 보존된다는 것이다. 논리→물리 모델 전환 감리에서는 카디널리티별 사상 규칙 준수와 참조 무결성 설정을 점검한다.

관계 카디널리티별 외래키 사상 규칙:

관계 유형 사상 방법
1:1 한쪽에 상대 PK를 FK로 두거나, 두 엔터티를 병합
1:N N측에 1측의 PK를 FK로 둔다
N:M 별도의 관계 릴레이션(교차 테이블) 생성, 양측 PK를 FK로
  • 1:1은 유일하게 "병합" 옵션이 성립한다. 단, 어떤 경우든 관계를 잇는 외래키(또는 병합)가 반드시 있어야 한다.
실무 적용 사례

논리 모델을 물리 테이블로 전환하는 감리 시, 1:1 관계 테이블에 조인 키(외래키)가 누락되면 두 테이블을 연결할 수 없어 관계 정보가 유실된다. 감리원은 FK 제약 또는 병합 설계 여부를 확인한다.

감리사 시험 출제 포인트

"외래키가 없는 독립 릴레이션"은 관계를 표현하지 못한다는 원리를 묻는다. 1:1에서는 외래키 위치가 자유롭고 병합도 가능하다는 점을 이용해, 정상 사상 3개 사이에 "관계 소실" 보기(③)를 함정으로 배치한다.

정답 해설

릴레이션의 각 투플은 기본키 값으로 유일하게 식별된다. 즉 서로 다른 투플은 반드시 서로 다른 기본키 값을 가져야 하므로, 가능한 투플의 최대 개수는 기본키를 구성하는 속성 도메인 크기의 곱을 넘을 수 없다.

여기서 기본키는 (StudentID, CourseID, Semester)이므로 최대 조합 수는:

1000(StudentID) × 50(CourseID) × 8(Semester) = 400,000

Grade는 기본키에 포함되지 않는 비주요 속성이므로, 각 기본키 조합에 대해 하나의 Grade 값이 정해질 뿐 투플 수를 늘리지 않는다(Grade 도메인 크기 5는 곱하지 않는다). 따라서 최대 투플 개수는 400,000개이고 정답은 ②번이다.

정답 해설

세 보기 ①·②·④는 모두 주문이 없는 고객까지 결과에 포함시킨다. ①은 LEFT JOIN 후 GROUP BY로 주문 없는 고객의 total_sales가 NULL로 표시되고, ②는 상관 서브쿼리의 SUM이 NULL, ④는 미리 집계한 서브쿼리에 LEFT JOIN하여 s가 NULL로 나타난다. 세 쿼리의 결과 집합(행 수·값)은 동일하다.

관련 이론 정리

SQL의 NULL은 '값이 없음/알 수 없음'을 뜻하며, 일반 값과 달리 3치 논리(참·거짓·unknown)로 평가된다. NULL과의 모든 비교(= NULL, <> NULL, NULL >= 0 등)는 참도 거짓도 아닌 unknown을 돌려주고, WHERE·HAVING·ON 같은 조건절은 결과가 참일 때만 행을 통과시키므로 unknown인 행은 걸러진다. 또 집계에서 SUM·AVG·COUNT(열)는 NULL을 무시하지만 COUNT(*)는 행 자체를 세고, 대상이 모두 NULL이거나 빈 그룹이면 SUM 결과가 NULL이 된다. 이 때문에 외부 조인으로 보존한 행이 이후 HAVING 집계 >= 0 같은 NULL 비교 조건에서 탈락하거나, NOT IN 목록에 NULL이 섞여 결과가 통째로 비는 함정이 생긴다. 실무에서는 IS NULL/IS NOT NULL, COALESCE로 명시적으로 처리한다. 감리에서는 NULL 처리 누락으로 통계·조건 결과가 왜곡되지 않는지 확인한다.

NULL과 조건 평가(3치 논리):

식 결과
NULL >= 0 unknown
NULL = NULL unknown
WHERE/HAVING의 unknown 행 제외(참이 아님)
SUM(빈 집합/전부 NULL) NULL
  • LEFT JOIN은 매칭 없는 좌측 행을 보존하지만, 이후 HAVING/WHERE에서 NULL 조건을 걸면 그 행이 사라질 수 있다.
실무 적용 사례

"모든 고객의 매출(주문 없으면 0)" 리포트 감리 시, HAVING/WHERE에 집계 NULL 조건이 걸려 주문 없는 고객이 누락되는 버그를 자주 발견한다. COALESCE(SUM(saleprice),0) 처리 여부를 점검한다.

정답 해설

이 질의는 관계 나눗셈(division)의 전형적 SQL 표현이다. 안쪽 EXCEPT는 (1번 부서가 수행하는 프로젝트 집합) - (해당 직원이 참가한 프로젝트 집합)을 구한다. 이 차집합이 공집합이라는 것은 "1번 부서의 프로젝트 중 그 직원이 참가하지 않은 것이 하나도 없다", 즉 그 직원이 1번 부서의 모든 프로젝트에 참가했다는 의미다. 바깥 NOT EXISTS가 이 차집합이 공집합인 직원만 선택하므로, 결과는 "1번 부서가 수행하는 모든 프로젝트에 참가한 직원"이다.

따라서 ②는 정확한 설명이고, ③(안쪽 상관질의가 해당 직원이 참가한 프로젝트 선택), ④(동일 결과를 나눗셈 관계대수로 표현)도 옳다. 반면 ①은 "참가하지 않은 직원"이라고 정반대로 서술하여 옳지 않다. 그러므로 정답(옳지 않은 것)은 ①번이다.

오답 분석

보기①

정답(옳지 않음). 실제 질의는 "모든 프로젝트에 참가한" 직원을 찾는데, ①은 "참가하지 않은"으로 의미를 뒤집었다.

보기②

옳음. EXCEPT 결과가 공집합 = 1번 부서 전 프로젝트에 참가 → 정확한 해석.

보기③

옳음. WHERE E.ENO=ENO로 바깥 직원 값을 참조하는 상관 질의이며, 그 직원이 참가한 프로젝트(W의 PNO)를 선택한다.

보기④

옳음. Π_ENO,PNO(W) ÷ Π_PNO(σ_DNO=1(P))는 "1번 부서 프로젝트 전체 집합을 포함하는 직원"을 구하는 나눗셈으로 동일 결과를 낸다.

관련 이론 정리

관계 나눗셈(division)은 "S의 모든 원소와 짝을 이루는" 튜플을 찾는 for-all(전칭) 질의를 표현하는 연산이다. 예컨대 R(학생, 과목) ÷ S(과목)은 S의 모든 과목을 수강한 학생을 돌려준다. SQL에는 나눗셈 연산자가 없어 관용적으로 이중 NOT EXISTS나 NOT EXISTS + EXCEPT로 구현한다. 핵심 논리는 "그 학생이 수강하지 않은 필수 과목이 존재하지 않는다"는 이중 부정으로, (필수 집합 − 보유 집합)이 공집합이면 모두 보유한 것이다. 관계대수로는 R ÷ S = π(R) − π((π(R)×S) − R) 형태로 프로젝션·카티션 곱·차집합을 사용해 대체 표현된다. GROUP BY와 HAVING COUNT를 이용해 "보유한 필수 과목 수 = 전체 필수 과목 수"로도 표현할 수 있다. 감리에서는 "모든 ~을 만족" 요구가 합집합(OR)이 아니라 나눗셈/교집합으로 올바로 구현됐는지 확인한다.

관계 나눗셈(division)의 SQL 관용 표현:

방식 표현
이중 NOT EXISTS "참가하지 않은 프로젝트가 존재하지 않는" 직원
NOT EXISTS + EXCEPT (요구집합 − 보유집합) = 공집합
관계대수 R ÷ S = Π(R) − Π((Π(R)×S) − R)
  • 나눗셈은 "S의 모든 원소와 짝을 이루는" 튜플을 찾는 for-all 질의에 대응한다.
실무 적용 사례

"특정 자격증을 모두 보유한 지원자", "전 지점을 모두 이용한 고객" 같은 for-all 조회를 감리할 때 이중 NOT EXISTS/EXCEPT 패턴의 논리 방향(참가/미참가)이 요구사항과 일치하는지 검증한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

전체 선수 수 = 10(L) + 40(H) + 30(K) + 20(미정) = 100명.

가. SELECT 소속팀 FROM 선수 : 중복 제거 없이 전체 투플을 그대로 반환하므로 카디널리티는 100이다(소속팀이 NULL인 20명도 모두 포함).

나. SELECT DISTINCT 소속팀 : 서로 다른 소속팀 값의 개수다. 팀은 L, H, K 세 개이고, NULL(미정)도 DISTINCT에서는 하나의 값으로 취급되어 별도 그룹이 된다. 따라서 L, H, K, NULL = 4이다.

다. GROUP BY 소속팀 HAVING COUNT(*) >= 20 : 그룹별 인원은 L=10, H=40, K=30, NULL(미정)=20이다. GROUP BY도 NULL을 하나의 그룹으로 만든다. COUNT()는 각 그룹의 행 수이므로 HAVING COUNT() >= 20을 만족하는 그룹은 H(40), K(30), 미정(20) = 3개이다(L=10은 탈락).

따라서 (가, 나, 다) = (100, 4, 3)으로 정답은 ③번이다.

오답 분석

보기①·②

정답. 가:100(전체), 나:4(L·H·K·NULL), 다:3(H·K·미정 그룹이 20 이상).

관련 이론 정리

집계·그룹화에서 NULL은 일반 값과 다르게 취급되므로 결과 카디널리티 계산에 주의해야 한다. 단순 SELECT 열은 중복 제거 없이 모든 행을 반환하므로 그 열이 NULL인 행까지 그대로 포함한다. DISTINCT는 서로 다른 값의 개수를 세는데 이때 여러 NULL을 하나의 값으로 취급해 한 번만 센다. GROUP BY도 마찬가지로 NULL을 하나의 그룹으로 묶는다. 집계 함수는 COUNT(*)가 행 수(널 무관)를 세는 반면 COUNT(열)·SUM·AVG는 NULL을 제외한다. 따라서 "학과가 NULL인 학생"이 있으면 DISTINCT 학과나 GROUP BY 학과의 결과 개수에 NULL 그룹이 하나 더해진다. 이 규칙을 놓쳐 NULL 그룹을 빼거나 이중 계상하면 통계가 왜곡된다. 감리에서는 미분류·미배정(NULL) 범주가 집계에서 하나의 그룹으로 정확히 반영되는지 점검한다.

NULL의 집계·그룹화 처리:

구문 NULL 처리
SELECT 컬럼 NULL 행도 그대로 반환
DISTINCT NULL을 하나의 값으로 취급(1개 그룹)
GROUP BY NULL을 하나의 그룹으로 취급
COUNT(*) 행 수(널 무관), COUNT(컬럼)은 NULL 제외
실무 적용 사례

집계 리포트 감리 시 NULL 값(미분류/미배정)이 DISTINCT·GROUP BY에서 하나의 범주로 잡히는지 확인한다. NULL 그룹을 누락하거나 이중 계상하면 통계 수치가 왜곡된다.

감리사 시험 출제 포인트

SELECT 컬럼(중복·NULL 포함 전체) vs DISTINCT(NULL 1개 포함) vs GROUP BY+HAVING(NULL 그룹 포함, 임계값 필터)의 차이를 정확히 구분하는지 묻는다. NULL을 빼먹는 80·2 같은 함정에 유의.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

"모든 배를 예약 받은 항해사"는 예약 릴레이션을 배 릴레이션(의 배 번호 전체)으로 나눗셈하는 전형적 division 질의다. 나눗셈은 관계대수의 기본 연산으로 다음과 같이 대체 표현된다.

R ÷ S = π_c(R) − π_c( (π_c(R) × S) − R )

여기에 등장하는 연산자는 프로젝트(π, 라), 곱집합(×, 나), 차집합(−, 다) 세 가지다. 즉 배 번호 전체 집합 S와 예약 R을 곱집합으로 조합해 "가능한 모든 (항해사, 배) 쌍"을 만들고, 실제 예약에 없는 쌍을 차집합으로 제거한 뒤 다시 차집합으로 "예약하지 못한 배가 있는 항해사"를 제외하는 방식이다. 합집합(∪, 가)은 이 과정에 필요하지 않다. 따라서 필요한 연산자는 나·다·라이고 정답은 ④번이다.

오답 분석

보기①·②·③

모두 합집합(가)을 포함하고 있다. 나눗셈의 대체식에는 합집합이 등장하지 않으므로 합집합을 포함한 조합(①·②·③)은 부적절하다. 특히 ③은 곱집합(나)이 빠져 있어 "모든 조합 생성" 단계를 표현할 수 없다.

보기④

정답. 프로젝트(라)·곱집합(나)·차집합(다)만으로 나눗셈을 구성한다.

관련 이론 정리

관계대수에서 나눗셈은 기본 연산이 아니라 프로젝션·카티션 곱·차집합으로 유도되는 파생 연산이다. R ÷ S를 계산하는 대체식은 다음 단계로 이뤄진다. 먼저 나눗셈 대상 속성만 프로젝션한 π(R)을 구하고, 여기에 S를 카티션 곱하여 "가능한 모든 (대상, S원소) 조합"을 만든다. 그다음 실제 R에 없는 조합을 차집합으로 걸러내면 "적어도 하나의 S원소와 짝을 이루지 못한 대상"이 드러난다. 마지막으로 π(R)에서 이들을 다시 차집합으로 빼면 "S의 모든 원소와 짝을 이루는 대상", 곧 나눗셈 결과가 남는다. 식으로는 R ÷ S = π_c(R) − π_c((π_c(R) × S) − R)이며, 합집합은 필요하지 않다. 이 구조를 이해하면 SQL의 이중 NOT EXISTS 패턴과 자연스럽게 연결된다. 감리에서는 for-all 질의의 구현이 이 논리에 부합하는지 검토한다.

나눗셈 대체식 단계:

단계 연산 의미
1 π_c(R) 나눗셈 대상 속성만 추출
2 π_c(R) × S 가능한 모든 조합 생성(곱집합)
3 (2) − R 실제로 존재하지 않는 조합
4 π_c(3) "빠진 배"가 있는 항해사
5 π_c(R) − (4) 모든 배를 예약한 항해사
실무 적용 사례

"모든 필수 교육을 이수한 직원", "전 품목을 취급하는 거래처" 등 for-all 질의를 관계대수/SQL로 구현할 때 나눗셈 대체식의 곱집합·차집합 구조를 이해하면 정확한 쿼리를 설계·감리할 수 있다.

정답 해설

관계대수의 완전 집합은 셀렉트(σ), 프로젝트(Π), 합집합(∪), 리네임(ρ), 차집합(−), 곱집합(×)의 6개 기본 연산이다. 조인·교집합·나눗셈 등은 이들로 표현 가능한 파생 연산이다.

디비전(나눗셈)의 대체식은 R ÷ S = Π_c(R) − Π_c((Π_c(R) × S) − R)로, 여기에 사용되는 기본 연산은 프로젝트(Π), 곱집합(×), 차집합(−) 세 가지뿐이다. 셀렉트(σ), 리네임(ρ), 합집합(∪)은 나눗셈 표현에 필요하지 않다. 따라서 정답은 ③번 (Π, −, ×)이다.

오답 분석

보기①

σ, Π, −. 곱집합(×)이 빠져 "모든 조합 생성"을 표현할 수 없고, 불필요한 σ가 포함되어 있다.

보기②

σ, ∪, ρ, −, ×. 프로젝트(Π)가 빠져 있고 σ·∪·ρ 등 불필요한 연산이 다수 포함되어 있다.

보기③

정답. Π, ×, − 세 연산으로 나눗셈을 정확히 구성한다.

보기④

Π, ∪, −, ×. 합집합(∪)이 불필요하게 포함되어 있다. 나눗셈 대체식에 ∪는 등장하지 않는다.

관련 이론 정리

관계대수는 릴레이션을 입력받아 릴레이션을 출력하는 절차적 질의 언어로, 소수의 기본(완전 집합) 연산만으로 다른 모든 연산을 표현할 수 있다. 완전 집합은 셀렉트(σ, 조건 만족 행 선택), 프로젝트(π, 특정 열 추출), 합집합(∪), 차집합(−), 카티션 곱(×), 그리고 리네임(ρ)의 여섯 가지다. 교집합·조인·나눗셈 등은 이 기본 연산으로 유도되는 파생 연산이다. 예를 들어 교집합은 R − (R − S), 자연 조인은 카티션 곱 후 셀렉트·프로젝트, 나눗셈은 프로젝트·카티션 곱·차집합의 조합으로 표현된다. 어떤 파생 연산이 어떤 기본 연산으로 분해되는지 알면, 특정 질의에 반드시 필요한 연산과 불필요한 연산을 구분할 수 있다. 질의 최적화·실행 계획 감리에서 고수준 연산이 기본 연산으로 어떻게 처리되는지 이해하는 기반이 된다.

관계대수 연산 분류:

구분 연산
기본(완전 집합) σ, Π, ∪, ρ, −, ×
파생(유도) ∩(교집합), ⋈(조인), ÷(나눗셈)
나눗셈 구성 Π, ×, −
교집합 구성 R ∩ S = R − (R − S)
실무 적용 사례

쿼리 최적화·논리 검증 감리 시, 고수준 연산(조인·나눗셈)이 기본 연산으로 어떻게 분해되는지 이해하면 실행 계획과 결과 정확성을 검증하는 데 도움이 된다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

두 외래키의 참조 옵션을 정확히 구분해야 한다.
- EMPSUPERFK(Super_ENO → EMPLOYEE.ENO): ON DELETE SET NULL, ON UPDATE CASCADE
- EMPDEPTFK(DNO → DEPARTMENT.DNO): ON DELETE SET DEFAULT, ON UPDATE CASCADE

②번은 "부서가 삭제되면 그 직원도 연쇄 삭제된다"고 서술하지만, DNO 외래키의 삭제 옵션은 SET DEFAULT이다. 따라서 부서가 삭제되면 직원이 삭제되는 것이 아니라 해당 직원의 DNO가 기본값(DEFAULT 1)으로 설정된다. 연쇄 삭제(CASCADE)가 아니므로 ②는 옳지 않다. 정답은 ②번이다.

①(상사 삭제 → Super_ENO SET NULL), ③(부서번호 갱신 → DNO CASCADE로 함께 변경), ④(상사 ENO 갱신 → Super_ENO CASCADE로 함께 변경)는 모두 정의문과 일치한다.

오답 분석

보기①

옳음. Super_ENO는 ON DELETE SET NULL이므로 상사(200번) 삭제 시 100번의 상사 값이 NULL이 된다.

보기②

정답(옳지 않음). DNO는 ON DELETE SET DEFAULT이므로 부서 삭제 시 직원이 삭제되지 않고 DNO가 기본값 1로 바뀐다. "연쇄 삭제"는 틀린 설명.

보기③

옳음. DNO는 ON UPDATE CASCADE이므로 부서번호가 5→6으로 바뀌면 직원의 DNO도 5→6으로 연쇄 갱신된다.

보기④

옳음. Super_ENO는 ON UPDATE CASCADE이므로 상사 ENO가 200→300으로 바뀌면 참조하는 직원의 Super_ENO도 300으로 갱신된다.

관련 이론 정리

참조 무결성은 자식(참조하는) 릴레이션의 외래키 값이 부모(참조되는) 릴레이션에 반드시 존재하도록(또는 NULL) 보장하는 제약이다. 부모의 튜플이 삭제되거나 키가 갱신될 때 이를 참조하는 자식을 어떻게 처리할지 정하는 것이 참조 동작(referential action)이며, ON DELETE/ON UPDATE 절로 지정한다. CASCADE는 부모의 삭제·변경을 자식에 연쇄 적용하고, SET NULL은 자식 외래키를 NULL로, SET DEFAULT는 미리 정한 기본값으로 바꾸며, RESTRICT/NO ACTION은 참조가 남아 있으면 삭제·변경을 거부한다. 중요한 점은 이 동작이 부모의 변경이 자식에 미치는 방향으로 적용된다는 것과, 한 테이블 안에서도 외래키마다 서로 다른 옵션(예: 상사 FK는 SET NULL, 부서 FK는 SET DEFAULT)을 가질 수 있다는 것이다. 감리에서는 각 외래키의 삭제·갱신 옵션이 업무 규칙(부서 폐지 시 사원 처리 등)과 일치하고 고아 레코드가 생기지 않는지 확인한다.

참조 무결성 유지 옵션:

옵션 동작
CASCADE 부모 변경/삭제를 자식에 연쇄 반영
SET NULL 자식 외래키를 NULL로 설정
SET DEFAULT 자식 외래키를 기본값으로 설정
RESTRICT/NO ACTION 참조가 있으면 변경/삭제 거부
실무 적용 사례

테이블 설계 감리 시 외래키의 ON DELETE/UPDATE 옵션이 업무 규칙과 일치하는지 검증한다. 부서 폐지 시 직원을 삭제할지(CASCADE), 미배정 처리할지(SET NULL/DEFAULT)는 업무 정책에 따라 달라진다.

감리사 시험 출제 포인트

동일한 CREATE TABLE 내 서로 다른 외래키가 서로 다른 삭제 옵션(SET NULL vs SET DEFAULT)을 갖는 점을 이용해, 옵션을 뒤섞은 보기를 오답으로 배치한다. "연쇄 삭제(CASCADE)"와 "SET DEFAULT"를 혼동시키는 것이 핵심 함정.

관련 이론 정리

B-트리 계열은 디스크 기반 DBMS 인덱스에 널리 쓰이는 균형 다진 탐색 트리로, 모든 리프가 같은 레벨에 있어(균형) 탐색·삽입·삭제가 트리 높이에 비례한 안정적 성능(O(log n))을 낸다. B-트리는 키와 데이터 포인터를 내부·리프 노드 모두에 저장해 중간 노드에서 탐색이 끝날 수 있는 반면, B+-트리는 실제 데이터(키)를 리프에만 저장하고 내부 노드는 탐색 경로용 인덱스만 두며, 리프끼리 순차 포인터로 연결돼 범위 검색과 순차 접근에 유리하다. 삽입 시 노드가 가득 차면(오버플로) 분할(split)이 일어나 중간 키가 상위로 올라가고, 이것이 루트까지 전파되면 트리 높이가 증가한다. 삭제 시 노드가 최소치 미만이면 병합·재분배가 일어난다. 트리 높이는 곧 디스크 접근 횟수이므로, 인덱스 설계·재구성(rebuild) 감리에서 삽입 패턴에 따른 분할·높이 변화를 검토한다.

- ② 전역 깊이 = 2 → 옳음

관련 이론 정리

확장성 해싱(extendible hashing)은 데이터 증감에 맞춰 디렉토리와 버킷을 동적으로 늘리고 줄이는 동적 해시 기법이다. 디렉토리는 해시값의 상위 비트로 색인되는 포인터 배열이고, 전역 깊이(global depth) d는 디렉토리가 사용하는 비트 수로 디렉토리 엔트리 수는 2^d다. 각 버킷은 자신을 구분하는 데 실제로 사용하는 비트 수인 지역 깊이(local depth) p를 가지며 항상 p ≤ d다. 하나의 버킷을 여러 디렉토리 엔트리가 공유하면 그 버킷의 지역 깊이는 전역 깊이보다 작다(공유 엔트리 2개면 p = d−1). 버킷이 넘칠 때 지역 깊이가 전역 깊이보다 작으면 버킷만 분할하지만, 지역 깊이 = 전역 깊이인 포화 버킷이 넘치면 디렉토리를 2배로 확장(전역 깊이 +1)한 뒤 버킷을 나눈다. 반대로 삭제로 버킷이 비면 버디(짝) 버킷과 병합하고, 모든 지역 깊이가 전역 깊이보다 작아지면 디렉토리를 절반으로 줄인다. 감리에서는 삽입·삭제가 잦은 해시 파일의 저장 효율과 접근 성능을 검토한다.

오버플로 버킷 분할, 필요 시 디렉터리 2배 확장

동적 해시 파일(확장성/선형 해싱) 성능 감리 시 삭제가 잦은 테이블에서 버킷 병합·디렉터리 축소가 적절히 일어나 저장공간과 접근 효율이 유지되는지 점검한다.

GROUP BY는 지정한 컬럼(들)의 값이 같은 행을 하나의 그룹으로 묶고, 각 그룹마다 집계 함수(COUNT·SUM·AVG·MAX·MIN)를 적용해 그룹당 한 행을 반환한다. 표준 SQL에서는 SELECT 절에 그룹 기준 컬럼과 집계 함수만 올 수 있으며, 그룹 기준이 아닌 비집계 컬럼을 함께 쓰면 오류다(어느 행 값을 대표로 쓸지 모호하기 때문). 그룹 집계 결과에 조건을 걸 때는 WHERE가 아니라 HAVING을 쓰는데, WHERE는 그룹화 이전 개별 행을, HAVING은 그룹화 이후 집계값을 필터한다. 뷰나 리포트에서 "~별 집계"를 구현할 때는 반드시 그 분류 축(예: 부서·학과)으로 GROUP BY 해야 하며, GROUP BY를 빠뜨리면 전체가 한 번에 집계된다. 감리에서는 집계 지표의 분류 기준(GROUP BY 컬럼)이 요구 지표와 일치하고 HAVING 조건이 올바른지 확인한다.

집계 뷰(부서별 인원, 지점별 매출 등)를 설계·감리할 때 GROUP BY 기준이 요구 지표의 분류 축과 일치하는지, SELECT 절에 비집계 컬럼이 잘못 포함되지 않았는지 확인한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

다중 단위 로킹에서 IX는 "하위 수준에 X(배타) 로크를 걸 의도", SIX는 "현재 수준 전체에 S(공유) + 하위에 X 의도"를 의미한다. 양립성은 다음과 같다.

  • (가) IX 요청 vs IX 현재: 둘 다 하위에 X를 걸 "의도"만 표현하므로 서로 다른 하위 항목에 배타 로크를 걸 수 있어 양립(허용)된다.
  • (나) IX 요청 vs SIX 현재: SIX는 이미 현재 수준 전체에 S를 포함한다. 여기에 IX(하위 X 의도)를 겹치면 SIX의 공유 범위와 충돌하므로 비양립(대기).
  • (다) SIX 요청 vs IX 현재: SIX의 S 부분이 기존 IX(하위 X 의도)와 충돌 → 비양립(대기).
  • (라) SIX 요청 vs SIX 현재: 양쪽 모두 S + 하위 X 의도를 포함해 서로 배타적이므로 비양립(대기).

따라서 (가)허용, (나)대기, (다)대기, (라)대기로 정답은 ④번이다.

관련 이론 정리

다중 단위 로킹(multiple granularity locking)은 데이터베이스-테이블-페이지-행처럼 계층적으로 구성된 데이터 항목에 대해, 잠그려는 수준과 그 상위 수준의 의도를 함께 표시해 로크 충돌을 효율적으로 판정하는 기법이다. 하위 항목에 실제 로크를 걸기 전에 상위 항목에 의도 로크(intention lock)를 걸어 "아래에서 무슨 로크가 진행 중"인지 알린다. IS(하위에 공유 로크 의도), IX(하위에 배타 로크 의도), SIX(현재 수준 전체에 공유 + 하위에 배타 의도)가 대표적이다. 양립성 규칙에서 IS는 대부분과 양립하고, IX는 IX끼리 양립(서로 다른 하위 항목에 배타 로크 가능)하지만 S·SIX·X와는 충돌한다. SIX는 이미 공유를 포함하므로 IX·S·SIX·X 대부분과 충돌한다. 이런 양립성 행렬로 서로 다른 트랜잭션이 계층의 여러 수준에서 동시에 작업할 수 있게 병행성을 조율한다. 감리에서는 로크 승격(escalation) 정책과 교착·경합 관리를 점검한다.

실무 적용 사례

대용량 트랜잭션 동시성 감리 시 테이블·페이지·행 등 계층적 로킹 수준에서 의도 로크(IS/IX/SIX)를 통해 잠금 충돌과 병행성을 조율한다. 로크 승격(lock escalation) 정책도 함께 점검한다.

정답 해설

HDFS는 파일을 블록으로 나눠 여러 DataNode에 복제(기본 3벌) 저장한다. 읽기는 클라이언트와 네트워크상 가까운(지역성 높은) DataNode에서 블록을 가져오므로 빠르다. 반면 쓰기는 블록을 여러 DataNode에 복제 파이프라인으로 전파하고 모든 복제본 저장이 확인되어야 완료되므로 상대적으로 느리다.

따라서 "읽기는 가까운 DataNode에서 가져와 빠르고, 쓰기는 모든 복제본 저장을 기다려 느리다"는 ①번이 옳다. NameNode는 메타데이터만 관리하며 실제 데이터 입출력은 DataNode가 담당하므로 ③은 틀리고, HDFS는 write-once/read-many 모델로 ②·④의 설명도 잘못됐다.

관련 이론 정리

HDFS(Hadoop Distributed File System)는 대용량 데이터를 여러 노드에 분산 저장하는 마스터-슬레이브 구조의 분산 파일시스템이다. NameNode(마스터)는 파일→블록→위치의 메타데이터를 관리하고 어느 DataNode에 어떤 블록이 있는지 유지하며(실제 데이터는 저장하지 않음), DataNode(슬레이브)가 실제 데이터 블록을 저장하고 클라이언트에 제공한다. 파일은 큰 블록(기본 128MB 등)으로 나뉘어 기본 3벌로 복제 저장되므로 일부 노드가 고장나도 데이터가 유지되고(내결함성), 랙 인식(rack awareness)으로 복제본을 분산 배치한다. 접근 모델은 write-once, read-many로, 읽기는 클라이언트와 가까운 복제본에서 가져와 빠르지만 쓰기는 여러 복제본에 파이프라인으로 전파·확인해야 해 상대적으로 느리다. Secondary NameNode는 메타데이터 체크포인트/백업 역할을 한다. 감리에서는 복제 계수·NameNode 이중화(HA)·데이터 지역성 설정이 성능·가용성 요구에 맞는지 검토한다.

실무 적용 사례

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

Apriori의 핵심은 하향 폐쇄(downward closure) = 비-단조 성질이다. 즉 "어떤 항목집합이 빈발하지 않으면(최소 지지도 미달), 그것을 포함하는 모든 상위 집합도 빈발할 수 없다". 상위 집합의 지지도는 부분집합의 지지도보다 크지 않기(작거나 같기) 때문이다. 이 성질 덕분에 빈발하지 않은 집합의 상위 집합을 후보에서 미리 제거(가지치기)하여 탐색 공간을 줄인다.

따라서 ①번이 옳다. 지지도는 집합이 커질수록 감소(비증가)하므로 ②는 틀리고, 빈발집합의 상위집합이 항상 빈발한 것은 아니므로 ③도 틀리며, ④는 지지도와 연관규칙 가능성의 관계를 잘못 서술했다.

오답 분석

보기①

정답. 비-단조(하향 폐쇄): 빈발하지 않은 집합의 상위 집합은 빈발하지 않다 → 가지치기의 근거.

보기②

반대. 항목집합이 커질수록 지지도는 감소(비증가)한다.

보기③

빈발집합의 상위집합이 반드시 빈발한 것은 아니다(상위로 갈수록 지지도 감소). 상향 폐쇄는 성립하지 않는다.

보기④

지지도가 낮으면 오히려 빈발집합에서 제외되어 연관규칙이 될 가능성이 낮다. 서술이 잘못됨.

관련 이론 정리

연관 규칙 마이닝은 "X를 사면 Y도 산다"처럼 항목 간 동시 발생 패턴을 찾는 기법으로, 대표 알고리즘이 Apriori다. 규칙의 흥미도는 세 지표로 평가한다. 지지도(support)는 전체 거래 중 특정 항목집합이 함께 등장하는 비율, 신뢰도(confidence)는 X를 포함한 거래 중 Y도 포함하는 비율(=support(X∪Y)/support(X)), 향상도(lift)는 신뢰도를 Y의 지지도로 나눈 값으로 우연 대비 상관 강도를 나타낸다. Apriori의 핵심 원리는 비-단조성(하향 폐쇄)으로, 어떤 항목집합이 최소지지도를 넘지 못하면(비빈발) 그것을 포함하는 모든 상위 집합도 비빈발이라는 성질이다. 이를 이용해 비빈발 집합의 상위 후보를 미리 가지치기(pruning)해 탐색 공간을 줄인다. 항목집합이 커질수록 지지도는 감소(비증가)하므로 빈발 1-항목에서 시작해 크기를 늘려간다. 장바구니 분석·추천 감리에서 최소지지도·신뢰도 설정과 가지치기의 타당성을 검토한다.

연관 규칙 마이닝 지표:

지표 정의
지지도(support) 전체 거래 중 항목집합 포함 비율
신뢰도(confidence) X→Y에서 X 포함 거래 중 Y도 포함 비율
향상도(lift) 신뢰도 / Y의 지지도(독립 대비 상관)
비-단조성 부분집합이 비빈발이면 상위집합도 비빈발
실무 적용 사례

장바구니 분석·추천 시스템 감리 시 최소 지지도·신뢰도 임계값 설정과 Apriori 가지치기로 후보를 줄이는 과정이 대용량 거래 데이터에서 효율적으로 동작하는지 검토한다.

감리사 시험 출제 포인트

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

베이즈 정리는 다음과 같다.

P(C1|X) = P(X|C1)·P(C1) / [ P(X|C1)·P(C1) + P(X|C2)·P(C2) ]

주어진 값을 대입하면:
- 분자 = P(X|C1)·P(C1) = 0.6 × 0.5 = 0.30
- 분모 = 0.6×0.5 + 0.4×0.5 = 0.30 + 0.20 = 0.50

P(C1|X) = 0.30 / 0.50 = 0.6

따라서 정답은 ④번 0.6이다. 사전확률이 동일(0.5)하므로 사후확률은 우도의 비율 0.6/(0.6+0.4)=0.6으로도 바로 구할 수 있다.

오답 분석

보기①

보기②

보기③

0.4는 P(X|C2) 또는 C2 방향 값을 혼동한 오답이다.

보기④

정답. 0.30 / 0.50 = 0.6.

관련 이론 정리

베이즈 분류는 관측된 특성을 근거로 각 클래스일 확률(사후확률)을 계산해 가장 확률이 높은 클래스로 분류한다. 베이즈 정리는 P(C|X) = P(X|C)·P(C) / P(X)로, 사전확률 P(C)(데이터 관측 전 클래스 확률)에 우도 P(X|C)(그 클래스에서 특성 X가 나타날 확률)를 곱하고, 전체 확률 P(X)=Σ P(X|Ci)P(Ci)로 나눠 정규화한다. 여기서 정규화(분모)를 빠뜨리고 분자만 계산하는 것이 대표적 실수이며, 사전확률이 모두 같으면 사후확률은 우도의 비율로 간단히 구해진다. 나이브 베이즈는 특성들이 클래스 조건 하에서 서로 독립이라고 가정해 여러 특성의 우도를 곱으로 계산하는 실용적 변형으로, 스팸 필터·문서 분류에 널리 쓰인다. 확률이 0이 되는 문제는 라플라스 보정으로 완화한다. 감리에서는 사전확률·우도 추정이 데이터에 맞고 정규화·평활 처리가 적절한지 검토한다.

증거 P(X) 전확률(정규화 상수) = Σ P(X|Ci)P(Ci)
사후확률 P(Ci|X) 특성 관측 후 클래스 확률
실무 적용 사례

스팸 필터·문서 분류·이상 탐지 등 나이브 베이즈 기반 모델 감리 시 사전확률·우도 추정과 정규화가 올바른지, 클래스 불균형이 사전확률에 반영됐는지 확인한다.

따라서 복구 후 값이 틀린 것은 ③번(C는 8이 아니라 9)이다.

관련 이론 정리

트랜잭션 회복은 장애 발생 시 로그를 이용해 데이터베이스를 일관된 상태로 되돌리는 절차다. 로그에는 각 변경의 <트랜잭션, 데이터, 변경 전 값(BFIM), 변경 후 값(AFIM)>이 기록되며, 회복 시 커밋이 완료된 트랜잭션은 REDO(변경 후 값으로 재실행), 커밋되지 않은 트랜잭션은 UNDO(변경 전 값으로 취소)한다. 갱신 방식에 따라 즉시 갱신(immediate update)은 커밋 전에도 디스크에 반영될 수 있어 UNDO와 REDO가 모두 필요하고, 지연 갱신(deferred update)은 커밋 전까지 반영하지 않아 UNDO가 불필요(NO-UNDO)하며 REDO만 필요하다. WAL(로그 선기록) 원칙에 따라 데이터보다 로그를 먼저 기록하고, 체크포인트를 두면 그 이전 커밋분은 반영 완료로 간주해 REDO 범위를 줄인다. 감리에서는 로그 방식·체크포인트 주기·REDO/UNDO 대상 선정이 표준(ARIES 등)에 부합해 복구 정합성을 보장하는지 확인한다.

로그 기반 회복 규칙:

트랜잭션 상태 처리
commit 존재 redo(변경 후 값)
commit 없음(active) undo(변경 전 값)
checkpoint 이전 커밋분은 반영 완료로 간주(redo 범위 축소)
  • 즉시 갱신은 undo와 redo가 모두 필요(지연 갱신은 redo만).
실무 적용 사례

DBMS 장애 복구(crash recovery) 감리 시 로그 방식(즉시/지연), 체크포인트 주기, redo/undo 대상 선정이 ARIES 등 표준 절차에 부합하는지, 복구 후 데이터 정합성이 보장되는지 점검한다.

감리사 시험 출제 포인트

"커밋=redo(후값), 미커밋=undo(전값)"를 각 데이터에 정확히 적용한다. 커밋된 T3의 C를 변경 전 값(8)으로 남겨두는 함정이 핵심이다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

DTD <!ELEMENT x (a, (b | c), d)*>는 x의 자식이 "a, (b 또는 c), d" 그룹이 0회 이상 반복됨을 뜻한다. 즉 한 그룹은 반드시 a → (b 또는 c) → d 순서와 개수를 지켜야 한다.

  • ② <x><a/><d/></x>: a 다음에 (b | c) 요소가 없이 곧바로 d가 왔다. 필수인 (b 또는 c)가 누락되어 콘텐츠 모델을 위반하므로 파싱(유효성) 오류가 발생한다. 따라서 정답은 ②번이다.

①은 그룹 0회(*)로 허용, ③은 a·b·d 한 그룹으로 정상, ④는 (a·b·d)(a·c·d) 두 그룹으로 정상이다.

보기①

보기②

정답(오류). a와 d 사이에 필수인 (b | c)가 빠져 콘텐츠 모델 위반.

보기④

관련 이론 정리

DTD(Document Type Definition)는 XML 문서의 구조(요소·속성·중첩·순서·개수)를 정의하는 스키마로, 인스턴스가 이 규칙을 지키는지 검증(유효성 검사)한다. 요소의 콘텐츠 모델은 자식 요소의 구성을 연산자로 표현한다. 콤마(,)는 순서 있는 시퀀스(반드시 그 순서대로 모두), 파이프(|)는 선택(택일), 별표(*)는 0회 이상 반복, 플러스(+)는 1회 이상, 물음표(?)는 0 또는 1회를 뜻한다. 예컨대 (a, (b|c), d)*는 "a → (b 또는 c) → d" 그룹이 0회 이상 반복됨을 의미하므로, 각 반복 단위에는 a·(b 또는 c)·d가 모두 순서대로 있어야 한다. 따라서 (b|c)를 통째로 생략하면 필수 요소 누락으로 파싱(유효성) 오류가 난다. XSD는 DTD보다 자료형·네임스페이스를 정교하게 지원하는 후속 스키마다. 전자문서·연계 표준 감리에서 스키마와 인스턴스의 유효성 검증을 확인한다.

DTD 콘텐츠 모델 연산자:

기호 의미

| 선택(택일)
* 0회 이상 반복
+ 1회 이상 반복
? 0 또는 1회(선택)
  • (a, (b|c), d)*: 각 반복 단위는 a, (b 또는 c), d를 모두·순서대로 포함해야 함.
실무 적용 사례

XML 기반 전자문서·연계 표준(전자정부 문서, 금융 메시지 등) 감리 시 스키마(DTD/XSD)와 실제 인스턴스의 유효성 검증을 확인한다. 필수 요소 누락·순서 오류가 파싱 실패로 이어지는지 점검한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

NoSQL(분산 데이터베이스)의 설계 트레이드오프는 CAP 정리로 설명된다. CAP는 일관성(Consistency, 나), 가용성(Availability, 마), 분할 감내성(Partition tolerance, 다)의 세 성질을 말하며, 분산 환경에서는 이 셋을 동시에 만족할 수 없어 둘을 택한다는 것이 핵심이다. 따라서 NoSQL과 관련된 성질은 나·다·마이며 정답은 ③번이다.

반면 원자성(atomicity, 가)과 격리성(isolation, 라)은 전통적 관계형 트랜잭션의 ACID 속성으로, CAP(NoSQL 문맥)의 구성 요소가 아니다. 그러므로 가·라를 포함한 ①·②·④는 부적절하다.

오답 분석

보기①

가(원자성)·라(격리성)는 ACID 속성으로 CAP/NoSQL과 직접 관련이 적다.

보기②

가·라가 포함되어 부적절. 다(분할 감내성)만 CAP 요소.

보기③

정답. 나(일관성)·다(분할 감내성)·마(가용성) = CAP 정리의 세 성질.

보기④

라(격리성)가 포함되어 부적절.

관련 이론 정리

관계형 트랜잭션의 신뢰성은 ACID로 요약된다. 원자성(Atomicity, 전부 또는 전무), 일관성(Consistency, 제약 유지), 격리성(Isolation, 동시 실행이 직렬 실행과 동등), 지속성(Durability, 커밋 결과 영속)이다. 반면 여러 노드에 복제되는 분산·NoSQL 시스템의 트레이드오프는 CAP 정리로 설명된다. 일관성(Consistency, 모든 노드가 같은 데이터), 가용성(Availability, 항상 응답), 분할 감내성(Partition tolerance, 네트워크 분할에도 동작) 셋 중 동시에 둘까지만 만족할 수 있다는 것이다. 분산 환경에서 네트워크 분할(P)은 사실상 불가피하므로 실제로는 C와 A 사이에서 절충하며, 많은 NoSQL은 강한 일관성 대신 궁극적 일관성(eventual consistency)을 수용하는 BASE(기본 가용·연성 상태·궁극적 일관성) 모델을 지향한다. 원자성·격리성은 ACID 속성이지 CAP 요소가 아니라는 점이 혼동 포인트다. 저장소 선정 감리에서 요구 일관성 수준과 가용성·분할 내성 우선순위를 CAP 관점으로 검토한다.

ACID vs CAP:

CAP(분산/NoSQL) 일관성·가용성·분할 감내성
BASE(NoSQL 지향) 기본가용·연성상태·최종일관성
  • 일관성(consistency)은 ACID·CAP 양쪽에 등장하나 의미가 약간 다르다(CAP는 분산 복제본 간 일관성).
실무 적용 사례

대규모 분산 데이터 저장소 선정 감리 시 요구되는 일관성 수준(강한/최종)과 가용성·파티션 내성 요구를 CAP 관점에서 평가해 적합한 NoSQL(문서/키값/컬럼/그래프)을 선택했는지 검토한다.

감리사 시험 출제 포인트

ACID(원자성·격리성)와 CAP(가용성·분할 감내성)를 혼동시키는 것이 핵심 함정이다. 일관성은 양쪽에 걸쳐 있음에 유의한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

이진 분류에서 지니 지수는 Gini = 1 − Σ p_i² 이다.

  • ① (0, 6): 한 클래스 비율 0, 다른 클래스 1 → Gini = 1 − (0² + 1²) = 0(순수 노드). 옳음.
  • ② (1, 5): p = 1/6, 5/6 → Gini = 1 − ((1/6)² + (5/6)²) = 1 − (1/36 + 25/36) = 1 − 26/36 ≈ 0.28 > 0. 옳음.
  • ③ (3, 3): p = 0.5, 0.5 → 분류 오류 = 1 − max(0.5, 0.5) = 0.5. 옳음.
  • ④ (3, 3): Gini = 1 − (0.5² + 0.5²) = 1 − 0.5 = 0.5이다. 보기 ④는 "Gini Index는 1이다"라고 했으나 실제는 0.5이므로 틀렸다.

이진 분류에서 지니 지수의 최댓값은 0.5(완전 반반)이며 1이 될 수 없다. 따라서 옳지 않은 설명은 ④번이다.

보기①

보기②

관련 이론 정리

의사결정 트리는 각 노드에서 데이터를 가장 잘 나누는 속성을 골라 분기하며, 이때 노드의 불순도(impurity)를 얼마나 줄이는지(정보 이득)로 분기를 결정한다. 이진 분류에서 노드의 클래스 비율이 p, (1−p)일 때 대표 불순도 지표는 다음과 같다. 지니 지수(Gini) = 1 − Σp²는 완전히 한 클래스만 있으면(순수) 0, 반반(0.5,0.5)이면 최대 0.5다. 분류 오류(classification error) = 1 − max(p)도 순수면 0, 반반이면 0.5가 최대다. 엔트로피 = −Σp·log₂p는 순수면 0, 반반이면 최대 1이다. 즉 이진 분류에서 지니·분류오류의 최댓값은 0.5인 반면 엔트로피 최댓값은 1이라는 차이가 핵심 함정이다. 예로 (3,3)이면 Gini=0.5, 오류율=0.5, 엔트로피=1이다. 불순도가 높을수록 분기가 필요하며, 과적합을 막기 위해 가지치기·최대깊이를 둔다. 모델 감리에서 분기 기준·과적합 방지 설정을 검토한다.

실무 적용 사례

의사결정 트리·랜덤포레스트 모델 감리 시 분기 기준(Gini/엔트로피/오분류율)과 과적합 방지(가지치기·최대깊이) 설정이 적절한지, 불순도 감소(정보 이득)가 분기 타당성을 뒷받침하는지 확인한다.

정답 해설

각 포맷의 특징을 보면, ①(Key-Value: NoSQL 고속 조회), ②(JSON: 속성-값·트리 구조·문자열 기반·통신에 사용), ③(XML: 태그 기반 계층 구조·문서 교환)은 모두 정확한 설명이다.

그러나 ④번은 CSV에 대해 "중첩 구조를 표현할 수 있다"고 서술한 부분이 틀렸다. CSV는 쉼표/탭으로 구분된 평면(flat)적인 행·열 2차원 구조로 정형 데이터에는 적합하지만, JSON·XML과 달리 중첩(계층) 구조를 표현할 수 없다. 따라서 가장 적합하지 않은 설명은 ④번이다.

오답 분석

보기①

옳음. Key-Value Store는 키-값 형태의 고속 조회 구조로 NoSQL에서 널리 사용된다.

보기②

옳음. JSON은 속성-값 쌍의 트리 구조, 문자열 기반, XML과 유사한 데이터 교환 용도로 통신에 자주 쓰인다.

보기③

옳음. XML은 태그 기반 계층 구조로 문서 저장·교환에 사용된다.

보기④

정답(부적합). CSV는 평면 2차원 구조로 중첩 구조를 표현할 수 없다. "중첩 구조 표현 가능"은 잘못된 설명.

관련 이론 정리

데이터 저장·교환 포맷은 구조 표현력과 용도가 서로 다르다. CSV는 쉼표/탭으로 구분한 평면(flat) 행·열 2차원 구조로, 표 형태 정형 데이터에 간단·효율적이지만 중첩(계층) 구조는 표현할 수 없다. JSON은 속성-값 쌍과 배열로 트리(중첩) 구조를 표현하는 문자열 기반 경량 포맷으로 웹 API·통신에 널리 쓰인다. XML은 태그로 계층 구조를 표현하고 스키마(DTD/XSD)·네임스페이스로 엄격한 검증이 가능해 문서·전자거래 교환에 쓰이나 JSON보다 장황하다. Key-Value 스토어는 키로 값을 빠르게 조회하는 단순 구조로 NoSQL 캐시·세션 저장에 적합하다. 핵심 구분은 CSV는 평면(중첩 불가), JSON·XML은 계층(중첩 가능)이라는 점이다. 데이터 연계·적재(ETL) 감리에서는 중첩 데이터를 평면 포맷으로 옮기며 정보 손실이 없는지, 교환 포맷 선택이 적절한지 확인한다.

데이터 포맷 비교:

포맷 구조 중첩 표현 주 용도
Key-Value 키-값 제한적 NoSQL 고속 조회
JSON 트리(객체/배열) 가능 웹/통신 데이터 교환
XML 태그 계층 가능 문서 저장·교환
CSV 평면 행·열 불가 정형 표 데이터
실무 적용 사례

데이터 연계·적재(ETL) 감리 시 계층/중첩 데이터를 CSV로 평면화하며 정보가 손실되지 않는지, 반대로 중첩이 필요한 교환에는 JSON/XML을 쓰는지 포맷 선택의 적정성을 점검한다.

감리사 시험 출제 포인트

CSV는 평면 구조로 중첩 불가, JSON·XML은 계층 표현 가능이라는 대비가 핵심이다. CSV에 "중첩 표현 가능"을 붙인 보기가 대표 오답이다.
