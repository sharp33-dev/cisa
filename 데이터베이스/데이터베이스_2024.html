영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

NoSQL 데이터 모델은 크게 네 가지로 분류된다. 키-값(Key-Value), 문서(Document), 컬럼 패밀리(Column-family), 그래프(Graph) 모델이다. 이 중 문서 기반(document based) 모델은 JSON/BSON 형식으로 복잡한 구조(중첩 객체·배열)를 하나의 문서로 저장하며, 대표 제품이 바로 MongoDB이다.

문제의 설명 "JSON을 기반으로 복잡한 구조의 객체를 저장" + "대표 제품 MongoDB"는 정확히 문서 기반 모델의 특징이다. 따라서 정답은 ①번이다. 나머지 보기의 '객체 지향', '구조 기반', '자바 기반' 데이터 모델은 NoSQL의 표준 분류 명칭이 아니다.

오답 분석

보기①

정답. JSON 기반 복잡 객체 저장 + MongoDB = 문서 기반(Document) 모델.

보기②

객체 지향 데이터 모델(OODB)은 프로그래밍 객체를 그대로 저장하는 별개 개념으로, NoSQL 문서 모델의 명칭이 아니다.

보기③

'구조 기반 데이터 모델'은 표준 NoSQL 분류에 없는 명칭이다.

보기④

'자바 기반 데이터 모델'도 존재하지 않는 분류다. MongoDB는 특정 언어에 종속되지 않는다.

관련 이론 정리

NoSQL 데이터 모델 4대 유형:

유형 특징 대표 제품
Key-Value 키-값 단순 저장, 초고속 조회 Redis, DynamoDB
Document JSON/BSON 문서, 중첩 구조 MongoDB, CouchDB
Column-family 컬럼 단위 저장, 대용량 분산 Cassandra, HBase
Graph 노드·간선 관계 표현 Neo4j
실무 적용 사례

빅데이터·서비스 아키텍처 감리 시 데이터의 구조(중첩/관계/단순 조회)에 맞는 NoSQL 유형을 선택했는지 검토한다. 스키마 변화가 잦고 문서형 데이터가 많으면 MongoDB 같은 문서 모델이 적합하다.

감리사 시험 출제 포인트

NoSQL 4대 유형과 대표 제품 매칭은 빈출이다. "JSON + MongoDB = 문서형"을 정확히 연결할 수 있어야 한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

각 연산의 결과 카디널리티를 계산한다.

① R ÷ Π_B(S): Π_B(S) = {10, 20, 30}. R의 A값 중 {10,20,30} 모두와 짝을 이루는 것을 찾는다. A=1은 {10,30}(20 없음)→제외, A=2는 {10,30,20} 모두→포함, A=3은 {30}만→제외. 결과 {2} → 카디널리티 1.

② R ⋈_N S (B로 자연 조인): R의 6개 투플이 각각 S의 해당 B와 1:1 매칭(10→a, 20→b, 30→c) → 6개.

③ R ⋈_{R.B<S.B} S (R.B < S.B): 각 R투플에 대해 S.B(10,20,30) 중 더 큰 값 개수. (1,10)→2, (1,30)→0, (2,10)→2, (2,30)→0, (2,20)→1, (3,30)→0 = 5개.

④ σ_{B=10}(R): B=10인 R투플 (1,10),(2,10) → 2개.

가장 큰 값은 자연 조인의 6, 가장 작은 값은 나눗셈의 1이다. 합 = 6 + 1 = 7. 정답은 ②번이다.

정답. 최대(자연 조인 6) + 최소(나눗셈 1) = 7.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

조인 구현 알고리즘 중 정렬 합병 조인(sort-merge join)은 두 릴레이션이 조인 속성으로 정렬되어 있을 때 양쪽을 병합(merge)하듯 순차 스캔하며 매칭한다. 두 릴레이션이 이미 조인 속성 A로 오름차순 정렬되어 있으면 별도 정렬 비용 없이 곧바로 합병 단계만 수행할 수 있어, "정렬되어 있을 경우에만" 효율적으로 적용된다.

반면 중첩 루프(①)는 정렬과 무관하게 모든 쌍을 비교하고, 인덱스 조인(②)은 인덱스가 필요하며, 해시 조인(③)은 해시 테이블을 사용하므로 정렬 전제와 무관하다. 따라서 정렬 전제에 부합하는 방법은 ④번 정렬 합병이다.

오답 분석

보기①

중첩 루프(nested loop) 조인은 한 릴레이션의 각 행에 대해 다른 릴레이션 전체를 탐색한다. 정렬 여부와 무관하다.

보기②

인덱스 조인은 조인 속성에 인덱스가 있을 때 효율적이며, 정렬이 아니라 인덱스가 전제다.

보기③

해시 조인은 조인 속성으로 해시 테이블을 만들어 매칭한다. 정렬 전제가 아니다.

보기④

정답. 정렬 합병 조인은 양쪽이 조인 속성으로 정렬되어 있을 때 합병으로 조인한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

낙관적 로킹은 트랜잭션 수행 중에는 로크를 걸지 않고, 커밋 시점에만 충돌(버전 불일치)을 검사한다. 비관적 로킹은 데이터 접근 전에 미리 로크를 획득한다.

인터넷/웹 환경은 사용자 think-time이 길고 작업을 중단하는 등 예측 불가능하다. 이런 환경에서 비관적 로킹을 쓰면 사용자가 화면을 열어둔 채 방치할 때 로크가 장시간 유지되어 다른 사용자를 막게 되어 매우 비효율적이다. 따라서 웹 환경에서는 오히려 낙관적 로킹이 더 적합하다. ①은 이런 환경에서 "비관적 로킹이 더 효율적"이라고 반대로 서술했으므로 옳지 않다. 정답은 ①번이다.

②(복잡·저속 클라이언트에 낙관적 로킹 유리), ③(낙관적 로킹은 로크 보유 시간이 짧음), ④(특정 행에 작업 집중 시 비관적 로킹 유리)는 모두 옳은 설명이다.

오답 분석

보기①

정답(옳지 않음). 인터넷/웹처럼 사용자 행동이 예측 불가능한 환경에서는 로크를 오래 잡는 비관적 로킹이 비효율적이며, 낙관적 로킹이 적합하다.

보기②

옳음. 트랜잭션이 복잡하거나 클라이언트가 느리면 로크를 잡지 않는 낙관적 로킹이 동시성·생산성 면에서 유리하다.

보기③

옳음. 낙관적 로킹은 커밋 시점에만 검사하므로 로크 보유 시간이 매우 짧다.

보기④

옳음. 특정 행에 갱신이 집중되어 충돌 가능성이 높으면 미리 로크를 잡는 비관적 로킹이 낫다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

"계좌(부모) 삭제 시 예금자(자식) 투플도 삭제"하려면, 자식인 예금자 릴레이션에 계좌번호를 외래키로 두고 ON DELETE CASCADE를 지정해야 한다. 외래키는 항상 참조하는 쪽(자식)에 정의하며, 삭제 연쇄 규칙은 ON DELETE CASCADE가 표준 SQL 구문이다(IN DELETE는 잘못된 구문).

따라서 "예금자 생성 시 FOREIGN KEY(계좌번호) REFERENCES 계좌(계좌번호) ON DELETE CASCADE"인 ①번이 옳다. ②는 IN DELETE라는 잘못된 구문, ③·④는 외래키를 부모(계좌)에 두어 참조 방향이 반대이므로 틀렸다.

오답 분석

보기①

정답. 자식(예금자)에 외래키 + ON DELETE CASCADE로 계좌 삭제 시 예금자 연쇄 삭제.

보기②

IN DELETE CASCADE는 존재하지 않는 잘못된 구문이다. 올바른 구문은 ON DELETE CASCADE.

보기③

외래키를 부모인 계좌에 두고 예금자를 참조 → 참조 방향이 반대라 요구를 충족하지 못한다.

보기④

참조 방향도 반대이고 IN DELETE라는 잘못된 구문까지 사용해 이중으로 틀렸다.

N:M 관계는 어느 한 쪽 개체에 외래키를 넣는 것으로 표현할 수 없고, 별도의 관계(교차) 릴레이션을 신설하여 양쪽 개체의 기본키(SNO, CNO)를 함께 외래키이자 복합 기본키로 두고, 관계 자체의 속성(Grade)을 포함시킨다.

따라서 STUDENT(SNO, Name), COURSE(CNO, CName)는 그대로 두고, ENROL(SNO, CNO, Grade)라는 관계 릴레이션을 만드는 ④번이 옳다. ①·②처럼 한 개체에 상대 키를 넣으면 N:M을 표현하지 못하고, ③처럼 ENROL에 키(SNO, CNO)가 없으면 어떤 학생-과목 쌍인지 식별할 수 없다.

오답 분석

보기①

STUDENT에 CNO를 외래키로 넣는 방식은 1:N에서나 가능하다. N:M에서는 한 학생이 여러 과목을 들으므로 표현 불가.

보기②

COURSE에 SNO를 넣는 것도 마찬가지로 N:M 표현 불가(한 과목에 여러 학생).

보기③

ENROL(Grade)에 SNO·CNO 키가 없어 어떤 수강 쌍인지 식별 불가. 관계 릴레이션 요건 미충족.

보기④

정답. ENROL(SNO, CNO, Grade) 교차 릴레이션으로 N:M을 정확히 사상.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

BCNF는 "모든 비자명 함수 종속의 결정자가 후보키"일 것을 요구한다. 각 분해에서 R1, R2가 모두 BCNF인지 확인한다.

④번: FD (A,B)→CD, C→A. 분해 R1(A, C), R2(B, C, D).
- R1(A, C): 성립하는 FD는 C→A. R1에서 C가 후보키(C→A로 전체 결정)이고 결정자 C가 키이므로 BCNF.
- R2(B, C, D): C→A(A는 R2에 없음), (A,B)→CD에서 C→A를 이용하면 (B,C)→D가 유도된다. R2의 후보키는 (B,C)이고 유일한 비자명 FD (B,C)→D의 결정자가 키이므로 BCNF.

두 분해 모두 BCNF이므로 정답은 ④번이다. 나머지 보기는 분해된 릴레이션 중 하나에 결정자가 후보키가 아닌 종속(예: C→D, B→C 등에서 결정자가 키가 아님)이 남아 BCNF를 만족하지 못한다.

오답 분석

보기①

R2(B, C, D)에 B→C, C→D가 남아 결정자 B·C가 후보키가 아니므로 BCNF 위반.

보기②

R1(A, B, C)에 B→C가 남고 B는 후보키(A)가 아니므로 BCNF 위반(이행 종속 존재).

보기③

R1(A, B, C)에 C→A 등 결정자가 키가 아닌 종속이 남아 BCNF 위반.

보기④

정답. R1(A, C)·R2(B, C, D) 모두 결정자가 후보키 → BCNF.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

4NF는 "하나의 테이블에 서로 독립적인 다치 종속이 둘 이상 존재하지 않을 것"을 요구한다. 직원이름 →→ 학위명, 직원이름 →→ 형제이름은 서로 독립적인 다치 종속이므로, 이 둘을 한 테이블에 두면(예: 직원이름, 학위명, 형제이름) 학위와 형제의 모든 조합이 곱해져 중복이 발생하여 4NF 위반이다.

따라서 각 다치 종속을 별도 테이블로 분리해야 한다. ① 직원_학위(직원이름, 학위명)는 직원이름→→학위명 하나만 담아 4NF를 만족한다. ②는 독립 MVD 두 개(학위·형제)를 함께 담아 위반, ③은 파트키트_파트_가격에 파트명·가격이 함께 있으나 이는 함수 종속(파트키트명→파트명, →가격)으로 4NF 자체 위반은 아니지만 본 문항에서 "4NF에 해당하는 테이블"로 가장 명확한 정답은 ①이다(④는 상품과 학위를 섞어 부적절). 그러므로 정답은 ①번이다.

오답 분석

보기①

정답. 직원이름→→학위명 하나의 다치 종속만 포함 → 4NF 만족.

보기②

직원이름→→학위명과 →→형제이름 두 독립 다치 종속을 함께 담아 4NF 위반(조합 중복 발생).

보기③

파트키트명이 결정하는 파트명·가격을 함께 두었는데, 4NF 관점의 독립 다치 종속 분리라는 요구에 정확히 부합하는 대표 정답은 ①이다(본 보기는 서로 다른 속성 결합으로 부적절).

보기④

담당상품번호(함수 종속)와 학위명(다치 종속)을 섞어 부적절하다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

인덱스 ROWID로 테이블 블록을 읽는 전형적 절차는 다음과 같다.

  1. ㄹ: 인덱스에서 ROWID를 읽고, 그 물리 위치를 해시 함수에 적용해 해시값(버킷)을 구한다.
  2. ㄷ: 해당 해시 버킷의 해시 체인을 스캔해 원하는 블록(버퍼 헤더)이 캐시에 있는지 찾는다.
  3. (캐시에 없으면 - 캐시 미스) ㄱ: LRU 리스트를 스캔하며 재사용할 빈 버퍼를 찾는다.
  4. ㅁ: 빈 버퍼가 없으면 Dirty 버퍼를 디스크에 기록(flush)해 공간을 확보한다.
  5. ㄴ: 디스크에서 블록을 읽어 버퍼 캐시에 적재한다.

따라서 순서는 ㄹ → ㄷ → ㄱ → ㅁ → ㄴ으로 정답은 ②번이다.

보기①

보기③

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

질의는 직원에 대해 두 개의 EXISTS 조건을 AND로 결합한다.
- 첫 EXISTS: 그 직원의 직원번호 = 부양가족의 부양직원번호 → 부양가족을 최소 한 명 보유한 직원.
- 둘째 EXISTS: 그 직원의 직원번호 = 부서의 관리자_직원번호 → 어떤 부서의 관리자인 직원.

AND로 결합했으므로 두 조건을 모두 만족하는 직원, 즉 "부양가족을 가진 관리자"의 이름을 검색한다. 따라서 정답은 ②번이다. ①은 첫 조건만, ③은 AND를 OR로 오해, ④는 검색 대상(직원 이름 vs 부양가족 이름)을 잘못 서술했다.

보기②

정답. 부양가족 보유 AND 관리자 = 부양가족을 가진 관리자.

보기③

보기④

정답 해설

F-스코어는 정밀도와 재현율의 조화 평균이다. 조화 평균은 두 값 중 작은 값 쪽으로 치우치는(가까워지는) 성질이 있다(둘 중 하나가 낮으면 F-score도 낮아짐). 따라서 ④가 "더 큰 쪽으로 가까워진다"고 서술한 것은 틀렸다. 정답(적절하지 않은 것)은 ④번이다.

①(재현율 정의), ②(precision=3/5=60%, recall=3/10=30%), ③(평균 정밀도 정의)은 모두 옳다.

보기②

옳음. precision = 관련 검색/전체 검색 = 3/5 = 60%, recall = 3/10 = 30%.

보기③

옳음. 평균 정밀도는 관련 문서들에서의 정밀도 값 평균이다.

보기④

정답(부적절). F-score(조화 평균)는 두 값 중 작은 쪽으로 치우친다. "큰 쪽으로 가까워진다"는 틀린 설명.

각 규칙의 성립 여부를 검토한다.

  • 가: σ_{θ1∧θ2}(E) ≡ σ_{θ1}(σ_{θ2}(E)) — 연접 조건의 셀렉션은 순차 셀렉션으로 분해 가능. 참.

정답 해설

직원이 관계에 전체 참여(최소 1개 부서 소속)한다는 것은, 직원 테이블의 외래키 부서명이 반드시 값을 가져야 함을 의미한다. 이를 관계 모델에서 강제하려면 직원 테이블의 부서명 외래키를 NOT NULL로 정의하면 된다. 그러면 부서명이 NULL인 직원(어떤 부서에도 속하지 않는 직원)이 존재할 수 없으므로 최소 카디널리티 1이 보장된다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

R2의 age 집합은 {30, 40, NULL}이다. R1.age NOT IN (30, 40, NULL)은 R1.age <> 30 AND R1.age <> 40 AND R1.age <> NULL로 평가되는데, 어떤 값과 NULL의 비교(<> NULL)는 항상 unknown이다.

  • 매칭되는 행(age=30 등)은 명백히 false.
  • 매칭되지 않는 행(age=10 등)도 10<>NULL이 unknown이라 전체 AND가 unknown이 되어 참이 아니다.

즉 부속질의에 NULL이 하나라도 있으면 NOT IN은 어떤 행도 반환하지 못한다. 따라서 실행결과는 빈 테이블이며 정답은 ③번이다. 이것은 NOT IN + NULL의 대표적 함정이다.

오답 분석

보기①·②·④

보기③

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

삽입할 모조키 000010은 상위 비트가 000인 버킷으로 가는데, 이 버킷 [000101,000001,000110,000100]은 이미 4개로 포화이고 지역 깊이(3) = 전역 깊이(3)이다. 지역 깊이가 전역 깊이와 같은 버킷이 오버플로하면 디렉터리를 2배로 확장(전역 깊이 3→4)하고 해당 버킷을 분할한다.

분할 후 전역 깊이는 4가 되고, 오버플로 버킷은 지역 깊이 4인 두 버킷으로 나뉜다. 이때 전체 버킷의 지역 깊이 합을 다시 계산하면 14가 된다. 따라서 전역 깊이 4, 지역 깊이의 합 14로 정답은 ④번이다.

(포화 버킷의 지역 깊이가 전역 깊이와 같을 때만 디렉터리가 확장되며, 지역 깊이가 전역 깊이보다 작으면 디렉터리 확장 없이 버킷만 분할된다.)

오답 분석

보기①·③

전역 깊이를 3으로 유지한다고 보았으나, 지역 깊이=전역 깊이인 포화 버킷 분할은 디렉터리를 2배로 확장(전역 깊이 4)시킨다.

보기②

전역 깊이 4는 맞으나 지역 깊이 합을 11로 잘못 계산했다(분할로 증가한 지역 깊이를 반영하면 14).

보기④

정답. 디렉터리 확장으로 전역 깊이 4, 분할 반영 후 지역 깊이 합 14.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

Apache Spark의 핵심 데이터 추상화는 RDD(Resilient Distributed Dataset)이다. RDD는 클러스터에 분산되어 있고, 불변(immutable)이며, 계보(lineage)를 통해 장애 시 복구 가능한(resilient) 데이터셋이다. Spark의 연산자(변환/액션)는 RDD를 입력으로 받아 RDD를 반환하는 방식으로 계산을 구성한다.

따라서 ㉮에 들어갈 용어는 RDD이며, 정확한 명칭은 Resilient Distributed Dataset인 ③번이다. ①·②의 HDFS는 저장 파일시스템이지 Spark의 계산 추상화가 아니고, ④의 명칭(Reproducible Duplicated Dataset)은 잘못된 풀이다(HDFS의 정확한 명칭도 Distributed이지 Duplicated가 아니다).

오답 분석

보기①

HDFS는 하둡의 분산 파일시스템(저장소)으로, Spark의 데이터 표현 추상화가 아니다.

보기②

'Hadoop Duplicated File System'은 잘못된 명칭이다(정확히는 Distributed).

보기③

정답. RDD = Resilient Distributed Dataset, Spark의 핵심 추상화.

보기④

'Reproducible Duplicated Dataset'은 잘못된 풀이다. 정확히는 Resilient Distributed Dataset.

관련 이론 정리

Spark RDD 특성:

특성 내용

연산 변환(lazy) + 액션(실행 트리거)
실무 적용 사례

빅데이터 처리 아키텍처 감리 시 Spark의 RDD/DataFrame 사용, 변환의 지연 실행(lazy)·계보 기반 복구, 셔플 비용 등이 성능·내결함성에 적절한지 검토한다.

영역 2 · 정답 해설 / 오답 분석 / 이론
정답 해설

WAL에서 각 로그 엔트리의 용도는 다음과 같다.
- AFIM(After Image, 변경 후 이미지): 커밋된 트랜잭션의 변경이 디스크에 반영되지 않은 채 장애가 났을 때 다시 적용(REDO)하기 위해 필요하다. 따라서 (가) = REDO.
- BFIM(Before Image, 변경 전 이미지): 커밋되지 않은 트랜잭션의 변경을 취소(UNDO)해 원상 복구하기 위해 필요하다. 따라서 (나) = UNDO.

그러므로 (가) REDO, (나) UNDO인 ③번이 옳다. VOLATILE/PERSISTENT는 이 문맥의 회복 용어가 아니다.

오답 분석

보기①·②

VOLATILE/PERSISTENT는 저장 매체 특성 용어로, AFIM/BFIM의 회복 목적(REDO/UNDO)과 무관하다.

보기③

정답. AFIM→REDO(재적용), BFIM→UNDO(취소).

보기④

(가)와 (나)가 뒤바뀌었다. After Image는 REDO, Before Image는 UNDO다.

관련 이론 정리

WAL 로그와 회복:

이미지 용도 대상
AFIM(After) REDO(재적용) 커밋됐으나 미반영
BFIM(Before) UNDO(취소) 미커밋 변경
WAL 원칙 로그를 데이터보다 먼저 기록 회복 보장
실무 적용 사례

DBMS 회복 정책 감리 시 WAL 준수(로그 선기록), 체크포인트, REDO/UNDO 로그 보존이 장애 복구를 보장하는지 확인한다.

정답 해설

k-Means는 매 반복(loop)마다 각 점을 가장 가까운 중심(centroid)에 할당하기 위해 점-중심 간 거리를 계산한다. 중심이 군집 대상 100개의 점 중에서 선택되므로, 중심으로 지정된 5개 점은 자기 자신이 속한 중심으로 자명하게 배정되어 거리 계산이 필요 없다.

따라서 매 반복당 거리 계산은 (중심이 아닌 점 95개) × (중심 5개) = 475회이다. 반복이 5회이므로:

475 × 5 = 2375회

정답은 ④번이다. (만약 자기 자신 거리를 제외하지 않으면 100×5×5=2500이 되지만, 중심이 실제 점이라는 조건을 반영해 5개 중심점을 제외하여 2375가 된다.)

"거리 계산 = (점 수) × k × 반복"의 기본식에 "중심이 실제 점이라 자기 거리를 제외(−5×반복)"하는 조건을 반영할 수 있는지 묻는다.
